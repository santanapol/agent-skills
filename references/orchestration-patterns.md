# Orchestration Patterns

บัญชีอ้างอิงของรูปแบบการประสานงาน (Orchestration patterns) ของ Agent ที่ Repo นี้ให้การรับรอง รวมถึงข้อผิดพลาดที่ควรหลีกเลี่ยง (Anti-patterns) โปรดอ่านเอกสารนี้ก่อนที่จะเพิ่ม Slash command ใหม่เพื่อประสานงานระหว่างหลาย Personas หรือก่อนที่จะแนะนำ Persona ใหม่ที่เอาไป "ห่อหุ้ม" Persona เดิมที่มีอยู่

กฎข้อบังคับที่สำคัญที่สุด: **ผู้ใช้ (หรือ Slash command) คือผู้ประสานงานหลัก (Orchestrator) ส่วน Personas จะต้องไม่เรียกใช้ Personas อื่น** ทักษะ (Skills) คือขั้นตอนไฟลต์บังคับที่อยู่ภายใน Workflow ของแต่ละ Persona

---

## Endorsed patterns (รูปแบบที่แนะนำให้ใช้)

### 1. Direct invocation (ไม่มีการประสานงาน)

ใช้ Persona เดียว, มีมุมมองเดียว, สร้างผลลัพธ์ (Artifact) เดียว เป็นค่าเริ่มต้นและเป็นวิธีที่ประหยัดต้นทุนที่สุด

```
ผู้ใช้ (user) → code-reviewer → รายงาน (report) → ผู้ใช้ (user)
```

**ใช้เมื่อ:** งานนั้นอาศัยมุมมองเดียวสำหรับผลลัพธ์เดียว และคุณสามารถอธิบายความต้องการได้ในประโยคเดียว

**ตัวอย่าง:**
- "ช่วยรีวิว PR นี้หน่อย" → `code-reviewer`
- "หาปัญหาด้านความปลอดภัยในไฟล์ `auth.ts`" → `security-auditor`
- "มี Test ไหนหายไปบ้างสำหรับขั้นตอนการชำระเงิน (checkout flow)?" → `test-engineer`

**ต้นทุน:** ใช้เวลาสื่อสาร 1 รอบ (One round trip) เป็นบรรทัดฐานที่คุณควรนำไปเปรียบเทียบกับรูปแบบการประสานงานอื่นๆ เสมอ

---

### 2. Single-persona slash command

ใช้ Slash command หนึ่งคำสั่งไปครอบ Persona หนึ่งตัว พร้อมกับทักษะต่างๆ ของโปรเจกต์ ช่วยให้ผู้ใช้ไม่ต้องอธิบายขั้นตอนการทำงานเดิมซ้ำทุกครั้ง

```
/review → code-reviewer (พร้อมกับทักษะ code-review-and-quality) → รายงาน
```

**ใช้เมื่อ:** มีการเรียกใช้ Persona เดิมซ้ำๆ ด้วยการตั้งค่าเดียวกัน

**ตัวอย่างใน Repo นี้:** `/review`, `/test`, `/code-simplify`

**ต้นทุน:** เท่ากับการเรียกใช้โดยตรงแบบ Direct invocation เพราะ Slash command เป็นเพียงการบันทึก Prompt เอาไว้

**สัญญาณเตือน (Anti-signal):** หากเนื้อหาของ Slash command มุ่งเน้นไปที่ "การตัดสินใจว่าจะเรียกใช้ Persona ตัวไหนดี" ให้ลบทิ้ง แล้วปล่อยให้ผู้ใช้เรียกใช้ Persona โดยตรงแทน

---

### 3. Parallel fan-out with merge

ให้หลาย Personas ทำงานกับข้อมูล (Input) เดียวกันแบบขนานกัน (Concurrent) โดยแต่ละตัวจะสร้างรายงานแยกต่างหากของตัวเอง จากนั้นจะมีขั้นตอนการรวมข้อมูล (Merge step - ภายใน Context ของ Agent หลัก) เพื่อสังเคราะห์รายงานทั้งหมดให้ออกมาเป็นการตัดสินใจเดียว

```
                     ┌─→ code-reviewer    ─┐
/ship → กระจายงาน ───┼─→ security-auditor ─┤→ merge → ตกลง/ยกเลิก + แผน rollback
                     └─→ test-engineer    ─┘
```

**ใช้เมื่อ:**
- งานย่อยแต่ละงานเป็นอิสระต่อกันอย่างแท้จริง (ไม่มีการแชร์สถานะที่เปลี่ยนแปลงได้, ไม่มีลำดับความเชื่อมโยงก่อน-หลัง)
- Agent ย่อยแต่ละตัวจะได้ประโยชน์จากการมี Context window ของตัวเอง
- ขั้นตอนการรวมข้อมูล (Merge step) มีขนาดเล็กพอที่จะอยู่ใน Context ของ Agent หลักได้
- ความเร็วตามเวลาจริง (Wall-clock latency) เป็นเรื่องสำคัญ

**ตัวอย่างใน Repo นี้:** `/ship`

**ต้นทุน:** ใช้ Context ของ Agent ย่อย N ตัว + การทำงานรอบ Merge อีก 1 รอบ มีต้นทุนสูงกว่าการเรียกใช้โดยตรง แต่ใช้เวลาจริงน้อยกว่า และสร้างรายงานที่ดีกว่า เพราะ Agent ย่อยแต่ละตัวได้จดจ่ออยู่กับมุมมองเดียวของตัวเอง

**ตรวจสอบรายการนี้ก่อนนำรูปแบบนี้ไปใช้:**
- [ ] ฉันสามารถรัน Agent ย่อยทั้งหมดพร้อมกันได้โดยไม่มีปัญหาเรื่องลำดับก่อนหลังใช่ไหม?
- [ ] แต่ละ Persona สร้างข้อค้นพบที่ *แตกต่างกัน* จริงๆ ใช่ไหม (ไม่ใช่แค่การค้นพบสิ่งเดียวกันแต่มองจากมุมอื่น)?
- [ ] ขั้นตอนการรวมข้อมูล (Merge step) จะใส่ลงในบริบทที่เหลือของ Agent หลักได้พอใช่ไหม?
- [ ] เวลาที่ผู้ใช้ต้องรอนานพอที่จะเห็นความแตกต่างของความเร็วจากการรันขนานกันได้ใช่ไหม?

หากคำตอบข้อใดข้อหนึ่งคือ "ไม่ใช่" ให้กลับไปใช้แบบ Direct invocation หรือ Single-persona command

### 4. Sequential pipeline as user-driven slash commands

ผู้ใช้เป็นคนรัน Slash commands ตามลำดับที่กำหนด เพื่อนำพาบริบท (Context หรือ ประวัติ Commit) จากขั้นหนึ่งไปสู่อีกขั้นหนึ่ง ในกระบวนการนี้ไม่มี Agent ที่เป็นผู้ประสานงาน — ตัวผู้ใช้เองนั่นแหละคือผู้ประสานงาน

```
ผู้ใช้รัน:  /spec  →  /plan  →  /build  →  /test  →  /review  →  /ship
```

**ใช้เมื่อ:** กระบวนการทำงานมีความเชื่อมโยงกัน (ขั้นตอนถัดไปต้องใช้ผลลัพธ์จากขั้นตอนก่อนหน้า) และการตัดสินใจของมนุษย์ระหว่างขั้นตอนนั้นมีประโยชน์ต่อตัวงาน

**ตัวอย่างใน Repo นี้:** วงจรชีวิตทั้งหมดตั้งแต่ DEFINE → PLAN → BUILD → VERIFY → REVIEW → SHIP

**ต้นทุน:** ใช้ Context ของ Agent ย่อย 1 ตัวต่อ 1 ขั้นตอน ส่วนที่ไม่มีต้นทุนเพิ่มคือชั้นของการประสานงาน (Orchestration layer) เพราะมันไม่มี Agent ผู้ประสานงานเข้ามาเกี่ยว

**เหตุผลที่ไม่ควรใช้ระบบอัตโนมัติมาจัดการทั้งหมด:** เพราะ LLM "Lifecycle orchestrator" จะ (a) ทำรายละเอียดบางอย่างหล่นหายไปในแต่ละขั้นตอนเพราะต้องสรุปข้อมูลเพื่อส่งต่อ, (b) ข้ามจุดตรวจของมนุษย์ซึ่งอาจช่วยจับข้อผิดพลาดก่อนที่จะถลำลึกไปผิดทาง และ (c) เสียต้นทุน Token เป็นสองเท่าจากการให้ Agent สรุปความไปมา

### 5. Research isolation (context preservation)

เมื่อมีงานที่ต้องอ่านข้อมูลปริมาณมาก ซึ่งอาจเข้าไปปะปนรบกวนบริบทหลัก ให้แตก Agent ย่อยสำหรับการค้นคว้า (Research sub-agent) ออกมา ซึ่งมันจะตอบกลับแค่บทสรุปเท่านั้น

```
Agent หลัก → Research sub-agent (ไปอ่าน 50 ไฟล์) → ได้บทสรุป (digest) → Agent หลักทำงานต่อ
```

**ใช้เมื่อ:**
- เซสชันหลักต้องจดจ่ออยู่กับงานถัดไป
- ผลการสืบค้นมีขนาดเล็กกว่าข้อมูลที่มันต้องอ่านอย่างมาก
- การตัดสินใจจะมีคุณภาพดีขึ้น หาก Agent หลักมีพื้นที่เหลือไว้ใช้คิด

**ตัวอย่าง:** "หาจุดที่มีการเรียกใช้ API ที่เลิกใช้แล้วทุกแห่งใน Monorepo", "สรุปมาซิว่า ADR ทั้ง 30 ฉบับนั้นพูดถึงเรื่องการทำ Caching อย่างไรบ้าง"

**ต้นทุน:** ใช้ Context ของ Agent ย่อยที่แยกตัวออกไป 1 ครั้ง ถือว่าคุ้มค่ามากทุกครั้งที่ทางเลือกคือต้องโหลดไฟล์หลายร้อยไฟล์เข้าไปอัดแน่นอยู่ใน Context หลัก

**หากใช้บน Claude Code ให้ใช้ Subagent `Explore` ที่มีมาให้แล้ว** ดีกว่าไปสร้าง Research persona ขึ้นมาเอง `Explore` นั้นรันบนโมเดล Haiku, จะถูกระงับสิทธิ์ในการเขียน/แก้ไข และมันถูกออกแบบมาเพื่อรูปแบบนี้โดยเฉพาะ จะกำหนด Research subagent เองก็ต่อเมื่อ `Explore` ไม่ตอบโจทย์ (เช่น คุณต้องการกำหนด System prompt เฉพาะทางที่โมเดลเดาทางเองไม่ได้)

## Claude Code compatibility

บัญชีรูปแบบการประสานงานนี้ไม่ได้ผูกมัดกับแพลตฟอร์มใด แต่ผู้อ่านส่วนใหญ่มักจะใช้รันบน Claude Code นี่คือวิธีที่แต่ละรูปแบบถูกเชื่อมโยงกับความสามารถพื้นฐานของ Claude Code — และเป็นจุดที่แพลตฟอร์มช่วยบังคับใช้กฎให้กับเรา

### Personas อยู่ที่ไหน

Plugin subagents ถูกวางไว้ในแฟ้ม `agents/` ที่ root ของ Plugin Repo นี้คือ Plugin ตัวหนึ่ง (ดูได้จาก `.claude-plugin/plugin.json`) ดังนั้นไฟล์ `agents/code-reviewer.md`, `agents/security-auditor.md`, และ `agents/test-engineer.md` จะถูกค้นพบโดยอัตโนมัติเมื่อเปิดใช้งาน Plugin โดยไม่ต้องไปตั้งค่า path เพิ่มเติม

### Subagents เทียบกับ Agent Teams

Claude Code มีวิธีรันแบบคู่ขนาน (Parallelism) สองแบบ รูปแบบที่ 3 (Parallel fan-out with merge) เข้ากันได้กับ **subagents** แต่ถ้าคุณต้องการทีมที่คุยโต้ตอบกันเองได้ ให้ใช้ **Agent Teams** แทน

| | Subagents | Agent Teams |
|--|-----------|-------------|
| การประสานงาน | Agent หลักกระจายงาน, Agent ย่อยแค่รายงานผลกลับ | สมาชิกทีมแชทคุยกันเอง, แชร์คิวงานร่วมกัน |
| บริบท (Context) | Agent ย่อยมี Context window ของตัวเอง | สมาชิกแต่ละตัวมี Context window ของตัวเอง |
| ใช้เมื่อ | ทำงานอิสระที่ต้องรายงานผล | ทำงานที่ต้องปรึกษาหารือ |
| สถานะ | เสถียร | ทดลองใช้งาน — ต้องกำหนด `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1` |
| ต้นทุน | ต่ำกว่า | สูงกว่า — สมาชิกแต่ละตัวเปรียบเหมือน Claude อีก Instance หนึ่ง |

**Personas ใน Repo นี้ทำงานได้ทั้งสองโหมด** หากมันถูกเรียกในฐานะ Subagents (เช่น ถูกเรียกจาก `/ship`) พวกมันจะรายงานผลลัพธ์กลับไปยัง Session หลัก แต่หากมันถูกเรียกในฐานะ Teammates (`Spawn a teammate using the security-auditor agent type…`) พวกมันสามารถท้าทายโต้แย้งข้อค้นพบของกันและกันได้โดยตรง สิ่งที่แตกต่างมีแค่สภาวะที่มันถูกเรียก ส่วนนิยามของ Persona จะยังคงเดิม

ข้อควรระวัง: ช่อง Frontmatter ของ `skills` และ `mcpServers` ใน Persona จะทำงานเมื่อถูกรันเป็น Subagent แต่มันจะ **ไม่ถูกสนใจ (ignored) เมื่อรันในฐานะ Teammate** — Teammates จะโหลด Skills และ MCP servers จากโปรเจกต์และผู้ใช้ของคุณแทน เหมือนกับ Session ปกติ หากมี Persona ไหนที่ต้องอาศัย Skill หรือ MCP server บางตัว ให้ไปตั้งค่าที่ระดับ Session เพื่อให้มีใช้งานได้ทั้งสองโหมด

### กฎที่ถูกบังคับใช้โดยแพลตฟอร์ม

มีกฎสองข้อในบัญชีนี้ที่ไม่ใช่แค่ธรรมเนียมปฏิบัติ — แต่ Claude Code บังคับใช้งานเลย:

- **"Subagents ไม่สามารถเรียกใช้ Subagents อื่นได้"** (ตามเอกสารอ้างอิง) ดังนั้น Anti-pattern B (Persona เรียกใช้ Persona) และ Anti-pattern D (การซ้อน Persona กันลึกๆ) จึงไม่สามารถเกิดบน Claude Code ได้โดยกำเนิด
- **"ห้ามมี Teams ซ้อนกัน"** — สมาชิกทีมไม่สามารถสร้างทีมย่อยของตัวเองได้ เป็นการบล็อก Anti-patterns ในระดับของการสร้างทีม

ซึ่งหมายความว่าคุณสามารถนำรูปแบบในบัญชีนี้ไปใช้ได้เลย โดยไม่ต้องกลัวว่าทีมงานจะสร้าง Anti-patterns ออกมาโดยไม่ได้ตั้งใจ เพราะมันจะพังโหลดไม่ขึ้นตั้งแต่แรก

### Subagents พื้นฐานที่ควรทราบ

ก่อนที่จะเริ่มสร้าง Subagent ใหม่ ควรเช็คก่อนว่ามีพวกนี้ครอบคลุมบทบาทแล้วหรือยัง:

| Built-in | หน้าที่ |
|----------|---------|
| `Explore` | ค้นหาและวิเคราะห์โค้ดแบบอ่านอย่างเดียว (Read-only) ใช้สำหรับ Pattern 5 (research isolation) |
| `Plan` | ค้นคว้าข้อมูลแบบอ่านอย่างเดียว (Read-only) ระหว่างโหมด Plan |
| `general-purpose` | ใช้สำหรับงานหลายขั้นตอนที่ต้องการทั้งการสำรวจและปรับเปลี่ยนแก้ไข |

อย่าไปนิยามพวกนี้ซ้ำ แต่ให้ซ้อนชั้น Persona เฉพาะทางของคุณ (เช่น code-reviewer, security-auditor, test-engineer) ทับลงไปแทน

### ข้อจำกัดของ Frontmatter สำหรับ Plugin agents

Plugin subagents **ไม่รองรับ** ฟิลด์ Frontmatter ดังนี้: `hooks`, `mcpServers`, หรือ `permissionMode` — ค่าเหล่านี้จะถูกข้ามเงียบๆ หากมี Persona ตัวใหม่ต้องการใช้ฟิลด์เหล่านี้ ผู้ใช้จะต้องคัดลอกไฟล์ไปวางใน `.claude/agents/` หรือ `~/.claude/agents/` เอง

ฟิลด์ที่ **ใช้ได้** ใน Plugin agents ได้แก่: `name`, `description`, `tools`, `disallowedTools`, `model`, `maxTurns`, `skills`, `memory`, `background`, `effort`, `isolation`, `color`, `initialPrompt` คุณสามารถกำหนดรุ่น `model` เฉพาะ Persona ได้หากต้องการปรับจูนราคา (เช่น ให้ใช้ Haiku สแกนหา Test coverage ด้วย `test-engineer`, ใช้ Sonnet สำหรับ `code-reviewer`, และใช้ Opus สำหรับ `security-auditor`)

### การสร้าง Subagents หลายตัวพร้อมกันแบบคู่ขนาน

ใน Claude Code, การรัน Parallel fan-out (Pattern 3) จำเป็นต้องเรียกใช้เครื่องมือ **Agent หลายตัวในรอบ (turn) ของ Assistant เดียวกัน** หากเรียกคนละรอบมันจะทำงานแบบเรียงลำดับแทน `/ship` เป็นตัวอย่างที่ทำสิ่งนี้อย่างชัดเจน ดังนั้น Orchestrator command ใหม่ๆ ก็ควรทำแบบเดียวกัน

## Worked example: Agent Teams for competing-hypothesis debugging

ตัวอย่างนี้จะแสดงว่าเมื่อไหร่ควรใช้ **Agent Teams** แทนการกระจาย Subagent ของ `/ship` รูปแบบสองอันนี้มองเผินๆ เหมือนจะคล้ายกัน — ทั้งคู่จะเรียก Persona ทั้ง 3 ตัวเหมือนกัน — แต่มูลค่าของมันมาจากคนละแหล่งกัน

### สถานการณ์

> *กระบวนการชำระเงิน (Checkout) มักจะค้างนาน ~30 วินาทีเป็นพักๆ ก่อนจะสำเร็จ มันเกิดขึ้นประมาณ 1 ใน 50 เซสชัน ไม่มีข้อผิดพลาดขึ้นใน Logs เริ่มเป็นหลังจากที่เพิ่ง Release ระบบไปเมื่อสัปดาห์ที่แล้ว*

สาเหตุที่น่าจะเป็นไปได้ (ซึ่งไม่สามารถเกิดขึ้นพร้อมกันได้ และเข้ากับอาการทุกอย่าง):

1. มีปัญหา Race condition ในขั้นตอนยืนยันการชำระเงิน (Payment-confirmation) ที่เพิ่งทำใหม่
2. การตรวจสอบ Auth บางครั้งอาจตกหล่นไปเข้ากระบวนการ Network call แบบซิงโครนัสที่ช้ามาก
3. ไม่มี Index ใน Query ที่มีปริมาณการสเกลตามขนาดของตะกร้าสินค้า
4. API ภายนอก (Third-party) ทำงานผิดพลาด โดยที่ SDK พยายาม Retry เองเงียบๆ ก่อนจะ Time out

Agent ตัวเดียวจะเลือกสมมติฐานแรกที่ดูสมเหตุสมผลและหยุดตรวจสอบต่อ ส่วนการใช้ `/ship` เพื่อกระจาย Subagent แต่ละ Persona จะรายงานกลับแยกกัน — แต่รายงานของพวกมันจะไม่เคยมาเจอกัน ดังนั้นจึงไม่มีใครมาแย้งทฤษฎีที่ผิดทิ้งไป

นี่แหละคือเคสที่ตรงกับเอกสารของ Agent Teams เป๊ะๆ: *"เมื่อมีผู้ตรวจสอบอิสระหลายคนพยายามหาหลักฐานมาหักล้างกัน ทฤษฎีใดที่อยู่รอด ก็มักจะเป็นสาเหตุที่แท้จริง"*

### ทำไมเคสนี้ถึง *ไม่ใช่* งานของ `/ship`

| | `/ship` (subagents) | Agent Teams |
|--|--------------------|-------------|
| สิ่งที่ Sub-agents เห็น | มองเห็น Diff เดียวกัน, แต่มองจากมุมที่ต่างกัน | มีคิวงานแชร์กัน, มองเห็นข้อความของคนอื่นด้วย |
| ผลลัพธ์ | รายงานอิสระ 3 อัน → นำมารวมกันทีเดียว | เกิดการถกเถียงโต้แย้ง → จนพบสาเหตุที่เป็นฉันทามติ |
| เหมาะที่จะใช้เมื่อ | คุณต้องการข้อสรุปสำหรับ Artifact ที่มีอยู่แล้ว | คุณต้องการ *ค้นหา* สาเหตุท่ามกลางสมมติฐานต่างๆ |

`/ship` คือการลงคำตัดสิน; ส่วน Agent Teams คือกระบวนการสืบสวน

### การตั้งค่า (ตั้งค่าครั้งเดียว, แยกตามสภาพแวดล้อม)

Agent Teams ยังเป็นช่วงทดลอง ในไฟล์ `~/.claude/settings.json`:

```json
{
  "env": {
    "CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS": "1"
  }
}
```

ต้องใช้ Claude Code เวอร์ชัน v2.1.32 เป็นต้นไป Personas ใน Repo นี้จะถูกดึงไปใช้โดยอัตโนมัติ — ไม่ต้องมาสร้างไฟล์การตั้งค่าทีมด้วยมือ

### คำสั่ง Trigger prompt

ให้พิมพ์ลงใน Session หลัก ด้วยภาษาพูดแบบธรรมชาติ:

```
ผู้ใช้รายงานว่ากระบวนการ Checkout มักจะค้างประมาณ ~30 วินาทีเป็นช่วงๆ หลังจาก Release เมื่อสัปดาห์ที่แล้ว 
ไม่มีแจ้งเตือน error ใน logs 

สร้าง Agent team เพื่อดีบักเรื่องนี้ด้วยสมมติฐานที่หักล้างกัน ให้สร้างเพื่อนร่วมทีม (teammates) 3 ตัว โดยใช้ Agent types ที่มีอยู่แล้ว:

  - code-reviewer — ค้นหาปัญหา race conditions และ blocking calls
                    ในขั้นตอนโค้ดของการ checkout
  - security-auditor — ค้นหาในขั้นตอน auth, การจัดการเซสชัน, 
                       และ synchronous network calls ที่เพิ่มเข้าใหม่
  - test-engineer — เสนอแบบการทดสอบเพื่อแยกแยะสมมติฐานออกจากกัน
                    และตรวจสอบช่องโหว่ด้าน coverage ในการ checkout

ให้พวกเขาส่งข้อความหากันโดยตรงเพื่อหักล้างทฤษฎีของกันและกัน อัปเดตข้อค้นพบเมื่อเกิดฉันทามติ 
และให้สรุปผลก็ต่อเมื่อมี teammates 2 ตัวเห็นด้วยว่าสามารถหักล้างทฤษฎีของคนอื่นได้
```

Agent หลักจะสร้างเพื่อนร่วมทีม 3 คนโดยอ้างอิงจากชื่อ Persona ที่มีอยู่ เนื้อหาของ Persona จะถูก **แทรกต่อท้าย** ไปที่ System prompt ของ Teammate เพื่อเป็นคำสั่งเพิ่มเติม (ทับไปบนคำสั่งประสานงานทีมที่ Agent หลักกำหนดไว้) ส่วน Trigger prompt ด้านบนจะกลายเป็น "งาน (Task)" ของพวกมัน

### สิ่งที่เกิดขึ้น

1. เพื่อนร่วมทีมแต่ละคนจะรันใน Context window ของตัวเอง ทำการสำรวจโค้ดผ่านมุมมองของตัวเอง
2. สมาชิกทีมใช้คำสั่ง `message` เพื่อส่งข้อความหากันโดยตรง Agent หลักไม่ต้องคอยส่งผ่านให้
3. รายการงานที่แชร์ร่วมกันจะแสดงว่าใครกำลังค้นหาอะไรอยู่ — สามารถดูได้ตลอดเวลาด้วย `Ctrl+T` (ในโหมด in-process) หรือใน Tmux pane (ในโหมด split)
4. เมื่อ `code-reviewer` พบว่ามี `Promise.all` ตัวหนึ่งที่น่าจะต้องทำงานแบบต่อเนื่อง (sequential) มันจะส่งข้อความหา `security-auditor` เพื่อยืนยันว่าการเรียก Auth ไม่ได้เป็นส่วนหนึ่งของการเกิด Race ส่วน `security-auditor` ก็จะตรวจสอบและตอบกลับ — ซึ่งอาจยืนยันว่าการเกิด Race นั้นแหละคือปัญหา หรืออาจให้หลักฐานที่ตรงข้ามกัน
5. `test-engineer` จะเสนอ Integration test แบบโฟกัสเป้าหมายเพื่อพิสูจน์ว่าทฤษฎีใดชนะ ซึ่งทั้งทีมจะนำไปใช้เพื่อยืนยันก่อนประกาศมติเห็นพ้อง (consensus)
6. Agent หลักทำการสังเคราะห์ผลลัพธ์ที่เป็นเอกฉันท์และนำเสนอให้คุณ

คุณสามารถแทรกแซงสมาชิกคนไหนก็ได้โดยกดหมุนวน (cycle) ด้วย `Shift+Down` แล้วพิมพ์ — มีประโยชน์มากสำหรับการดึงตัวคนตรวจสอบที่กำลังออกนอกเรื่องกลับมา

### จะทำความสะอาดเมื่อไหร่

เมื่อกระบวนการสืบสวนหาสาเหตุสิ้นสุดลง ให้บอก Agent หลักว่า:

```
ทำความสะอาดข้อมูลทีม (Clean up the team)
```

การล้างทีมจะต้องทำผ่าน Agent หลักเสมอ อย่าทำผ่านเพื่อนร่วมทีม (ตามเอกสารบอกไว้: Teammates ขาด Context ของภาพรวมทีมในการล้างข้อมูล)

### ความคาดหวังเรื่องต้นทุน

สมาชิก Sonnet 3 ตัวที่รันหาคำตอบประมาณ 10–15 นาที จะแพงกว่า Persona ทั้ง 3 ที่ถูกรันเป็น Subagents ผ่าน `/ship` อย่างเห็นได้ชัด แต่สิ่งที่แลกมาคือ *คุณภาพของข้อสรุป* — สำหรับการสืบหาบั๊กในระดับ Production ซึ่งการแก้ปัญหาผิดจุดนั้นสร้างความเสียหายที่แพงกว่า การเสียค่า Token เล็กน้อยแค่นี้นับว่าคุ้มค่ามาก แต่ถ้ารีวิว PR ทั่วไป ก็ใช้ `/ship` ต่อไปดีแล้ว

### รูปแบบที่ห้ามใช้ (Anti-pattern) ในสถานการณ์นี้

**ห้าม** สร้างมันเป็น Slash command อย่าง `/debug` ที่ส่ง Subagents ออกไป Subagents ไม่สามารถส่งข้อความหากันได้ — คุณจะสูญเสียการโต้แย้งซึ่งเป็นหัวใจที่ทำให้แพทเทิร์นนี้มันทำงาน หากมีกระบวนการนี้เกิดขึ้นบ่อย ให้สร้าง Trigger prompt ไว้เป็น Snippet ดีกว่าที่จะเอามันไปผูกกับ Slash command ที่เรียกใช้งาน Subagents ผิดวัตถุประสงค์

### เมื่อไหร่ *ไม่ควร* ใช้ Agent Teams

- เมื่อต้องการคำตัดสินในระดับ Production สำหรับ Diff ที่รู้แน่นอนแล้ว → ใช้ `/ship` (subagents)
- ต้องการผู้เชี่ยวชาญ 1 มุมมอง ไปตรวจArtifact 1 ชิ้น → เรียก Persona โดยตรง
- การทำงานตามวงจรชีวิตแบบตามลำดับ (spec → plan → build) → ผู้ใช้รัน Slash command เอาเอง (Pattern 4)
- งานอ่าน/ค้นคว้าเยอะๆ แล้วต้องการแค่บทสรุปสั้นๆ → ใช้ Subagent แบบ `Explore`

จงหยิบ Agent Teams มาใช้ ก็ต่อเมื่อสมาชิก **จำเป็นต้อง** ถกเถียงกัน เพื่อให้ได้คำตอบที่ถูกต้องที่สุดเท่านั้น

## Anti-patterns

### A. Router persona ("meta-orchestrator")

Persona ที่มีหน้าที่เดียวคือตัดสินใจว่าจะไปเรียก Persona ไหนดี

```
/work → router-persona → "อันนี้ต้องใช้การรีวิว" → code-reviewer → router (สรุปความ) → ผู้ใช้
```

**ทำไมถึงล้มเหลว:**
- มันเป็นแค่ชั้นกระจายงานที่ไม่ก่อให้เกิดมูลค่าใดๆ
- ทำให้เกิดการพูดสรุปความเพิ่มถึง 2 ต่อ → ข้อมูลหล่นหาย + ต้นทุนโทเค็นเพิ่มเกือบ 2 เท่า
- ผู้ใช้น่าจะรู้อยู่แล้วว่าต้องการการรีวิว; พวกเขาสามารถเรียก `/review` ได้โดยตรง
- ทำงานซ้ำซ้อนกับสิ่งที่ Slash commands และระบบ intent mapping ใน `.cursor/rules/agent-skills.mdc` ทำไว้อยู่แล้ว

**สิ่งที่ควรทำแทน:** ปรับปรุงหรือเพิ่ม Slash commands และทำคู่มือระบุเจตนา (intent) → คำสั่ง (command) เอาไว้ใน project rules

---

### B. Persona ที่เรียก Persona อื่น

`code-reviewer` ถูกสั่งให้เรียกใช้ `security-auditor` อยู่ภายในตัวมันเองเมื่อเห็นโค้ด Auth

**ทำไมถึงล้มเหลว:**
- Personas ถูกออกแบบมาให้สร้างแค่มุมมองเดียว; การจับพวกมันมาต่อกันเป็นการขัดต่อเจตนารมณ์
- บทสรุปที่ Persona ตัวแรกส่งต่อให้ตัวที่สอง จะมีเนื้อหาบางส่วนขาดหายไป
- ข้อผิดพลาดต่างๆ มีโอกาสเกิดทวีคูณ (รูปแบบการตอบของใครจะชนะ? กฎของใครใช้ได้บ้าง?)
- ซ่อนต้นทุนให้ผู้ใช้มองไม่เห็น

**สิ่งที่ควรทำแทน:** ให้ Persona แนะนำให้ใช้ Audit ต่อในรายงานของมันแทน แล้วให้ผู้ใช้ หรือ Slash command รันกระบวนการนั้น

---

### C. Sequential orchestrator that paraphrases

Agent ที่เรียกใช้ `/spec`, จากนั้น `/plan`, จากนั้น `/build` ฯลฯ ในนามของผู้ใช้

**ทำไมถึงล้มเหลว:**
- เสียจุดตรวจทานของมนุษย์ ที่ช่วยเบรกงานที่ทำผิดทิศทาง
- การส่งต่องานแต่ละขั้นจะใช้การสรุปความ — ทำให้บริบทหล่นหายสะสมจนเละในท่อการทำงานแบบยาว
- ต้นทุน Token เพิ่มสองเท่า: เสีย Token ตอน Agent สั่งการ + ตอน Sub-agent ตอบ กลับไปกลับมาทุกขั้นตอน
- ตัดอำนาจการตัดสินใจของผู้ใช้ออกไป ในจุดที่ต้องการวิจารณญาณมากที่สุด

**สิ่งที่ควรทำแทน:** ให้ผู้ใช้เป็นผู้ประสานงานเอง เขียนแนะนำลำดับการเรียกที่เหมาะสมเอาไว้ใน `README.md` แล้วให้ผู้ใช้เรียกเอาเอง

---

### D. Deep persona trees

`/ship` ไปเรียก `pre-ship-coordinator` ซึ่งก็ไปเรียก `quality-coordinator` ซึ่งก็ดันไปเรียก `code-reviewer` อีกที

**ทำไมถึงล้มเหลว:**
- แต่ละชั้นสร้างความหน่วงเวลาและเพิ่มค่า Token โดยที่ไม่ได้ช่วยการตัดสินใจอะไรเลย
- เวลาจะรัน Debug หาปัญหา จะกลายเป็นการสืบสวนหลายมิติที่วุ่นวาย
- Persona ปลายทาง (Leaf personas) เสียบริบทเพราะผ่านการสรุปมาหลายตลบ

**สิ่งที่ควรทำแทน:** คุมความลึกของการประสานงาน (Orchestration depth) ให้อยู่ที่ไม่เกินระดับ 1 (Slash command → Personas) ส่วนเรื่องการยุบรวมข้อมูล (Merge) ให้มาทำใน Agent หลักแทน

---

## Decision flow

เวลาที่คุณคิดจะสร้าง Orchestrated workflow ใหม่ ให้ตอบคำถามตามนี้:

```
งานนี้ต้องการหนึ่งมุมมองต่อผลงานหนึ่งชิ้น ใช่ไหม?
├── ใช่ → เรียกใช้ตรง (Direct invocation) แล้วจบ
└── ไม่ใช่ → การจับคู่/ส่วนประกอบต่างๆ มีการทำงานซ้ำซากไหม?
         ├── ไม่ใช่ → เรียกใช้ตรง (Direct invocation) แบบตามสถานการณ์ (Ad hoc) แล้วจบ
         └── ใช่ → งานย่อยๆ ทำงานเป็นอิสระต่อกันใช่ไหม?
                  ├── ไม่ใช่ → ให้ผู้ใช้รัน Sequential slash commands เอาเอง (Pattern 4)
                  └── ใช่ → กระจายงานแล้วมารวมที่เดียว (Parallel fan-out with merge / Pattern 3)
                           ตรวจสอบกับรายการด้านบนก่อน
                           หากตกหล่นข้อไหน → ให้กลับไปใช้ Single-persona command (Pattern 2)
```

## เมื่อไหร่ควรเพิ่ม Pattern ใหม่ลงไปในคู่มือนี้

เพิ่มตัวเลือกใหม่ลงไป ก็ต่อเมื่อ:

1. คุณได้ใช้รูปแบบนั้นในการทำงานจริงไปแล้วอย่างน้อย 2 ครั้ง
2. คุณสามารถระบุ Artifact จริงใน Repo นี้ที่ใช้รูปแบบดังกล่าวได้
3. คุณสามารถอธิบายได้ว่าทำไมถึงใช้ Pattern เดิมที่มีอยู่ไม่ได้
4. คุณสามารถอธิบายว่ารูปแบบ Anti-pattern ของมันจะเป็นอย่างไร (คนจะเอาไปทำผิดแบบไหนบ้าง)

การรีบเพิ่มคู่มือแบบคาดการณ์ล่วงหน้า จะกลายเป็นเอกสารอุดมคติที่ไม่มีใครทำตามในที่สุด
