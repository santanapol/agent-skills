# Skill Anatomy (โครงสร้างกายวิภาคของ Skill)

เอกสารฉบับนี้จะมาอธิบายเรื่องของโครงสร้างและรูปแบบของตัวไฟล์ skill ในระบบ agent-skills. เอาไว้ใช้เป็นคู่มือเพื่อส่องนำทาง เวลาอยากจะงอก skill ตัวใหม่ๆ เพิ่ม หรือจะเอาไว้ช่วยให้เข้าใจของเดิมก็ได้นะ

## File Location (ถิ่นที่อยู่)

ทุกๆ skill จะมีวิมานโฟลเดอร์ของตัวเองอาศัยอยู่ใต้ชายคา `skills/`:

```
skills/
  skill-name/
    SKILL.md           # ข้อบังคับ (Required): ไฟล์อธิบายตัว skill เอง
    scripts/           # ทางเลือก (Optional): ที่ซ่อนตัวช่วยแบบสคริปต์รันคำสั่ง สำหรับเวิร์กโฟลว์ของ skill 
    supporting-file.md # ทางเลือก (Optional): เอกสารอ้างอิง ที่จะถูกโหลดมาเฉพาะตอนจำเป็น
```

ในนั้น `SKILL.md` คือไฟล์ไฟท์บังคับตัวเดียวที่คุณต้องมี. ส่วน `scripts/` จะงอกขึ้นมาก็ต่อเมื่อ skill ตัวนี้มีไฟล์สคริปต์ให้รันช่วยทำงานจริงๆ เท่านั้น, และถ้าเป็น skill สายอธิบายด้วยหน้ากระดาษ (markdown-only) ล้วนๆ ก็ละโฟลเดอร์นี้ทิ้งไปได้เลย

## SKILL.md Format (รูปแบบตัวไฟล์ SKILL.md)

### Frontmatter (Required) (ส่วนหน้าต่างหัวเปิด - บังคับ)

```yaml
---
name: skill-name-with-hyphens
description: คอยชี้ทางให้ agents ในเรื่อง [task/workflow]. โผล่มาใช้ยาม (Use when) [เงื่อนไขเฉพาะหน้างานที่โดนกระตุ้น].
---
```

**Rules (กฎเหล็ก):**
- `name`: พิมพ์เล็กล้วน, เคาะแยกคำด้วยขีดกลาง. และชื่อต้องตรงกับชื่อโฟลเดอร์ที่มันอยู่แบบเป๊ะๆ
- `description`: บรรทัดแรกต้องเริ่มด้วยการแนะนำตัวแบบบุรุษที่สามว่า skill ตัวนี้เสกมาทำอะไร, จากนั้นให้เติมเงื่อนไขชี้ชะตาว่า `Use when` (ใช้เมื่อไหร่) ให้ชัดเจน. ต้องมีครบทั้งทำอะไร (*what*) และทำเมื่อไหร่ (*when*). จำกัดความยาวห้ามเกิน 1024 ตัวอักษร.

**Why this matters (ทำไมเรื่องนี้ถึงสำคัญ):** Agents จะออกค้นหาเจอมันได้โดยการส่องอ่านตรง descriptions เนี่ยแหละ. ตัว description จะโดนฉีดเข้าไปฝังเป็นส่วนหนึ่งของคำสั่ง (system prompt), มันเลยจำเป็นอย่างยิ่งที่จะต้องสื่อสารบอก agent ให้รู้ว่ามันให้พลังพิเศษอะไร และควรจะโผล่มาใช้งานเมื่อไหร่. ห้ามเอาขั้นตอน(workflow) มาพิมพ์ลงไปในช่องนี้เด็ดขาด — ไม่งั้นเดี๋ยว agent มันจะดื้อดึงไปอ่านแค่ที่สรุปมา แล้วแอบเทไม่ยอมลงไปอ่านรายละเอียดเต็มๆ ในตัว skill

### Standard Sections (Recommended Pattern) (รูปแบบสัดส่วนหัวข้อที่แนะนำ)

ด้านบนที่เป็น frontmatter นั่นถือเป็นสัญญารูปแบบบังคับตายตัว. ส่วนโครงร่างเค้าโครงหัวข้อย่อยด้านล่างนี้ เป็นแค่แบบฟอร์มแนะนำ (recommended pattern), ไม่ได้แข็งขืนบีบคั้น (rigid template): การขยับขยายไปใช้หัวข้อคำที่มีความหมายตีความใกล้เคียงกัน ย่อมถือว่ารับได้ ถ้ารูปประโยคมันสื่อความหมายได้แจ่มแจ้งกว่า

```markdown
# Skill Title (ชื่อตัว Skill)

## Overview (ภาพรวม)
สรุปสั้นๆ สัก 1-2 ประโยค เพื่ออธิบายว่า skill นี้มีดีอะไร และทำไมคุณถึงควรแคร์มัน.

## When to Use (ใช้งานเมื่อไหร่)
- ฤกษ์ยามสัญญาณกระตุ้นเตือนว่าน่าจะต้องใช้มันแล้ว (อาการหน้างาน, แนวของตัว task)
- เมื่อไหร่ที่ห้ามใช้มันเด็ดขาด (สิ่งที่อยู่นอกเขต)

## [Core Process / The Workflow / Steps] (ขั้นตอนกระบวนการทำ)
หัวใจของเวิร์กโฟลว์, จับมาสับแบ่งเป็นข้อๆ ไม่ก็เป็นช่วงๆ (phases) ไปเลย.
โยนโค้ดตัวอย่าง(Code examples) เสริมลงไปได้ถ้ามันช่วยให้บรรลุเป้า.
ถ้ามีการตัดสินใจแยกทาง, จะแทรกแผนผัง (flowcharts ASCII) ด้วยก็ได้.

## [Specific Techniques / Patterns] (เทคนิค/รูปแบบ เฉพาะกิจ)
กางสเปกชี้แนะกันแบบลึกๆ ไว้รับมือเคสเฉพาะหน้า.
อาจจะเป็นพวก ตัวอย่างโค้ด, เทมเพลต, ไม่ก็ท่าเซ็ต config.

## Common Rationalizations (ข้ออ้างยอดฮิต)
| Rationalization (คำแก้ตัว) | Reality (โลกความจริง) |
|---|---|
| ข้ออ้างที่พวก agents ชอบอ้างเพื่อโดดข้ามขั้นตอน | คำด่าสวนกลับว่าทำไมข้ออ้างนี้มันงี่เง่า |

## Red Flags (สัญญาณเตือนภัย)
- สัญญาณพฤติกรรมบอกใบ้ว่าตัว skill ตัวนี้กำลังโดนแหกกฎ
- จุดจับผิดที่ต้องคอยส่องระหว่างตอนกำลังตรวจ (review) งาน

## Verification (การตรวจทาน)
หลังจากทำพิธีของ skill จนจบกระบวนความแล้ว, ให้มายืนยันดูว่า:
- [ ] เช็คลิสต์ตรวจสอบว่างานผ่านเกณฑ์ยัง (exit criteria)
- [ ] สิ่งที่ต้องนำมาเป็นหลักฐานยืนยัน (Evidence requirements)
```

## Section Purposes (จุดประสงค์ของแต่ละหัวข้อ)

### Overview (ภาพรวม)
เป็นหน้าด่านโฆษณาสรรพคุณ (elevator pitch) ให้ตัว skill. ต้องช่วยตอบคำถามได้ว่า: ไอ้ skill ตัวนี้มีดีอะไร, และทำไม agent ต้องสลดหัวยอมเดินตามมันด้วย?

### When to Use (ใช้งานเมื่อไหร่)
หน้าต่างชี้ทางที่ช่วยให้ agents กับมนุษย์ตัดสินใจว่า skill ตัวนี้น่าจะเข้าทางกับหน้างานตรงหน้ารึเปล่า. ต้องระบุตัวกระตุ้น (triggers) ทิศทางบวก ("เอามาใช้เมื่อเจออาการ X") และข้อห้าม (exclusions) รั้งไว้ ("ห้ามลากมาใช้กับเคส Y").

### Core Process (ขั้นตอนกระบวนการทำ)
ก้อนหัวใจของตัว skill. นี่คือสูตรสเตปบายสเตป (step-by-step workflow) ที่จะจับมือให้ agent เดินตาม. ต้องระบุชัดเจนและจับต้องลงมือทำได้เลย (specific and actionable) — ไม่ใช่ออกมาเป็นคำแนะนำกว้างๆ เป็นพระลอยๆ.

**แบบที่ดี (Good):** "รัน `npm test` แล้วไปตรวจสอบซะว่ามันรันผ่านให้หมด"
**แบบที่พัง (Bad):** "ตรวจเช็คดูด้วยนะว่าไอ้ที่เทสไปมันยังทำงานได้มั้ย"

### Common Rationalizations (ข้ออ้างยอดฮิต)
สิ่งประดิษฐ์ชั้นยอดที่ทำให้ skill เหล่านี้ถูกขับให้ดูโดดเด่นสะดุดตา. มันคือข้ออ้างที่ agents ชอบใช้เป็นข้อแก้ตัวเพื่อหนีข้ามขั้นตอนที่สำคัญๆ, แล้วก็จะโดนจับตอกกลับด้วยคำด่าสวนรัวๆ. ของแบบนี้แหละที่จะกันท่าไม่ให้ agent โชว์ความฉลาดแกมโกง (rationalizing) แล้วหาวิธีหลบหนีไม่ยอมเดินตามขั้นตอน.

ลองนึกภาพตามนะ ตอนที่ agent ทำหน้าทะเล้นบอกว่า "เดี๋ยวฉันค่อยเติม tests ให้ทีหลังนะ" หรือไม่ก็ "งานนี้กล้วยๆ ข้ามไปเลยดีกว่าไม่ต้องทำ spec หรอก" — พวกมุกแก้ตัวทรงนี้แหละที่จะมาซุกซ่อนอยู่ในนี้ พร้อมโดนคำด่าสวนโต้แย้งแบบแสบๆ คันๆ (counter-argument).

### Red Flags (สัญญาณเตือนภัย)
สิ่งสังเกตการณ์ที่ฟ้องให้รู้ว่า skill กำลังถูกแหกกฎคามือ. ถือว่ามีประโยชน์อย่างแรงยามที่ต้องมารีวิวโค้ดหรือตรวจสอบตัวเอง

### Verification (การตรวจทาน)
บรรทัดสุดท้ายของการจบงาน (exit criteria) แผ่นป้ายเช็คลิสต์ที่ตัว agent จะต้องหยิบขึ้นมาเช็คความชัวร์ว่าภารกิจผ่านด่านแล้ว ทุกช่องเช็คจะต้องมีหลักฐานจับต้องได้ (evidence) ยืนยันให้เห็น (เช่น ผลลัพธ์ตอนรัน test, สัญญาณที่เด้งจากการ build, ร่องรอย screenshot, เป็นต้น).

## Supporting Files (ไฟล์เอกสารกองหนุน)

งอกตัวไฟล์กองหนุน(supporting files) เพิ่มมาเฉพาะเมื่อ:
- ข้อมูลความรู้ใช้อ้างอิงมันมีความยาวล้นหลามเกิน 100 บรรทัด (เก็บหน้ากระดาษของ `SKILL.md` หลักไว้ให้โปร่งๆ โล่งๆ เข้าไว้)
- เครื่องมือโค้ดตัวช่วย หรือ สคริปต์ต่างๆ มีความจำเป็นต้องใช้
- ลิสต์ตรวจสอบ (Checklists) ดูจะยืดยาวเกินไป จนสมควรแก่เวลาแยกห้องออกไปเป็นสัดส่วน

เก็บรายละเอียดลูกเล่นต่างๆ เอาไว้ให้แนบชิดกับเนื้อหา (inline) ก็พอ ถ้ามันยังสั้นกว่า 50 บรรทัด.

ถ้า skill ไก่นากระดูกอ่อนของคุณไม่ได้มีความจำเป็นอะไรที่ต้องใช้ตัวคำสั่งรันสคริปต์เสริม, ห้ามทะลึ่งไปปั้นโฟลเดอร์ `scripts/` เปล่าๆ เพียงเพื่อจะเอาไว้เล่นตลกเลียนแบบ skills ของชาวบ้าน. โฟลเดอร์ที่แกล้งทำเป็นเปล่าประโยชน์ มีแต่จะไปรกหูรกตาให้เกะกะสายตา โดยที่ไม่ได้ทำให้หน้าตา skill ดูเท่ขึ้นสักนิดเดียว.

## Writing Principles (หลักปฏิบัติในการเขียน)

1. **Process over knowledge (ให้ความสำคัญกับการลงมือทำมากกว่าทฤษฎี)** Skills คือหน้าตักงานเดินตาม(workflows), ไม่ใช่คู่มือสารานุกรมอ้างอิง(reference docs) เน้นที่สเตปลงมือ, ไม่ใช่มายัดเยียดเล่าเรื่องจริง.
2. **Specific over general (เฉพาะเจาะจงต้องมาก่อนอะไรกว้างๆ)** "ไปสั่ง `npm test` ซะ" มันมีประโยชน์กว่าคำว่า "ตรวจสอบ tests ด้วยล่ะ".
3. **Evidence over assumption (หลักฐานสำคัญกว่าคำนึกมโนเอาเอง)** ทุกๆ ครั้งที่จะไปติ๊กถูกหน้ากล่องเช็ค (verification checkbox), มันต้องตามมาด้วยหลักฐานยืนยัน.
4. **Anti-rationalization (ต่อต้านพวกขี้แก้ตัว)** ทุกจังหวะสำคัญที่สุ่มเสี่ยงต่อการแอบเทข้าม, จะต้องถูกเตรียมพร้อมด้วยข้อโต้แย้งด่าสวนกลับ รอไว้ในตาราง rationalizations.
5. **Progressive disclosure (ค่อยๆ ง้างเปิดให้ดูทีละชั้น)** ตัวหลัก `SKILL.md` รับหน้าที่เบิกโรงต้อนรับแขกก่อน. แล้วพวกลิ่วล้อไฟล์สนับสนุน ค่อยดึงโหลดตามหลังมาเฉพาะยามที่เรียกร้องเท่านั้น.
6. **Token-conscious (ตระหนักถึงปริมาณ Token)** ทุกตัวอักษรและบรรทัดที่ใส่เข้ามาจะต้องหาข้ออ้างได้ว่าจำเป็นต้องมีมัน. ถ้าคุณกลั้นใจตัดหัวข้อไหนออกไป แล้ว agent ยังคงไม่รู้สึกรู้สาแถมเดินตามเส้นทางได้ปกติ, ลบมันทิ้งซะเถอะ!

## Naming Conventions (ธรรมเนียมการตั้งชื่อ)

- พวกโฟลเดอร์ฝั่งของ skills: `พิมพ์เล็กทั้งหมด-เชื่อมเคาะวรรค-ด้วยขีดกลาง`
- ไฟล์หลักของ skill: `SKILL.md` (ตัวพิมพ์ใหญ่ทั้งหมดล้วนๆ)
- พวกไฟล์อ้างอิงกองหนุน (Supporting files): `พิมพ์เล็กทั้งหมด-เชื่อมเคาะวรรค-ด้วยขีดกลาง.md`
- ฝั่งของแหล่งอ้างอิง (References): อาศัยรวมตัวอยู่ภายใต้ชายคาโฟลเดอร์ `references/` ที่ติดสอยห้อยตามมาจาก project root, ไม่ได้ไปแทรกอยู่ข้างในฝั่ง skill

## Cross-Skill References (การพาดพิงอ้างอิงไปถึง Skill อื่น)

ชี้เป้าเรียกใช้งาน skill ตัวอื่นๆ ให้ขานเรียกด้วยชื่อมันเลย:

```markdown
ตามรอยกติกาของตัว skill อย่าง `test-driven-development` ให้ดีเพื่อเอาไว้เขียน tests
ถ้าดันเกิดการบิลด์พังขึ้นมาล่ะก็, ให้สะกิดเรียกตัว `debugging-and-error-recovery` เข้ามาใช้ซะ.
```

ห้ามเขียนเนื้อหาสาระลงไปซ้ำซ้อนกับ skills ตัวอื่นๆ เป็นอันขาด — ให้ใช้วิธีแนบอ้างอิงแล้วสร้างสะพานเชื่อมชี้เป้า (link) โยงไปหาแทน.

## Required vs Recommended (ระหว่าง ข้อบังคับ กับ แค่คำแนะนำ)

ของที่ บังคับต้องมี (Required):

- จะต้องมีไฟล์ `skills/<ชื่อ-skill>/SKILL.md` เอาไว้
- หน้าต่างการเปิด YAML frontmatter จะต้องตรงตามสูตรและต้องมีหัวข้อ `name` กับ `description`
- และหัวข้อ description จะต้องช่วยแถลงไขให้ครบ ทั้งทำอะไร และทำไปเมื่อไหร่

ของที่ แค่แนะนำเฉยๆ (Recommended):

- การจัดลำดับเค้าโครง section ให้ออกมาเหมือนที่โชว์เป๊ะๆ ด้านบน
- การตั้งชื่อ section ด้วยคำเสมือนอย่าง `How It Works`, `Core Process`, หรือ `Workflow` จะฟังดูลื่นไหลเข้าเค้ากับ skill ได้ดีกว่า
- พวกไฟล์อ้างอิงกองหนุน ควรใช้ก็ต่อเมื่อจะช่วยแยกชิ้นส่วนเพื่อให้ตัว `SKILL.md` ของเดิมดูคลีนๆ ขึ้นเท่านั้น
