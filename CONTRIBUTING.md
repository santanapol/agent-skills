# Contributing to Agent Skills

ขอขอบคุณที่สนใจเข้ามาร่วมสร้างสรรค์ผลงาน (contributing) กับเรา! โปรเจกต์นี้คือคลังรวบรวมทักษะวิศวกรรมระดับพร้อมใช้งานจริง (production-grade) สำหรับส่งให้เหล่า AI coding agents นำไปใช้งาน

## Adding a New Skill (การสร้างทักษะใหม่)

1. สร้างโฟลเดอร์ไว้ใต้ `skills/` โดยตั้งชื่อแบบ `kebab-case` (ใช้ขีดกลางเชื่อมคำ)
2. สร้างไฟล์ `SKILL.md` ตามรูปแบบมาตรฐานที่ระบุใน [docs/skill-anatomy.md](docs/skill-anatomy.md)
3. ใส่ส่วนหน้าต่าง YAML frontmatter โดยต้องมีฟิลด์ `name` และ `description` ให้ครบ
4. ให้แน่ใจว่าตัว `description` เริ่มต้นด้วยการอธิบายสรรพคุณว่ามันทำอะไร (แบบบุรุษที่สาม), จากนั้นค่อยตามด้วยเงื่อนไขกระตุ้นให้ใช้งาน (trigger conditions) ว่า `Use when` (ใช้เมื่อไหร่) 

### Skill Quality Bar (มาตรฐานคุณภาพของทักษะ)

ตัว Skills จะต้องมีลักษณะดังนี้:

- **Specific (เฉพาะเจาะจง)** — เป็นขั้นตอนที่ลงมือทำได้จริง ไม่ใช่อ้าปากแนะนำลอยๆ
- **Verifiable (พิสูจน์ได้)** — มีเกณฑ์จบงานที่ชัดเจน พร้อมทั้งต้องการหลักฐานยืนยัน
- **Battle-tested (ผ่านสมรภูมิ)** — อิงจากกระบวนการทำงานวิศวกรรมของจริง ไม่ใช่อิงจากทฤษฎีอุดมคติ
- **Minimal (น้อยแต่มาก)** — ใส่แค่เนื้อหาที่จำเป็นต่อการชี้ทางให้ agent ทำงานได้ถูกต้องก็พอ

### Structure (โครงสร้าง)

ทุกๆ ทักษะตัวใหม่ จะต้องมี:

- มีไฟล์ `SKILL.md` อาศัยอยู่ในโฟลเดอร์ของมัน
- มีส่วน YAML frontmatter พร้อมระบุ `name` และ `description` ที่ถูกต้อง

ทักษะใหม่ๆ ควรจะดำเนินรอยตามโครงสร้างมาตรฐาน (anatomy) ดังนี้:

- **Overview (ภาพรวม)** — ทักษะนี้เอาไว้ทำอะไร และทำไมมันถึงสำคัญ
- **When to Use (จังหวะที่ควรหยิบมาใช้)** — เงื่อนไขกระตุ้นให้เริ่มใช้ทักษะ (Triggering conditions)
- **Process (กระบวนการ)** — ขั้นตอนเวิร์กโฟลว์บอกแบบสเตปต่อสเตป
- **Common Rationalizations (ข้ออ้างยอดฮิต)** — ข้ออ้างที่ agents ชอบใช้เพื่อแอบข้ามขั้นตอน พร้อมคำด่าสวนกลับ (rebuttals)
- **Red Flags (สัญญาณเตือนภัย)** — สัญญาณเตือนว่ากำลังแหกกฎของทักษะนี้
- **Verification (การตรวจทาน)** — วิธีเช็คความชัวร์ว่าลงมือใช้ทักษะนี้ได้ถูกต้องแล้ว

บรรดาฟิลด์ใน frontmatter ด้านบนนั้นคือข้อบังคับ (required). ส่วนโครงสร้างหัวข้อนั้นเป็นเพียงแค่รูปแบบที่แนะนำ (recommended): การใช้หัวข้อที่มีความหมายเทียบเท่า เช่น `How It Works`, `Workflow`, หรือ `Core Process` ถือว่าใช้ได้ ตราบใดที่ยังคงจุดประสงค์เดิมและทำให้ทักษะมันทำตามได้ง่าย

### What Not to Do (สิ่งที่ไม่ควรทำ)

- ห้ามเขียนเนื้อหาซ้ำซ้อนทับกันระหว่าง skills — ให้ใช้วิธีอ้างอิง (reference) โยงไปหา skill อื่นแทน
- ห้ามสร้าง skills ที่เป็นแค่คำแนะนำลอยๆ กว้างๆ แทนที่จะเป็นกระบวนการที่ลงมือทำได้จริง
- ห้ามสร้างไฟล์สนับสนุนอื่นๆ (supporting files) เว้นแต่ว่าเนื้อหาจะทะลุ 100 บรรทัด
- ห้ามสร้างโฟลเดอร์ `scripts/` เปล่าๆ เพียงเพื่อเอาไปเลียนแบบหน้าตา skill อันอื่น — ให้เพิ่ม `scripts/` เฉพาะตอนที่ใน skill มีไฟล์คำสั่งให้เรียกใช้งานได้จริงๆ
- ห้ามเอาพวกเอกสารอ้างอิง (reference material) ไปยัดไว้ในโฟลเดอร์ skill — ให้ไปใช้โฟลเดอร์ `references/` แทน

## Modifying Existing Skills (การปรับปรุงทักษะที่มีอยู่เดิม)

- พยายามปรับแก้ให้แคบและกระทบของเดิมน้อยที่สุด
- รักษารูปแบบโครงสร้างและน้ำเสียง (tone) เดิมไว้ให้คงที่
- ทดสอบว่าส่วนหน้าต่าง YAML frontmatter ยังคงรันได้ปกติหลังจากแก้ไขเสร็จ

## Testing Hooks (การทดสอบระบบ Hooks)

ตัว hook ของ session-start (`hooks/session-start.sh`) จะทำหน้าที่ฉีดเอาตัว meta-skill ที่ชื่อ `using-agent-skills` เข้าไปในทุกๆ การเริ่มต้น session ใหม่ของ Claude Code ตัว regression test ที่อยู่ที่ `hooks/session-start-test.sh` จะเป็นคนคอยตรวจสอบความถูกต้องของก้อน JSON payload ของตัว hook — โดยเทสทั้งกรณีที่มีการติดตั้งโปรแกรม `jq` และกรณีที่ไม่มีโปรแกรม `jq`

ให้รันมันทุกครั้งก่อนที่จะเปิด PR ที่ไปยุ่งเกี่ยวกับ:

- `hooks/session-start.sh`
- `skills/using-agent-skills/SKILL.md` (เนื้อหา meta-skill ที่โดนฝังเข้าไปโดยตัว hook)

```bash
bash hooks/session-start-test.sh
```

ผลลัพธ์ที่คาดหวัง: `session-start JSON payload OK`. ถ้าการเช็คมีข้อผิดพลาด สคริปต์นี้จะโดนเด้งออกจากระบบด้วยรหัส (exit non-zero) ทันที

### Reproducing the no-jq fallback (การจำลองการรันแบบกรณีไม่มี jq ให้ใช้)

ตัว hook จะยอมถอยทัพ (gracefully degrades) ไปปล่อยเป็นก้อน payload แบบระดับ `INFO` แทน เวลาที่มันหาโปรแกรม `jq` บนระบบ `PATH` ไม่เจอ เพื่อทดสอบการทำงานบนเครื่องของคุณ ให้ทำการริบรายชื่อโฟลเดอร์ของ `jq` ออกจากตัวแปร `PATH` ตอนสั่งเรียกใช้งาน test ดู:

```bash
JQ_DIR=$(dirname "$(command -v jq)")
PATH=$(echo "$PATH" | tr ':' '\n' | grep -v "^${JQ_DIR}$" | tr '\n' ':' | sed 's/:$//') \
  bash hooks/session-start-test.sh
```

วิธีนี้จะรันผ่านฉลุยก็ต่อเมื่อโปรแกรม `jq` ของคุณมันอาศัยอยู่แยกในโฟลเดอร์ของตัวเอง (เช่น `/opt/homebrew/bin` จากโปรแกรม Homebrew, หรือ `/usr/local/bin` จากการติดตั้งมือ) ถ้าตัว `jq` ของคุณมันดันอาศัยปะปนกับเครื่องมือระบบ (system bin) อื่นๆ ที่ต้องพึ่งพา (เช่น `mktemp` ใน `/usr/bin`), แนะนำให้ใช้วิธีที่ง่ายกว่าคือ ลง `jq` แยกผ่านตัวติดตั้ง (package manager) ให้มันมีโฟลเดอร์ bin เป็นของตัวเองไปเลย แล้วค่อยรันใหม่

จากนั้นตัวเช็ค `command -v jq` ของ hook ก็จะพังเพราะโดนริบตัวแปร `PATH` ไป, จากนั้นระบบ fallback ระดับ `INFO` ก็จะทำงาน, แล้วตัว test ก็จะขึ้นข้อความเตือนให้ไปลงโปรแกรม `jq is required` แทนการปล่อยก้อน payload ปกติออกมา

## Reporting Issues (การแจ้งปัญหา)

เปิด Issue แจ้งมาได้เลยถ้าคุณเจอ:

- Skill ที่แนะนำได้ห่วย หรือตกยุคไปแล้ว
- การทำงาน (workflow) ด้านวิศวกรรมที่คนเค้าใช้กันทั่วไปแต่เราดันไม่มีของคลุม
- ความขัดแย้งตีกันเองระหว่าง skills แต่ละตัว
