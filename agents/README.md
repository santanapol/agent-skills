# Agent Personas (ตัวละครสมมติสำหรับ Agent)

Specialist personas (ตัวละครผู้เชี่ยวชาญ) ที่มีเพียงบทบาทเดียวและมุมมองเดียว ตัวละครแต่ละตัวจะอยู่ในรูปแบบไฟล์ Markdown ซึ่งพร้อมให้ระบบบังคับใช้(harness) ของคุณดึงไปใช้เป็น system prompt (เช่น Claude Code, Cursor, Copilot เป็นต้น)

| Persona | Role (บทบาท) | Best for (เหมาะสำหรับ) |
|---------|------|----------|
| [code-reviewer](code-reviewer.md) | Senior Staff Engineer | ส่องรีวิวละเอียดครบ 5 แกนมิติ ก่อนกด merge |
| [security-auditor](security-auditor.md) | Security Engineer | ตามล่าหาช่องโหว่, ส่องสไตล์ OWASP |
| [test-engineer](test-engineer.md) | QA Engineer | วางกลยุทธ์สร้าง test, ดูระดับความครอบคลุม, และการใช้ Prove-It pattern |

## How personas relate to skills and commands (ความสัมพันธ์ระหว่าง personas, skills, และ commands)

มี 3 ลำดับชั้น, และต่างมีหน้าที่ต่างกันชัดเจน:

| Layer (ชั้น) | What it is (มันคืออะไร) | Example (ตัวอย่าง) | Composition role (หน้าที่รับผิดชอบ) |
|-------|-----------|---------|------------------|
| **Skill** | วงจรการทำงานที่มีสเตปและเป้าหมายจบงานที่ชัดเจน | `code-review-and-quality` | เป็น *วิธีการ (how)* — โดนเรียกให้ทำงานจากข้างใน persona หรือ command อีกที |
| **Persona** | ตัวละครที่มีบทบาท, มีมุมมอง และรูปแบบการรายงานเฉพาะตัว | `code-reviewer` | เป็น *ตัวบุคคล (who)* — สวมบทบาทมุมมองต่างๆ, และเสกรายงาน |
| **Command** | คำสั่งหน้าบ้านที่ผู้ใช้เรียกใช้งาน | `/review`, `/ship` | เป็น *ช่วงเวลา (when)* — คอยประกอบร่างผสม persona และ skill เข้าด้วยกัน |

ผู้ใช้งาน (หรือ slash command) ทำหน้าที่เสมือนเป็นวาทยกร. **Personas ห้ามเรียกใช้งาน personas อื่นเด็ดขาด** Skills เป็นเหมือนเส้นทางบังคับที่อยู่ภายใต้การควบคุมการทำงานของ persona

## When to use each (เมื่อไหร่ควรใช้อะไร)

### Direct persona invocation (เรียกใช้ Persona ตรงๆ)
ให้เลือกท่านี้เมื่อคุณต้องการมุมมองเพียงมุมมองเดียว ที่จะมาส่องของที่แก้ตรงหน้า โดยมีผู้ใช้คอยสั่งการอยู่ในวงโคจร

- "ช่วยรีวิว PR นี้ให้หน่อย" → เรียกใช้ `code-reviewer` แบบตรงๆ
- "มีช่องโหว่ความปลอดภัยหลุดอยู่ใน `auth.ts` ไหม?" → เรียกใช้ `security-auditor` แบบตรงๆ
- "มี tests แบบไหนที่หายไปจากหน้าจอ checkout ไหม?" → เรียกใช้ `test-engineer` แบบตรงๆ

### Slash command (มี 1 persona แอบอยู่เบื้องหลัง)
ให้เลือกท่านี้เมื่อมีกระบวนการที่ทำซ้ำๆ บ่อยๆ และคุณขี้เกียจที่จะต้องมานั่งอธิบายซ้ำทุกรอบ

- `/review` → เอา `code-reviewer` ห่อทับเข้าไปในตัว skill ที่ใช้รีวิวของโปรเจกต์
- `/test` → เอา `test-engineer` ห่อทับเข้าไปในตัว skill แบบ TDD

### Slash command (orchestrator — กระจายกำลัง fan-out)
ให้เลือกท่านี้ **เฉพาะ** เมื่อมีกระบวนการสืบสวนอิสระ **(independent)** หลายๆ งาน ที่สามารถวิ่งทำขนานกันไปได้ และสามารถดึงเอารายงานกลับมาให้ agent ตัวหลักเพื่อรวมร่างได้

- `/ship` → กระจายกำลังพัดออก (fans out) ไปยัง `code-reviewer` + `security-auditor` + `test-engineer` แบบคู่ขนาน, จากนั้นค่อยจับเอารายงานของทุกคนมารวมร่างเพื่อฟันธงว่าจะเอาไปต่อ(go) หรือพักก่อน(no-go)

รูปแบบนี้ถือเป็นแพทเทิร์นการควบคุมประสานงาน(orchestration pattern) หนึ่งเดียวที่โปรเจกต์นี้รับรอง ดูเพิ่มเติมที่ [references/orchestration-patterns.md](../references/orchestration-patterns.md) เพื่อดูแคตตาล็อกรูปแบบการทำงานทั้งหมดรวมทั้งพวกท่าต้องห้าม (anti-patterns)

## Decision matrix (ตารางวิเคราะห์การตัดสินใจ)

```
งานตรงหน้านี้ต้องการเพียงแค่มุมมองเดียว เพื่อจัดการกับชิ้นงานเดียว ใช่หรือไม่?
├── ใช่ → เรียกใช้งาน Persona โดยตรง
└── ไม่  → ภารกิจย่อยต่างๆ มันแยกขาดจากกัน (ไม่มีการยุ่งเกี่ยวกับ state ร่วมกัน, ไม่มีการบังคับลำดับ) ใช่หรือไม่?
         ├── ใช่ → ใช้ Slash command แบบกระจายกำลังคู่ขนาน fan-out (เช่น /ship)
         └── ไม่  → ให้ผู้ใช้รันแบบ Sequential slash commands เป็นลำดับขั้นเอาเอง (/spec → /plan → /build → /test → /review)
```

## Worked example: valid orchestration (ตัวอย่างที่ถูกต้อง: ท่าประสานงานที่ใช้งานได้)

`/ship` คือท่ากระจายกำลังประสานงานที่เป็นมาตรฐานหลักของ repo นี้:

```
/ship
  ├── (ทำคู่ขนาน) code-reviewer    → เสกรายงานรีวิว
  ├── (ทำคู่ขนาน) security-auditor → เสกรายงานตรวจสอบ
  └── (ทำคู่ขนาน) test-engineer    → เสกรายงานขอบเขต
                  ↓
        merge phase (agent หลักลงมือรวมร่าง)
                  ↓
        ประทับตราตัดสินใจ go/no-go + แผนถอยทัพ
```

ทำไมถึงเวิร์ก:
- ตัว sub-agent แต่ละหน่อ รับสารตั้งต้นเดียวกันเป๊ะแต่ผลิต **มุมมองที่แตกต่างกัน**
- พวกเขาไม่ต้องมารอพึ่งพากันและกัน → มันคือการคู่ขนานที่แท้จริง, ช่วยประหยัดเวลาจริงๆ
- แต่ละหน่อจะรันและสิงในหน้าต่าง(context)ใหม่เอี่ยมของตัวเอง → ทำให้หน้าต่างสนทนาหลักไม่เละเทะ
- ขั้นตอนรวมร่างใช้พลังนิดเดียว และต้องการบริบท(context)แบบครบสูตร, เลยปล่อยให้ agent หลักจัดการไป

## Worked example: invalid orchestration (ตัวอย่างที่พัง: ท่าห้ามทำเด็ดขาด)

ตัวอย่าง persona สาย `meta-orchestrator` ที่ดันมีหน้าที่คอย "ชี้นิ้วสั่งว่าจะเรียก persona ตัวไหนมาทำดี":

```
/work-on-pr → meta-orchestrator
                  ↓ (ชี้ว่า "ไอนี่ต้องมีคนมารีวิว")
              code-reviewer
                  ↓ (ส่งงานกลับ)
              meta-orchestrator (จับมายำคำศัพท์ใหม่)
                  ↓
              user
```

ทำไมถึงบรรลัย:
- ชั้นเลเยอร์ทำหน้าที่แค่ตีบิลส่งต่อ ไม่ได้สร้างคุณค่าอะไรให้เนื้องานเลย
- ต้องทนเจอมันจับมารีไรต์คำศัพท์ใหม่ถึง 2 ต่อ → สารบิดเบือน + กินค่าโทเค็น x2
- ผู้ใช้งานก็รู้อยู่แก่ใจอยู่แล้วว่าอยากให้รีวิว; งั้นปล่อยให้พวกเขาเรียก `/review` เองตรงๆ จะดีกว่า
- เป็นการทำงานซ้ำซ้อนกับสิ่งที่ slash commands และ `AGENTS.md` intent-mapping ช่วยจัดการให้แล้ว

## Rules for personas (กฎบังคับสำหรับตัวละคร personas)

1. 1 Persona มีแค่ 1 บทบาท และผลิตรายงานหน้าตา 1 รูปแบบเท่านั้น ถ้าคุณรู้สึกตัวว่ากำลังยัดบทบาทที่สองให้มัน, จงแยกไปสร้าง persona อีกตัวซะ
2. **Personas ห้ามเรียกใช้งาน personas อื่นเด็ดขาด.** การประกอบร่างดนตรีเป็นหน้าที่ของ slash commands หรือของผู้ใช้เท่านั้น บนหน้าต่างของ Claude Code เองนี่ถือเป็นเรื่องคอขาดบาดตาย — *"subagents ไม่สามารถเบ่งลูก subagents ต่อได้"* — เพราะงั้นกฎข้อนี้มันบังคับให้คุณทำตามอยู่แล้ว
3. Persona สามารถเรียกใช้งาน skills (เรื่องของ *how*) ได้
4. ทุกๆ ไฟล์ของ persona จะต้องตบท้ายบรรทัดด้วยกล่อง "Composition (การนำไปประกอบร่าง)" เสมอ เพื่อบอกว่ามันเอาไปเสียบไว้ตรงไหน

## Claude Code interop (การทำงานร่วมกับ Claude Code)

ตัว Personas ใน repo นี้ถูกออกแบบมาให้ทำหน้าที่เป็น subagents ของ Claude Code และทำหน้าที่เสมือนเป็น Agent Teams teammates ได้เลย โดยไม่ต้องไปแก้ไขดัดแปลงใดๆ:

- **ฐานะ subagents:** จะถูกค้นพบโดยอัตโนมัติเมื่อเสียบใช้งานปลั๊กอิน (ไม่ต้องตั้งค่าระบุตำแหน่ง path เลย) เรียกใช้เครื่องมือ Agent ด้วยท่า `subagent_type: code-reviewer` (หรือ `security-auditor`, `test-engineer`). ตัวอย่างที่คลาสสิกสุดคือ `/ship`
- **ฐานะ Agent Teams teammates** (ของใหม่กำลังทดลอง, ขอดูใบผ่าน `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1`): สามารถเรียกขานชื่อ persona นั้นๆ ตอนที่กำลังคลอดลูกน้องร่วมทีม เนื้อหาของตัว persona จะถูก **นำไปต่อท้าย** system prompt ของเพื่อนร่วมทีมเพื่อเป็นคำสั่งเสริม (ไม่ใช่จับไปแทนที่ของเดิม), ดังนั้นบทบาทตัวละครที่คุณบรรจงเขียนมา จะลอยเด้งอยู่เหนือพวกคำสั่งประสานงานระดับทีมที่หัวหน้าใส่ให้ (พวกคำสั่ง SendMessage, task-list tools, ฯลฯ)

บรรดา Subagents จะคอยทำหน้าที่แค่วิ่งมารายงานผลกลับมายัง agent ตัวหลักเท่านั้น Agent Teams ปล่อยให้เหล่า teammates กระซิบแชตคุยกันเองตรงๆ ได้ ใช้ subagents ในกรณีที่คุณต้องการแค่ผลลัพธ์รายงาน; และให้หยิบ Agent Teams มาใช้ในจังหวะที่ต้องให้เหล่า sub-agents มานั่งเถียงทฤษฎีไขว้กัน (เช่น โหมดจับผิดช่วยกันสืบสวนบั๊ก - competing-hypothesis debugging). เข้าไปดูหน้าตาการจับคู่แบบเต็มๆ ได้ที่ [references/orchestration-patterns.md](../references/orchestration-patterns.md)

พวก Plugin agents จะปิดหูปิดตาเมินเฉยต่อคำสั่ง frontmatter ประเภท `hooks`, `mcpServers`, หรือ `permissionMode` อย่างสิ้นเชิง จงหลีกเลี่ยงการไปพึ่งพาพวกมัน เวลาที่คุณสร้าง personas ตัวใหม่ๆ ไว้ที่นี่

## Adding a new persona (การสร้าง persona ตัวใหม่)

1. สร้างไฟล์ `agents/<role>.md` ให้มีฟอร์แมตหน้าตาหัวเปิด frontmatter เหมือนพวก personas ตัวอื่นๆ
2. ร่ายรายละเอียดหน้าที่ บทบาท, ขอบเขตการดมกลิ่น, รูปแบบการรายงาน, และกฎข้อบังคับให้ครบ
3. ติดป้าย **Composition** (การนำไปประกอบร่าง) ปิดท้ายด้านล่าง (เรียกใช้ตรงๆ เมื่อ / เรียกผ่าน / ห้ามเรียกจาก)
4. เอาชื่อ persona เข้าไปใส่ในตารางแจกแจงพนักงานด้านบนสุดของไฟล์นี้ด้วย
5. ถ้าเกิด persona ตัวใหม่นี้มันดันไปก่อเกิดเป็นท่า orchestrator แบบใหม่ขึ้นมาล่ะก็, ให้ไปบันทึกรูปแบบใหม่นี้ไว้ใน `references/orchestration-patterns.md` แทนที่จะมาบัญญัติรูปแบบแปลกๆ เอาเองในไฟล์ของตัว persona
