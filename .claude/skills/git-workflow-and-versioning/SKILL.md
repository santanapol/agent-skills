# Git Workflow and Versioning

## Overview

Git คือตาข่ายนิรภัยของคุณ จงมองว่า commits คือจุดเซฟเกม, มอง branch คือกระบะทรายทดลองงาน, และมองประวัติการแก้ (history) เป็นดั่งเอกสารคู่มือ ในยุคที่ AI agents เสกโค้ดด้วยความเร็วสูง การควบคุมเวอร์ชันอย่างมีระเบียบวินัยคือกลไกเดียวที่จะช่วยให้การเปลี่ยนแปลงนั้นสามารถจัดการได้, ตรวจทานได้, และย้อนคืนได้

## When to Use

ตลอดเวลา ทุกๆ การเปลี่ยนแปลงของโค้ดจะต้องไหลผ่าน git

## Core Principles (หลักการสำคัญ)

### Trunk-Based Development (Recommended) (แนะนำให้ใช้)

รักษากิ่ง `main` ให้พร้อมนำขึ้น production (deployable) ตลอดเวลา ให้แยกไปพัฒนางานบนฟีเจอร์ branch ที่มีอายุสั้นๆ แล้วรีบนำกลับมา merge รวมให้เสร็จภายใน 1-3 วัน การดอง development branch ไว้นานๆ ถือเป็นต้นทุนแอบแฝง — มันจะเริ่มผิดเพี้ยน, ทำให้เกิด merge conflicts ระโยงระยาง, และทำให้การควบรวมระบบล่าช้า งานวิจัยจาก DORA ชี้ชัดเสมอว่าการทำงานแบบ trunk-based development แปรผันตรงกับทีมวิศวกรรมที่มีประสิทธิภาพสูง

```
main ──●──●──●──●──●──●──●──●──●──  (พร้อม deploy เสมอ)
        ╲      ╱  ╲    ╱
         ●──●─╱    ●──╱    ← ฟีเจอร์ branch อายุสั้น (1-3 วัน)
```

นี่คือค่าตั้งต้นที่แนะนำให้ใช้ สำหรับทีมที่ใช้ gitflow หรือสายดอง branch สามารถนำหลักการเหล่านี้ไปประยุกต์ใช้ได้ (การ commit แบบ atomic, เปลี่ยนแปลงทีละนิด, เขียนข้อความชัดเจน) ตามโมเดลของกิ่งที่ตัวเองมี — ระเบียบวินัยในการ commit สำคัญกว่ากลยุทธ์การแตกกิ่ง branch แบบไหนๆ

- **Dev branches คือต้นทุน** ทุกวันที่ branch ยังอยู่ มันกำลังสะสมความเสี่ยงในการ merge ให้พอกพูนขึ้น
- **Release branches เป็นสิ่งที่ยอมรับได้** เมื่อคุณต้องการแช่แข็งเวอร์ชันที่จะปล่อย (release) ให้เสถียร ในขณะที่กิ่ง main ก็วิ่งหน้าต่อไป
- **ใช้ Feature flags ดีกว่าดอง branch ไว้นานๆ** แนะนำให้นำโค้ดที่ยังทำไม่เสร็จไป deploy ซ่อนไว้หลัง flags ดีกว่าดองงานแช่ไว้ใน branch เป็นอาทิตย์ๆ

### 1. Commit Early, Commit Often (Commit ให้ไว, Commit บ่อยๆ)

ความสำเร็จเล็กๆ แต่ละชิ้นควรได้ commit เป็นของตัวเอง อย่าหมักดองการแก้เยอะๆ ไว้ใน commit เดียว

```
รูปแบบการทำงานที่ถูกต้อง:
  สร้างชิ้นงาน → ทดสอบ → ตรวจสอบซ้ำ → Commit → เริ่มทำชิ้นต่อไป

ไม่ใช่แบบนี้:
  สร้างแม่งให้หมดทุกอย่างเลย → ภาวนาขอให้รันผ่าน → โยนก้อน Commit มหึมา
```

Commits คือจุดเซฟเกม ถ้าการแก้ครั้งต่อไปไปทำอะไรพัง คุณจะได้ย้อนเวลากลับมาหาจุดล่าสุดที่ยังใช้งานได้ทันที

### 2. Atomic Commits (Commit แบบอะตอมมิค)

แต่ละ commit ต้องมีตรรกะเหตุผลของตัวมันเองแค่เรื่องเดียว:

```
# ดี (Good): แต่ละ commit สามารถยืนหยัดจบงานด้วยตัวเอง
git log --oneline
a1b2c3d Add task creation endpoint with validation
d4e5f6g Add task creation form component
h7i8j9k Connect form to API and add loading state
m1n2o3p Add task creation tests (unit + integration)

# แย่ (Bad): ยำรวมมิตรทุกอย่างไว้ด้วยกัน
git log --oneline
x1y2z3a Add task feature, fix sidebar, update deps, refactor utils
```

### 3. Descriptive Messages (ข้อความที่อธิบายได้แจ่มแจ้ง)

ข้อความ Commit ต้องบอกให้รู้ว่า *ทำไปทำไม (why)*, ไม่ใช่แค่ *ทำอะไร (what)*:

```
# ดี (Good): อธิบายความตั้งใจ
feat: add email validation to registration endpoint

Prevents invalid email formats from reaching the database.
Uses Zod schema validation at the route handler level,
consistent with existing validation patterns in auth.ts.

# แย่ (Bad): ไปอธิบายสิ่งที่เราเห็นๆ กันอยู่แล้วจากข้อมูล diff
update auth.ts
```

**รูปแบบ (Format):**
```
<ประเภท>: <คำอธิบายสั้นๆ>

<ส่วนเนื้อหา (ไม่บังคับ) อธิบายว่าทำทำไม, ไม่ใช่อธิบายว่าทำอะไร>
```

**ประเภท (Types):**
- `feat` — ฟีเจอร์ใหม่
- `fix` — แก้บั๊ก
- `refactor` — การแก้โค้ดที่ไม่ได้ไปแก้บั๊ก และไม่ได้เพิ่มฟีเจอร์ใดๆ
- `test` — เพิ่มหรืออัปเดต tests
- `docs` — แก้ไขเฉพาะตัวเอกสาร
- `chore` — การปรับเครื่องมือ tooling, dependencies, ไฟล์ config

### 4. Keep Concerns Separate (แยกเรื่องที่ไม่เกี่ยวกันออกจากกัน)

อย่าเอาการแก้เรื่องฟอร์แมตเว้นวรรค (formatting) ไปปะปนกับการแก้พฤติกรรมการทำงาน (behavior) อย่าเอาการทำ refactor ไปรวมกับการทำฟีเจอร์ การแก้แต่ละประเภทควรจะถูกแยกเป็นคนละ commit — และถ้าจะให้ดีควรจะเป็นคนละ PR ไปเลย:

```
# ดี (Good): แยกเรื่องกันชัดเจน
git commit -m "refactor: extract validation logic to shared utility"
git commit -m "feat: add phone number validation to registration"

# แย่ (Bad): ยำรวมกันเละเทะ
git commit -m "refactor validation and add phone number field"
```

**แยกงาน refactor ออกจากงานเพิ่มฟีเจอร์** โค้ดที่มีการปรับปรุงของเก่าและดันแถมฟีเจอร์ใหม่มาด้วย ถือเป็น 2 การเปลี่ยนแปลง — ให้ส่งแยกกัน การทำแบบนี้จะช่วยให้แต่ละงานง่ายต่อการรีวิว, คืนค่ากลับ (revert), และเข้าใจประวัติความเป็นมา การคลีนอัปเล็กๆ น้อยๆ (เช่น เปลี่ยนชื่อตัวแปร) สามารถรวมอยู่กับ commit งานฟีเจอร์ได้ ขึ้นอยู่กับดุลยพินิจของคนรีวิว

### 5. Size Your Changes (กะขนาดการเปลี่ยนแปลง)

ตั้งเป้าให้แก้ประมาณ ~100 บรรทัดต่อ commit/PR. การแก้เกิน ~1000 บรรทัดควรโดนจับไปหั่นแบ่งใหม่ ไปดูหลักกลยุทธ์การหั่นงานชิ้นใหญ่ที่ทักษะ `code-review-and-quality`

```
แก้ประมาณ ~100 บรรทัด  → ง่ายต่อการรีวิว, ง่ายต่อการย้อนกลับ
แก้ประมาณ ~300 บรรทัด  → พอรับได้ ถ้ามันเป็นเรื่องที่เกี่ยวข้องกันในตรรกะเดียว
แก้ประมาณ ~1000 บรรทัด → จับไปหั่นเป็นชิ้นเล็กๆ ซะ
```

## Branching Strategy (กลยุทธ์การแตกกิ่ง)

### Feature Branches (กิ่งฟีเจอร์)

```
main (พร้อม deploy เสมอ)
  │
  ├── feature/task-creation    ← กิ่งละหนึ่งฟีเจอร์
  ├── feature/user-settings    ← งานที่ทำคู่ขนาน
  └── fix/duplicate-tasks      ← สำหรับแก้บั๊ก
```

- แตกกิ่งเริ่มจาก `main` (หรือชื่อ branch หลักของทีม)
- ดองกิ่งฟีเจอร์ไว้ให้สั้นที่สุด (รวมร่าง merge ภายใน 1-3 วัน) — การดองกิ่งไว้นานๆ คือต้นทุนมหาศาลที่ซ่อนอยู่
- ลบ branch ทิ้งหลัง merge
- พยายามใช้ feature flags แทนการดอง branch ไว้นานๆ สำหรับฟีเจอร์ที่ยังทำไม่เสร็จ

### Branch Naming (การตั้งชื่อกิ่ง)

```
feature/<คำอธิบายสั้นๆ>   → feature/task-creation
fix/<คำอธิบายสั้นๆ>       → fix/duplicate-tasks
chore/<คำอธิบายสั้นๆ>     → chore/update-deps
refactor/<คำอธิบายสั้นๆ>  → refactor/auth-module
```

## Working with Worktrees (การทำงานกับระบบ Worktree)

สำหรับการให้ AI agent หลายตัวทำงานคู่ขนานกันไป, ให้ใช้ระบบ git worktrees เปิดรันหลายๆ branch พร้อมกัน:

```bash
# สร้าง worktree สำหรับ feature branch
git worktree add ../project-feature-a feature/task-creation
git worktree add ../project-feature-b feature/user-settings

# แต่ละ worktree จะสร้างโฟลเดอร์แยกของตัวเอง พร้อมกับเปิด branch นั้นๆ ไว้
# Agents สามารถทำงานพร้อมกันได้โดยไม่ตีกัน
ls ../
  project/              ← กิ่ง main 
  project-feature-a/    ← กิ่ง task-creation 
  project-feature-b/    ← กิ่ง user-settings 

# พองานเสร็จ, กด merge แล้วเคลียร์ทิ้ง
git worktree remove ../project-feature-a
```

ประโยชน์ที่ได้รับ:
- Agent หลายๆ ตัวสามารถช่วยปั่นฟีเจอร์ที่ต่างกันได้ในเวลาเดียวกัน
- ไม่ต้องมานั่งสลับ branch สลับหน้าจอไปมา (แต่ละโฟลเดอร์มี branch เป็นของตัวเอง)
- ถ้าลองรันตัวไหนแล้วพัง, แค่ลบโฟลเดอร์ worktree ทิ้ง — ไม่มีอะไรสูญหาย
- การแก้โค้ดจะถูกตัดขาดจากกัน จนกว่าจะสั่ง merge จริงๆ

## The Save Point Pattern (รูปแบบการใช้จุดเซฟเกม)

```
Agent เริ่มทำงาน
    │
    ├── ลงมือแก้โค้ด
    │   ├── Test รันผ่าน? → กด Commit → ทำงานต่อ
    │   └── Test ไม่ผ่าน? → ย้อนกลับไปหา commit ล่าสุด → นั่งหาสาเหตุ
    │
    ├── ลงมือแก้โค้ดจุดต่อไป
    │   ├── Test รันผ่าน? → กด Commit → ทำงานต่อ
    │   └── Test ไม่ผ่าน? → ย้อนกลับไปหา commit ล่าสุด → นั่งหาสาเหตุ
    │
    └── ฟีเจอร์เสร็จเรียบร้อย → Commits ทุกตัวร้อยเรียงเป็นประวัติที่สะอาดตา
```

รูปแบบนี้แปลว่าคุณจะไม่มีทางสูญเสียงานเกิน 1 increment ถ้า agent เริ่มเพี้ยนทำงานมั่วซั่ว, ให้ใช้คำสั่ง `git reset --hard HEAD` ดึงคุณกลับมาที่จุดเซฟล่าสุดที่ยังใช้การได้

## Change Summaries (บทสรุปการเปลี่ยนแปลง)

หลังจากการแก้โค้ดใดๆ, โปรดให้สรุปรายละเอียดแบบมีโครงสร้างชัดเจนด้วย สิ่งนี้ช่วยให้การรีวิวทำได้ง่ายขึ้น, ช่วยเก็บข้อมูลเชิงขอบเขตการทำงาน (scope discipline), และเผยให้เห็นผลกระทบที่อาจเกิดแบบไม่ได้ตั้งใจ:

```
การเปลี่ยนแปลงที่ทำ (CHANGES MADE):
- src/routes/tasks.ts: เพิ่มด่าน validation middleware เข้าไปในเส้น POST
- src/lib/validation.ts: สร้างตัว TaskCreateSchema โดยใช้แพ็กเกจ Zod

สิ่งที่ตั้งใจไม่เข้าไปแตะ (THINGS I DIDN'T TOUCH (intentionally)):
- src/routes/auth.ts: เห็นว่าช่องโหว่ validation คล้ายๆ กัน แต่มันอยู่นอกเหนือขอบเขตงานรอบนี้
- src/middleware/error.ts: รูปแบบ Error น่าจะปรับปรุงได้อีกนิด (ไปเปิดบั๊กตั้งเรื่องใหม่ดีกว่า)

ข้อกังวลที่อาจจะเกิดขึ้น (POTENTIAL CONCERNS):
- ตัวโครงสร้าง Zod เข้มงวดมาก — ปฏิเสธ fields แปลกปลอมทุกตัว รบกวนยืนยันหน่อยว่านี่คือพฤติกรรมที่ต้องการ
- เพิ่ม zod เข้าไปเป็น dependency (กินที่ 72KB gzipped) — แต่จริงๆ มันมีอยู่แล้วใน package.json
```

รูปแบบนี้ช่วยดักจับความเข้าใจผิดตั้งแต่เนิ่นๆ และเป็นการทำแผนที่นำทางให้คนรีวิวหมวด "DIDN'T TOUCH (สิ่งที่ตั้งใจไม่แตะ)" ถือเป็นของสำคัญมาก — มันโชว์ให้เห็นว่าคุณเคารพในขอบเขตการทำงานของตัวเอง ไม่แอบไปทุบบ้านทำเรื่องที่ไม่มีใครสั่ง

## Pre-Commit Hygiene (สุขอนามัยก่อนกด Commit)

ก่อนที่จะกด commit ทุกครั้ง:

```bash
# 1. เช็คว่าคุณกำลังจะกดเอาอะไรยัดลงไปบ้าง
git diff --staged

# 2. ตรวจสอบให้มั่นใจว่าไม่มีความลับใดๆ ติดเข้าไป
git diff --staged | grep -i "password\|secret\|api_key\|token"

# 3. รัน tests ดู
npm test

# 4. รันระบบกวาดล้าง Linting
npm run lint

# 5. รันระบบตรวจ type (Type checking)
npx tsc --noEmit
```

จับมันยัดเป็นระบบอัตโนมัติด้วย git hooks ซะ:

```json
// package.json (ใช้งาน lint-staged ร่วมกับ husky)
{
  "lint-staged": {
    "*.{ts,tsx}": ["eslint --fix", "prettier --write"],
    "*.{json,md}": ["prettier --write"]
  }
}
```

## Handling Generated Files (การรับมือกับไฟล์ที่ถูกสร้างขึ้นโดยอัตโนมัติ)

- **Commit พวก generated files เข้าไปด้วย** เฉพาะกรณีที่โปรเจกต์ต้องการมัน (เช่น `package-lock.json`, Prisma migrations)
- **ห้าม commit** ผลลัพธ์จากการ build (เช่น `dist/`, `.next/`), ไฟล์ตัวแปรระบบ (`.env`), หรือการตั้งค่าของโปรแกรมเขียนโค้ด IDE (`.vscode/settings.json` เว้นแต่ตกลงจะแชร์กับคนในทีม)
- **ต้องมี `.gitignore`** ที่ครอบคลุม: `node_modules/`, `dist/`, `.env`, `.env.local`, `*.pem`

## Using Git for Debugging (ใช้ Git ช่วยไล่ล่าบั๊ก)

```bash
# ตามล่าหาว่า commit ตัวไหนพาบั๊กเข้ามา
git bisect start
git bisect bad HEAD
git bisect good <known-good-commit>
# Git จะคอยกระโดดไปกลางทางให้; ให้ลองรัน test ดูทุกครั้งที่มันพาไป เพื่อช่วยกันบีบวงแคบหาคนร้าย

# ส่องดูว่าเพิ่งมีอะไรเปลี่ยนไปบ้าง
git log --oneline -20
git diff HEAD~5..HEAD -- src/

# ค้นหาว่าใครเป็นคนแตะบรรทัดนี้คนล่าสุด
git blame src/services/task.ts

# ค้นหาประวัติข้อความ commit ด้วยคีย์เวิร์ด
git log --grep="validation" --oneline
```

## Common Rationalizations (ข้ออ้างทั่วไป)

| ข้ออ้าง | ความเป็นจริง |
|---|---|
| "เดี๋ยวรอฟีเจอร์เสร็จค่อย commit ก็ได้" | commit มหึมาขนาดยักษ์มันยากเกินกว่าจะรีวิว, แกะหาบั๊ก, หรือคืนค่า(revert). จง commit งานเป็นชิ้นเล็กๆ |
| "ข้อความคอมมิตเขียนไปใครจะอ่าน" | ข้อความพวกนี้คือเอกสารคู่มือ อนาคตตัวคุณเอง (และอนาคตของ agents) จะต้องพึ่งพามันเพื่อทำความเข้าใจว่าอะไรเปลี่ยนไปบ้าง และเพื่ออะไร |
| "เดี๋ยวรวบรัด (squash) ทีหลังเอาก็ได้" | การ squash เป็นการล้างประวัติศาสตร์การพัฒนา จงเริ่มต้นเขียนคอมมิตทีละนิด (incremental commits) ให้คลีนตั้งแต่แรก |
| "การแตก branch ภาระเยอะ" | ฟีเจอร์ branch อายุสั้นแทบไม่มีภาระ และยังช่วยป้องกันไม่ให้งานตีกัน การดอง branch ไว้เป็นเวลานานต่างหากคือตัวปัญหา — ต้อง merge ให้จบใน 1-3 วัน |
| "เดี๋ยวย้อนกลับมาหั่นแบ่งการเปลี่ยนแปลงทีหลัง" | งานยักษ์มักจะรีวิวยากกว่า, เสี่ยงตอนนำไป deploy, และเอาคืน (revert) ยาก หั่นก่อนส่ง ไม่ใช่ส่งแล้วค่อยมาหั่น |
| "ผมไม่ต้องการไฟล์ .gitignore หรอก" | ใช่สิ จนกว่าความลับ production ในไฟล์ `.env` จะเผลอหลุดไปอยู่ใน commit นั่นแหละ ตั้งค่ามันเดี๋ยวนี้เลย |

## Red Flags (สัญญาณเตือนภัย)

- มีงานแก้บานตะไทตกค้างสะสมโดยไม่ยอม commit
- ข้อความ commit มีแค่คำหยาบๆ อย่าง "fix", "update", "misc"
- เอาการแก้โค้ดแค่เรื่องเว้นวรรคฟอร์แมต ไปตีปะปนกับการแก้ตรรกะระบบ
- โครงการดันไม่มีไฟล์ `.gitignore`
- ดันไปกด commit ยัด `node_modules/`, `.env`, หรือซากไฟล์ที่เกิดจากการ build เข้ามา
- ดอง branch ซะยืดยาวจนไกลห่างจากความจริงบนสายหลัก (main)
- ใช้ท่างัดข้อ Force-pushing กับ branch สาธารณะที่มีคนอื่นใช้ร่วมด้วย

## Verification

สำหรับทุกๆ การ commit:

- [ ] Commit จัดการงานเพียงตรรกะเดียวเท่านั้น
- [ ] ข้อความบอกว่า "ทำไปทำไม", เขียนล้อตามธรรมเนียมการตั้งคำนำหน้าประเภทงาน
- [ ] รัน Tests ผ่านก่อนที่จะกด commit
- [ ] ไม่มีความลับหลุดติดไปในไฟล์ diff
- [ ] ไม่เอาเรื่องการแก้ฟอร์แมตเว้นวรรคล้วนๆ ไปยำรวมกับการแก้พฤติกรรมโค้ด
- [ ] ไฟล์ `.gitignore` ระบุขอบเขตสิ่งที่ไม่ควรรวมครบถ้วน
