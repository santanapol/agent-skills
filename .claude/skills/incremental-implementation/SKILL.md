# Incremental Implementation

## Overview

สร้างงานเป็นชิ้นสไลซ์แนวตั้งบางๆ (thin vertical slices) — ลงมือทำแค่หนึ่งชิ้น, ทดสอบมัน, ยืนยันผล, แล้วจึงค่อยขยับขยายไปชิ้นต่อไป หลีกเลี่ยงการสร้างฟีเจอร์ทั้งหมดรวดเดียวจบ แต่ละส่วนที่เพิ่มเข้าไปต้องทิ้งระบบไว้ในสถานะที่ยังคงทำงานและทดสอบได้ นี่คือวินัยการทำงานที่ทำให้ฟีเจอร์ขนาดใหญ่สามารถจัดการได้ง่ายขึ้น

## When to Use

- เมื่อต้องการแก้ไขโค้ดที่มีผลกระทบหลายไฟล์
- เมื่อต้องสร้างฟีเจอร์ใหม่จาก Task ที่ถูกแตกไว้
- เมื่อต้องการ Refactoring โค้ดเดิม
- เมื่อใดก็ตามที่คุณรู้สึกอยากจะเขียนโค้ดรวดเดียวเกิน ~100 บรรทัดโดยไม่ลองรันทดสอบ

**เมื่อใดที่ไม่ควรใช้:** การแก้แบบจิ๊บจ๊อยไฟล์เดียว หรือฟังก์ชันเดียวที่มีขอบเขตเล็กมากๆ อยู่แล้ว

## The Increment Cycle (วงจรการเพิ่มทีละส่วน)

```
┌──────────────────────────────────────┐
│                                      │
│ ลงมือทำ(Implement) ──→ Test ──→ Verify(ยืนยันผล) ──┐  │
│       ▲                                            │  │
│       └───── Commit(บันทึก) ◄──────────────────────┘  │
│              │                                     │
│              ▼                                     │
│        เริ่มทำชิ้นต่อไป (Next slice)                 │
│                                                    │
└──────────────────────────────────────┘
```

สำหรับงานแต่ละชิ้นสไลซ์ (Slice):

1. **Implement (ลงมือทำ)** สร้างชิ้นส่วนที่เล็กที่สุดที่จะทำงานได้จริงแบบสมบูรณ์
2. **Test (ทดสอบ)** — สั่งรันชุดทดสอบ (หรือเขียน Test ขึ้นมาถ้ามันยังไม่มี)
3. **Verify (ยืนยันผล)** — ยืนยันว่างานชิ้นนั้นรันผ่านตามคาด (Test ผ่าน, Build ขึ้น, และเช็คด้วยตาผ่าน)
4. **Commit (บันทึก)** -- เซฟความคืบหน้าพร้อมเขียนคำอธิบายให้ดี (ดูที่ `.claude/skills/git-workflow-and-versioning/SKILL.md` สำหรับวิธีเขียน commit ที่ถูกต้อง)
5. **Move to the next slice (ขยับไปทำชิ้นต่อไป)** — เดินหน้าต่อ อย่าทำไปลบทำใหม่

## Slicing Strategies (กลยุทธ์การหั่นงาน)

### Vertical Slices (หั่นตามแนวตั้ง - แนะนำที่สุด)

สร้างเส้นทางการทำงาน (Path) ให้เสร็จสมบูรณ์ทะลุทั้งระบบ:

```
สไลซ์ 1: สร้าง Task (ฐานข้อมูล + API + หน้า UI เรียบง่าย)
    → Test ผ่าน, ผู้ใช้สร้าง Task ผ่าน UI ได้จริง

สไลซ์ 2: ดึงรายการ Task มาแสดง (ดึงข้อมูล + API + UI)
    → Test ผ่าน, ผู้ใช้เห็นรายการ Task ของตัวเองได้จริง

สไลซ์ 3: แก้ไข Task (อัปเดตข้อมูล + API + UI)
    → Test ผ่าน, ผู้ใช้ปรับแก้ Task ได้

สไลซ์ 4: ลบ Task (ลบข้อมูล + API + UI + ป๊อปอัปยืนยัน)
    → Test ผ่าน, ระบบจัดการฐานข้อมูล (CRUD) ทำงานสมบูรณ์
```

สไลซ์แต่ละชิ้นจะเป็นการส่งมอบฟังก์ชันการทำงานที่ใช้งานได้จริงตั้งแต่ต้นจนจบ

### Contract-First Slicing (ทำข้อตกลงกันก่อน)

เมื่อฝ่ายหลังบ้าน (Backend) และหน้าบ้าน (Frontend) ต้องทำงานขนานกันไปพร้อมๆ กัน:

```
สไลซ์ 0: กำหนดข้อตกลง API (สร้าง Types, interfaces, หรือสเปกของ OpenAPI)
สไลซ์ 1a: สร้าง Backend ตามข้อตกลงที่คุยไว้ + เขียน API tests
สไลซ์ 1b: สร้าง Frontend โดยอิงข้อมูลปลอม (Mock) ที่หน้าตาตรงตามข้อตกลงเป๊ะ
สไลซ์ 2: นำทั้งสองส่วนมาประกอบกัน แล้วเทสตั้งแต่ต้นจนจบ
```

### Risk-First Slicing (ทำส่วนที่เสี่ยงที่สุดก่อน)

จัดการกับส่วนที่เสี่ยงที่สุด หรือส่วนที่คุณกังวลและรู้ข้อมูลน้อยที่สุดก่อน:

```
สไลซ์ 1: พิสูจน์ให้ได้ก่อนว่าการเชื่อมต่อ WebSocket เสถียร (ความเสี่ยงสูงสุด)
สไลซ์ 2: สร้างระบบอัปเดตแบบเรียลไทม์ โดยใช้การเชื่อมต่อที่พิสูจน์แล้วจากข้อแรก
สไลซ์ 3: เพิ่มระบบรองรับตอนเน็ตหลุด (offline support) และการต่อเน็ตใหม่
```

ถ้าสไลซ์ 1 ดันพัง คุณจะได้รู้ตัวก่อนที่จะเสียเวลาลงทุนไปสร้างสไลซ์ 2 และ 3

## Implementation Rules (กฎการลงมือสร้าง)

### Rule 0: Simplicity First (ความเรียบง่ายต้องมาก่อน)

ก่อนจะเขียนโค้ดบรรทัดแรก ให้ถามตัวเองว่า: "อะไรคือวิธีที่ง่ายที่สุด ที่มันจะเวิร์ก?"

หลังจากเขียนโค้ดเสร็จ ให้ทบทวนมันด้วยคำถามพวกนี้:
- โค้ดนี้สามารถหดให้สั้นลงกว่านี้ได้ไหม?
- ความซับซ้อน (abstractions) ที่ใส่มา มันคุ้มค่าพอที่จะอยู่ตรงนี้ไหม?
- หากมีวิศวกรระดับ Staff มาดูโค้ดนี้ เขาจะพูดไหมว่า "ทำไมคุณไม่แค่..."?
- ฉันกำลังสร้างโค้ดเพื่อรองรับอนาคตที่ยังไม่เกิด หรือกำลังแก้ปัญหางานตรงหน้าอยู่กันแน่?

```
SIMPLICITY CHECK:
✗ สร้าง EventBus ครอบจักรวาลพร้อมระบบ middleware แค่เพื่อส่งแจ้งเตือน 1 ตัว
✓ สร้างฟังก์ชันโง่ๆ เรียกใช้ธรรมดา

✗ ใช้ Abstract factory pattern กับแค่คอมโพเนนต์สองตัวที่คล้ายๆ กัน
✓ สร้างคอมโพเนนต์ทื่อๆ 2 ตัว แล้วแยก Utilities ที่ใช้ร่วมกันออกมาต่างหาก

✗ สร้างเครื่องมือลากวางทำฟอร์ม (Config-driven form builder) เพื่อทำฟอร์มแค่ 3 หน้า
✓ สร้างคอมโพเนนต์ฟอร์มทื่อๆ ไปเลย 3 ตัว
```

โค้ดที่หน้าตาคล้ายกันซ้ำๆ กัน 3 บรรทัด ย่อมดีกว่าการรีบทำ Abstraction ล่วงหน้าที่ซับซ้อนเกินไป จงลงมือเขียนโค้ดด้วยวิธีบ้านๆ และได้ผลลัพธ์ชัดเจนก่อนเป็นอันดับแรก การทำ Optimize ควรทำก็ต่อเมื่อพิสูจน์แล้วว่าโค้ดนั้นรัน Test ผ่าน

### Rule 0.5: Scope Discipline (รักษาวินัยเรื่องขอบเขต)

แตะต้องเฉพาะไฟล์ที่ Task งานนั้นบังคับให้แตะ

ห้ามทำ:
- "ล้างโค้ด (Clean up)" ในไฟล์ข้างเคียงที่ไม่เกี่ยวกับงานคุณ
- จัดระเบียบการ Import ในไฟล์ที่คุณไม่ได้มีหน้าที่แก้ไข
- ลบ Comment ที่คุณไม่เข้าใจ
- เพิ่มฟีเจอร์ที่ไม่ได้ระบุใน Spec เพียงเพราะ "เห็นว่ามันน่าจะมีประโยชน์"
- ปรับโค้ดในไฟล์ที่คุณแค่มีหน้าที่เปิดอ่าน ให้ใช้ Syntax ทันสมัยขึ้น

หากคุณเจออะไรบางอย่างที่ควรปรับปรุง แต่อยู่นอกขอบเขตงานของคุณ ให้แค่จดโน้ตไว้ — ห้ามแอบไปแก้:

```
สิ่งที่สังเกตเห็น แต่จะยังไม่แตะต้อง (NOTICED BUT NOT TOUCHING):
- ไฟล์ src/utils/format.ts มีการ import ของที่ไม่ได้ใช้ (ไม่เกี่ยวกับงานเรารอบนี้)
- ตัว auth middleware ควรปรับแก้ข้อความ error ให้ดีกว่านี้ (จดไว้เป็นงานแยก)
→ อยากให้ฉันสร้าง Task เพื่อไปตามแก้เรื่องพวกนี้ทีหลังไหม?
```

### Rule 1: One Thing at a Time (ทำทีละอย่าง)

การเพิ่มชิ้นงานแต่ละครั้ง ต้องเป็นการแก้ไขตรรกะแค่เรื่องเดียว อย่าผสมเรื่องกัน:

**สิ่งที่แย่:** Commit เดียวที่ทั้งสร้างคอมโพเนนต์ใหม่, รื้อ (refactors) อันเดิม, และดันไปแก้ build config ด้วย
**สิ่งที่ดี:** แบ่งเป็น 3 Commits — ทำ Commit แยกให้ความเปลี่ยนแปลงแต่ละอัน

### Rule 2: Keep It Compilable (ทำให้ระบบรันขึ้นเสมอ)

หลังจบการเพิ่มชิ้นงานในแต่ละรอบ โปรเจกต์จะต้อง Build ผ่าน และ Test ที่มีอยู่ทั้งหมดต้องรันผ่าน อย่าทิ้ง Codebase ให้อยู่ในสภาพพังๆ ค้างไว้ระหว่างสไลซ์

### Rule 3: Feature Flags for Incomplete Features (เปิดสวิตช์ฟีเจอร์สำหรับงานที่ยังไม่เสร็จ)

หากฟีเจอร์นั้นยังไม่พร้อมให้ผู้ใช้เห็น แต่คุณจำเป็นต้อง Merge เข้าไปเพื่อเซฟความคืบหน้า:

```typescript
// สวิตช์ฟีเจอร์สำหรับงานที่ทำค้างไว้ (Feature flag for work-in-progress)
const ENABLE_TASK_SHARING = process.env.FEATURE_TASK_SHARING === 'true';

if (ENABLE_TASK_SHARING) {
  // UI การแชร์ใหม่
}
```

สิ่งนี้จะช่วยให้คุณสามารถทยอย Merge โค้ดชิ้นเล็กๆ เข้าสู่ Branch หลักได้ โดยไม่ต้องกลัวว่าผู้ใช้จะเห็นฟีเจอร์พังๆ

### Rule 4: Safe Defaults (ตั้งค่าเริ่มต้นให้ปลอดภัย)

โค้ดชุดใหม่ที่เพิ่มเข้าไป ควรถูกตั้งค่าเริ่มต้น (default) ให้ทำงานแบบระมัดระวังที่สุด:

```typescript
// ปลอดภัย: ปิดการใช้งานเป็นค่าเริ่มต้น ต้องถูกสั่งเปิดเท่านั้น (opt-in)
export function createTask(data: TaskInput, options?: { notify?: boolean }) {
  const shouldNotify = options?.notify ?? false;
  // ...
}
```

### Rule 5: Rollback-Friendly (พร้อมจะถอยกลับเสมอ)

ชิ้นงานที่ถูกเพิ่มแต่ละรอบ จะต้องสามารถกด Revert ถอยหลังกลับได้แบบชิ้นใครชิ้นมัน:

- การแก้ไขแบบ "เพิ่มเติม" (ไฟล์ใหม่, ฟังก์ชันใหม่) มักจะ Revert ได้ง่าย
- การปรับเปลี่ยนโค้ดดั้งเดิมที่มีอยู่ ควรทำให้เกิดผลกระทบน้อยและจำกัดขอบเขตที่สุด
- การแก้ไข Database migrations ควรจะมีโค้ดสำหรับการ rollback เตรียมพร้อมไว้คู่กันเสมอ
- หลีกเลี่ยงการลบบางสิ่งทิ้งแล้วเอาของใหม่ไปแทนที่ ภายใน Commit เดียวกัน — ควรแยก Commit กัน

## Working with Agents (การทำงานกับ Agents)

เวลาที่คุณจะสั่ง Agent ให้ใช้วิธีทยอยทำ (Implement incrementally):

```
"มาเริ่มทำ Task 3 จากแผนกันเถอะ

ให้เริ่มจากแก้ Database schema และทำ API endpoint มาก่อนนะ
ยังไม่ต้องแตะหน้า UI — เดี๋ยวเราค่อยทำในรอบหน้า

หลังจากทำเสร็จ ให้รัน `npm test` และ `npm run build` เพื่อตรวจสอบด้วยนะ
ว่าระบบไม่ได้พัง"
```

จงระบุให้ชัดเจนเสมอว่า ขอบเขตงานรอบนี้คืออะไร และ อะไรที่ไม่ใช่ขอบเขต

## Increment Checklist (เช็คลิสต์ก่อนจบแต่ละรอบ)

หลังจบแต่ละสไลซ์ ยืนยันว่า:

- [ ] การเปลี่ยนแปลงนี้ทำเพียง 1 เป้าหมาย และทำมันเสร็จสมบูรณ์ตามเป้า
- [ ] Test เดิมที่มีอยู่ทั้งหมดยังรันผ่านปกติ (`npm test`)
- [ ] สามารถรัน Build โปรเจกต์ขึ้น (`npm run build`)
- [ ] ประเภทตัวแปร (Type) ตรวจสอบผ่าน (`npx tsc --noEmit`)
- [ ] ล้างโค้ด (Linting) แล้วผ่านการตรวจ (`npm run lint`)
- [ ] ฟังก์ชันที่เพิ่งเพิ่มเข้ามาสามารถใช้งานได้จริงตามคาด
- [ ] โค้ดถูก Commit พร้อมคำอธิบายที่สื่อความหมายชัดเจน

**หมายเหตุ:** สั่งรันคำสั่งเหล่านี้ก็ต่อเมื่อโค้ดมีการเปลี่ยนแปลงที่อาจส่งผลกระทบเท่านั้น หากเพิ่งสั่งรันผ่านไป อย่ารันซ้ำเดิมเว้นแต่มีการแก้ไขโค้ดเพิ่ม — การรันซ้ำบนโค้ดที่ไม่ได้แก้ ไม่ได้ทำให้รู้อะไรเพิ่มขึ้น

## Common Rationalizations (ข้ออ้างทั่วไป)

| ข้ออ้าง | ความเป็นจริง |
|---|---|
| "เดี๋ยวฉันค่อยไปเทสทีเดียวตอนจบเลย" | บั๊กมันสะสมทับถมได้ บั๊กในสไลซ์ที่ 1 จะทำให้สไลซ์ที่ 2-5 ผิดไปด้วย เทสมันทุกๆ สไลซ์แหละดีแล้ว |
| "รวบทำทีเดียวเลยมันเร็วกว่า" | มัน *รู้สึก* ว่าเร็วกว่า จนกระทั่งบางอย่างพัง แล้วคุณก็หาไม่เจอว่าโค้ด 500 บรรทัดที่เปลี่ยนไป บรรทัดไหนที่เป็นตัวการ |
| "การแก้งานมันเล็กเกินไปกว่าที่จะแยก Commit" | การ Commit เล็กๆ มันฟรีไม่มีค่าใช้จ่ายนะ การทำ Commit ใหญ่เบ้อเร่อมันซ่อนบั๊กไว้ข้างใน และทำให้การย้อนกลับเป็นนรก |
| "เดี๋ยวฉันค่อยไปใส่สวิตช์ Feature flag ทีหลัง" | ถ้าฟีเจอร์ยังไม่เสร็จ มันก็ไม่สมควรให้ผู้ใช้เห็น ใส่ flag ตั้งแต่ตอนนี้เลย |

## Red Flags (สัญญาณเตือนภัย)

- เขียนโค้ดรวดเดียวเกิน 100 บรรทัดโดยไม่มีการกดรันทดสอบเลย
- แอบแก้ไขอะไรที่ไม่เกี่ยวข้องกันหลายอย่างยัดลงไปในการทำสไลซ์เดียว
- ขอบเขตงานเริ่มงอกแบบ "ไหนๆ ก็ทำแล้ว ขอฉันเพิ่มอันนี้ด้วยละกัน"
- ข้ามขั้นตอน Test/Verify เพียงเพราะอยากให้งานเสร็จไวขึ้น
- ระหว่างสไลซ์ ดันเผลอทำ Build หรือ Tests พังค้างไว้
- หมักหมมโค้ดใหม่จำนวนมหาศาลไว้โดยไม่ยอมสั่ง Commit
- รีบสร้างความซับซ้อน (abstractions) ทั้งๆ ที่ยังไม่ได้ใช้เกิน 3 เคสด้วยซ้ำ
- แอบไปแตะไฟล์อยู่นอกเขตแดนของ Task ตัวเอง เพียงเพราะอ้างว่า "ก็ผ่านมาเห็นพอดี"
- สร้างไฟล์ Utility กลางใหม่เพื่อรองรับการใช้งานแค่ครั้งเดียว (One-time operations)
- สั่งรันคำสั่ง Build/Test ซ้ำติดกันสองรอบโดยไม่มีการแก้โค้ดอะไรคั่นกลางเลย
