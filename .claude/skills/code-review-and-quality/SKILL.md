# Code Review and Quality

## Overview

การรีวิวโค้ดแบบหลายมิติที่มาพร้อมด่านตรวจคุณภาพ การเปลี่ยนแปลงทุกอย่างต้องผ่านการรีวิวก่อน merge เสมอ — ไม่มีข้อยกเว้น การรีวิวจะครอบคลุม 5 แกนหลัก ได้แก่: ความถูกต้อง, ความอ่านง่าย, สถาปัตยกรรม, ความปลอดภัย, และประสิทธิภาพ

**มาตรฐานการให้ผ่าน (The approval standard):** ให้ผ่าน (Approve) เมื่อการเปลี่ยนแปลงนั้นทำให้สุขภาพโดยรวมของโค้ดเบสดีขึ้นอย่างแน่นอน แม้ว่ามันจะไม่เพอร์เฟกต์ก็ตาม โค้ดที่สมบูรณ์แบบไม่มีอยู่จริง — เป้าหมายคือการพัฒนาให้ดีขึ้นอย่างต่อเนื่อง อย่าเตะถ่วงขัดขวางการแก้โค้ด เพียงเพราะเขายังไม่ได้เขียนในแบบที่คุณอยากให้เป็น ถ้ามันทำให้โค้ดโดยรวมดีขึ้นและทำตามธรรมเนียมของโปรเจกต์ ก็จงให้มันผ่านไป

## When to Use

- ก่อนทำการ Merge PR หรือความเปลี่ยนแปลงใดๆ
- หลังจากเขียนฟีเจอร์เสร็จเรียบร้อย
- เมื่อ Agent ตัวอื่นหรือโมเดลตัวอื่นผลิตโค้ดที่คุณต้องประเมิน
- เมื่อทำ Refactor โค้ดเก่า
- หลังการแก้ไขบั๊กทุกครั้ง (ให้รีวิวทั้งตัวโค้ดแก้บั๊ก และตัว regression test)

## The Five-Axis Review (การรีวิว 5 แกน)

การรีวิวทุกครั้งต้องประเมินโค้ดผ่านมิติเหล่านี้:

### 1. Correctness (ความถูกต้อง)

โค้ดทำงานได้ตามที่มันอ้างไว้ไหม?

- ทำงานตรงตาม spec หรือ requirements ไหม?
- มีการดักจับเคสขอบหลืบ (edge cases) ไหม (เช่น ค่า null, ค่าว่าง, ตัวเลขสุดขอบ)?
- มีการจัดการเส้นทาง Error ไหม (ไม่ใช่สนแต่ทางที่รันผ่านสบายๆ happy path)?
- รันผ่านทุก Tests ไหม? แล้วตัว Tests นั้นทดสอบได้ถูกจุดจริงๆ หรือเปล่า?
- มีบั๊กคำนวณเลขผิดไปหนึ่งแต้ม (off-by-one errors), race conditions, หรือสถานะขัดแย้งกันไหม?

### 2. Readability & Simplicity (ความอ่านง่าย & เรียบง่าย)

วิศวกรคนอื่น (หรือ Agent) สามารถเข้าใจโค้ดนี้ได้โดยไม่ต้องให้คนเขียนมานั่งอธิบายไหม?

- ชื่อตัวแปรสื่อความหมายและเป็นไปตามธรรมเนียมโปรเจกต์ไหม? (ห้ามใช้ชื่อตั้งลวกๆ อย่าง `temp`, `data`, `result` โดยไม่มีบริบทบอก)
- ลำดับการทำงาน (control flow) ตรงไปตรงมาไหม (หลีกเลี่ยงการเขียนเงื่อนไขซ้อนกันลึกๆ (nested ternaries) หรือ deep callbacks)?
- โค้ดถูกจัดระเบียบอย่างสมเหตุสมผลไหม (โค้ดกลุ่มเดียวกันอยู่ด้วยกัน, แบ่งขอบเขต module ชัดเจน)?
- มีทริคการเขียนที่ดูล้ำๆ "ฉลาดๆ" ซึ่งควรจะแก้ให้มันเข้าใจง่ายขึ้นไหม?
- **ทำให้สั้นลงกว่านี้ได้ไหม?** (โค้ด 1000 บรรทัด แต่จริงๆ เขียนแค่ 100 บรรทัดก็พอ ถือเป็นความล้มเหลว)
- **การสร้าง Abstractions มันคุ้มค่ากับความซับซ้อนที่เพิ่มขึ้นไหม?** (อย่าด่วนสร้างของครอบจักรวาล จนกว่าจะได้เจอกรณีใช้งานจริงๆ เป็นครั้งที่ 3)
- การใส่คอมเมนต์จะช่วยอธิบายเจตนาที่ดูไม่ออกได้ไหม? (แต่อย่าเขียนคอมเมนต์อธิบายโค้ดที่เห็นแล้วเข้าใจได้อยู่แล้ว)
- มีซากโค้ดตายๆ อยู่ไหม: ตัวแปรที่ประกาศแล้วไม่ได้ใช้ (`_unused`), โค้ดทำเผื่อเพื่อรองรับของเก่า, หรือคอมเมนต์ประเภท `// removed`?

### 3. Architecture (สถาปัตยกรรม)

การเปลี่ยนแปลงครั้งนี้เข้ากันกับการออกแบบของระบบไหม?

- มันเขียนตามแพทเทิร์นเดิมที่มีอยู่ หรือว่ามันสร้างแพทเทิร์นใหม่ขึ้นมา? ถ้าสร้างใหม่, มันสมเหตุสมผลไหม?
- ยังรักษาความชัดเจนของขอบเขตโมดูล (module boundaries) อยู่ไหม?
- มีโค้ดซ้ำซ้อนที่ควรจะดึงไปแชร์ใช้ร่วมกันไหม?
- การพึ่งพากัน (dependencies) ไหลไปในทิศทางที่ถูกต้องไหม (ต้องไม่มีการพึ่งพากันเป็นวงกลม circular dependencies)?
- ระดับความลึกของการทำ Abstraction เหมาะสมไหม (ไม่ over-engineered เกินไป, ไม่เอามาผูกมัดกันจนแกะไม่ออก)?

### 4. Security (ความปลอดภัย)

สำหรับคำแนะนำแบบเจาะลึกเรื่องความปลอดภัย ให้ดูทักษะ `security-and-hardening` การเปลี่ยนแปลงครั้งนี้นำพาช่องโหว่เข้ามาด้วยไหม?

- ข้อมูลจากผู้ใช้ (user input) ถูกตรวจสอบและทำความสะอาดแล้วไหม?
- เก็บความลับต่างๆ ให้ออกห่างจาก โค้ด, ล็อกการทำงาน(logs), และระบบจัดเก็บเวอร์ชัน(version control) ไหม?
- มีการตรวจเช็คระบบล็อกอิน/การอนุญาตใช้งาน (authentication/authorization) ตามจุดที่จำเป็นไหม?
- คำสั่ง SQL ถูกเตรียมข้อมูลแบบ parameterize ไว้ไหม (ห้ามเอาสตริงมาต่อกันตรงๆ เด็ดขาด)?
- ข้อมูลตอนแสดงผลถูกเข้ารหัส (encoded) ป้องกันการโจมตีแบบ XSS ไหม?
- dependencies ต่างๆ มาจากแหล่งที่เชื่อถือได้และไม่มีช่องโหว่ร้ายแรงไหม?
- ข้อมูลจากแหล่งภายนอก (APIs, logs, เนื้อหาจากผู้ใช้, ไฟล์ตั้งค่า config) ถูกมองว่าเป็นสิ่งอันตรายที่ไม่น่าไว้ใจไหม?
- เส้นทางข้อมูลภายนอกถูกตรวจสอบ ณ จุดพรมแดนระบบ ก่อนจะนำไปใช้คำนวณตรรกะหรือหน้าจอแสดงผลไหม?

### 5. Performance (ประสิทธิภาพ)

สำหรับเรื่องการจับเวลาโปรไฟล์ไฟล์ิ่ง(profiling) และการปรับแต่ง (optimization) แบบเจาะลึก, ดูที่ `performance-optimization` การเปลี่ยนแปลงนี้นำพาปัญหาเรื่องความอืดมาด้วยไหม?

- มีแพทเทิร์น N+1 query อยู่ไหม?
- มีลูปที่วนไม่สิ้นสุด หรือการดึงข้อมูลที่ไม่ได้จำกัดลิมิตไหม?
- มีคำสั่งที่ควรจะทำแบบ asynchronous แต่ดันเขียนให้ทำงานแบบ synchronous ไหม?
- ใน UI components มีจังหวะการเรนเดอร์หน้าจอซ้ำซ้อนที่ไม่จำเป็น (unnecessary re-renders) ไหม?
- ขาดหายการทำแบ่งหน้า (pagination) ใน endpoints ที่ส่งข้อมูลเป็นรายชื่อไหม?
- มีการสร้างอ็อบเจกต์ใหญ่เบ้อเริ่มในเส้นทางที่มีการเรียกใช้งานบ่อยๆ (hot paths) ไหม?

## Change Sizing (การประเมินขนาดการเปลี่ยนแปลง)

การเปลี่ยนแปลงเล็กๆ ที่โฟกัสตรงจุด จะง่ายต่อการรีวิว, เอาไปรวม (merge) ได้เร็ว, และปลอดภัยต่อการเอาขึ้น production (deploy) กะขนาดให้อยู่ในช่วงนี้:

```
แก้ประมาณ ~100 บรรทัด   → ดีเลย. สามารถนั่งรีวิวรวดเดียวจบ
แก้ประมาณ ~300 บรรทัด   → พอรับได้ ถ้ามันเป็นเรื่องที่เกี่ยวข้องกันในตรรกะเดียว
แก้ประมาณ ~1000 บรรทัด  → ใหญ่เกินไป. ไปหั่นแบ่งมาใหม่
```

**สิ่งที่นับว่าเป็น "หนึ่งการเปลี่ยนแปลง":** การปรับเปลี่ยนที่สามารถอยู่ได้ด้วยตัวของมันเองและแก้ปัญหาได้ 1 เรื่อง, รวม Tests ที่เกี่ยวข้อง, และระบบยังสามารถรันต่อได้หลังจากการบันทึก(submission) เป็นแค่ส่วนหนึ่งของฟีเจอร์ — ไม่ใช่เอาทั้งฟีเจอร์รวมมิตรมาทีเดียว

**กลยุทธ์การหั่น เมื่อการเปลี่ยนแปลงมีขนาดใหญ่เกินไป:**

| กลยุทธ์ | ทำอย่างไร | ใช้เมื่อไหร่ |
|----------|-----|------|
| **ทับซ้อน (Stack)** | ส่งการเปลี่ยนแปลงชุดเล็ก แล้วเริ่มทำชุดต่อไปโดยอิงจากชุดนั้น | งานที่ต้องทำตามลำดับขั้นตอน |
| **จัดกลุ่มไฟล์** | แยกเรื่องการแก้ไขออกเป็นกลุ่มๆ สำหรับคนรีวิวที่ต่างความถนัดกัน | งานที่ส่งผลกระทบข้ามสายงาน |
| **แนวนอน (Horizontal)** | สร้างโครงหรือของใช้ร่วมกันขึ้นมาก่อน แล้วค่อยสร้างตัวมาเรียกใช้ทีหลัง | สถาปัตยกรรมแบบแบ่งเลเยอร์ (Layered) |
| **แนวตั้ง (Vertical)** | หั่นฟีเจอร์ให้เป็นชิ้นเล็กๆ ที่ทำงานครบทะลุได้ตั้งแต่หน้าบ้านถึงหลังบ้าน (full-stack) | งานทำฟีเจอร์ |

**เมื่อใดที่การเปลี่ยนแปลงใหญ่ๆ พออนุโลมได้:** การลบไฟล์ทิ้งทีละเยอะๆ หรือการทำ automated refactoring ที่คนรีวิวแค่เช็คเป้าหมายเจตนารมณ์ ไม่จำเป็นต้องนั่งไล่ดูทีละบรรทัด

**จงแยกงาน Refactor ออกจากงานเพิ่มฟีเจอร์.** โค้ดที่มีการปรับปรุงของเก่าและดันแถมฟีเจอร์ใหม่มาด้วย ถือเป็น 2 การเปลี่ยนแปลง — ให้ส่งแยกกัน การคลีนอัปเล็กๆ น้อยๆ (เช่น เปลี่ยนชื่อตัวแปร) สามารถรวมอยู่ด้วยกันได้ ขึ้นอยู่กับดุลยพินิจของคนรีวิว

## Change Descriptions (คำอธิบายการเปลี่ยนแปลง)

ทุกๆ การเปลี่ยนแปลงต้องมีคำอธิบายที่สามารถยืนหยัดด้วยตัวเองได้ในประวัติของ version control

**บรรทัดแรก:** สั้น, เป็นประโยคคำสั่ง, อ่านเข้าใจด้วยตัวมันเอง "Delete the FizzBuzz RPC" ไม่ใช่ "Deleting the FizzBuzz RPC." ต้องมีข้อมูลมากพอที่คนมาค้นหาประวัติย้อนหลังจะสามารถเข้าใจว่าแก้อะไร โดยไม่ต้องกดเข้าไปดูตัวโค้ด Diff เลย

**เนื้อหา (Body):** อะไรถูกเปลี่ยน และ เปลี่ยนทำไม ใส่บริบท, การตัดสินใจ, และเหตุผลที่มองไม่เห็นได้จากในตัวโค้ด ใส่ลิงก์ไปยังหมายเลขบั๊ก, ผลการรันความเร็ว, หรือเอกสารดีไซน์ที่เกี่ยวข้อง ยอมรับชี้แจงข้อบกพร่องในวิธีการที่คุณเลือก หากมันมีอยู่จริง

**พฤติกรรมยอดแย่:** "Fix bug," "Fix build," "Add patch," "Moving code from A to B," "Phase 1," "Add convenience functions."

## Review Process (ขั้นตอนการรีวิว)

### Step 1: Understand the Context (เข้าใจบริบท)

ก่อนจะก้มหน้ามองโค้ด, ให้เข้าใจเจตนาเป้าหมายก่อน:

```
- งานนี้ทำขึ้นมาเพื่อเป้าหมายอะไร?
- มันกำลังทำตาม spec หรือ task ไหน?
- พฤติกรรมที่ควรจะเปลี่ยนแปลงไปคืออะไร?
```

### Step 2: Review the Tests First (ดู Tests ก่อน)

Tests จะเผยให้เห็นเจตนาและความครอบคลุม:

```
- มี Tests เขียนเตรียมไว้สำหรับการเปลี่ยนแปลงนี้ไหม?
- พวกมันทดสอบพฤติกรรมจริงๆ ใช่ไหม (ไม่ใช่ทดสอบลึกไปยันเรื่องของตัวโค้ดที่ซ่อนอยู่ข้างใน)?
- ครอบคลุมเคสแปลกๆ (edge cases) หรือยัง?
- Tests ตั้งชื่อได้อธิบายชัดเจนไหม?
- ถ้าระบบพัง Tests พวกนี้จะดักจับอาการเดิมที่กลับมาเป็นอีกรอบ (regression) ได้ไหม?
```

### Step 3: Review the Implementation (ดูตัวโค้ด)

เดินหน้าตรวจโค้ด โดยพกหลักพิจารณาทั้ง 5 แกนไว้ในใจเสมอ:

```
สำหรับทุกๆ ไฟล์ที่เปลี่ยน:
1. ความถูกต้อง (Correctness): โค้ดนี้ทำงานตรงตามที่ Test บอกว่ามันควรจะทำไหม?
2. ความอ่านง่าย (Readability): ฉันเข้าใจสิ่งนี้ได้โดยไม่ต้องให้ใครช่วยบอกไหม?
3. สถาปัตยกรรม (Architecture): มันเข้ากันได้กับสภาพระบบเดิมไหม?
4. ความปลอดภัย (Security): มีช่องโหว่อะไรไหม?
5. ประสิทธิภาพ (Performance): มีจุดคอขวดทำระบบช้าไหม?
```

### Step 4: Categorize Findings (จัดหมวดหมู่สิ่งที่ค้นพบ)

ติดป้าย(Label) ระดับความรุนแรงให้ทุกๆ คอมเมนต์ เพื่อให้คนเขียนโค้ดรู้ว่าอันไหนบังคับต้องแก้ อันไหนแค่แวะมาบอก:

| คำนำหน้า | ความหมาย | สิ่งที่คนเขียนต้องทำ |
|--------|---------|---------------|
| *(ไม่มีคำนำหน้า)* | เรื่องที่ต้องแก้ (Required change) | ต้องแก้ก่อนที่จะเอาไป merge |
| **Critical:** | ขวางการ Merge | ช่องโหว่ความปลอดภัย, ข้อมูลสูญหาย, พังเละเทะ |
| **Nit:** | เรื่องจิ๊บจ๊อย, ข้ามได้ | คนเขียนจะเมินก็ได้ — พวกเรื่องเว้นวรรคฟอร์แมต, รสนิยมความชอบส่วนตัว |
| **Optional:** / **Consider:** | ข้อเสนอแนะ | น่าเอาไปพิจารณานะ แต่ไม่ได้บังคับ |
| **FYI** | บอกให้รู้ไว้เฉยๆ (Informational only) | ไม่ต้องทำอะไร — แค่บอกเป็นบริบทเผื่อวันข้างหน้า |

การทำแบบนี้ป้องกันไม่ให้คนเขียนคิดว่าทุกคอมเมนต์คือคำสั่งตายตัว จนไปเสียเวลาแก้เรื่องที่ไม่ได้สำคัญ

### Step 5: Verify the Verification (ตรวจสอบกระบวนการตรวจสอบ)

เช็คเรื่องราวการตรวจสอบ(verification story) ของคนเขียน:

```
- รัน Test อะไรไปบ้าง?
- Build ผ่านไหม?
- มีการเทสด้วยมือคนไหม?
- มีรูปแคปหน้าจอมาให้ดูไหมสำหรับงาน UI?
- มีการเปรียบเทียบ ก่อน/หลัง (before/after) ไหม?
```

## Multi-Model Review Pattern (แพทเทิร์นรีวิวผ่านโมเดลหลายๆ ค่าย)

ใช้โมเดล AI ต่างๆ เพื่อรับมุมมองการรีวิวที่แตกต่างกัน:

```
Model A เป็นคนเขียนโค้ด
    │
    ▼
Model B รีวิวเรื่องความถูกต้องและสถาปัตยกรรม
    │
    ▼
Model A ตามแก้ตามที่คอมเมนต์มา
    │
    ▼
มนุษย์ เป็นคนตัดสินใจขั้นสุดท้าย
```

วิธีนี้ช่วยดักจับข้อผิดพลาดที่โมเดลตัวใดตัวหนึ่งอาจมองข้ามไป — โมเดลต่างค่ายต่างก็มีจุดบอดที่แตกต่างกัน

**ตัวอย่าง prompt สำหรับ Agent รีวิว:**
```
Review this code change for correctness, security, and adherence to
our project conventions. The spec says [X]. The change should [Y].
Flag any issues as Critical, Important, or Suggestion.
```

## Dead Code Hygiene (สุขอนามัยในการกำจัดโค้ดที่ตายแล้ว)

หลังจากการ Refactor หรือเปลี่ยนการทำงานใดๆ ก็ตาม, ให้มองหาโค้ดกำพร้าที่ถูกทิ้งไว้:

1. ระบุหาโค้ดที่ไม่มีทางรันไปถึง หรือไม่มีการเรียกใช้แล้ว
2. ลิสต์รายชื่อมันออกมาให้ชัดเจน
3. **ถามก่อนลบ:** "ฉันควรจะลบของที่ไม่ได้ใช้แล้วพวกนี้ทิ้งเลยไหม: [ลิสต์รายชื่อ]?"

อย่าปล่อยทิ้งซากโค้ดตายเกลื่อนกลาด — มันทำให้คนอ่านในอนาคตและ Agents สับสน แต่อย่าแอบลบเงียบๆ ถ้าคุณไม่มั่นใจว่าลบได้ไหม ถ้าไม่ชัวร์ ให้ถาม

```
พบเจอโค้ดตาย (DEAD CODE IDENTIFIED):
- formatLegacyDate() ใน src/utils/date.ts — ถูกแทนที่โดย formatDate()
- Component OldTaskCard ใน src/components/ — ถูกแทนที่โดย TaskCard
- ตัวแปรคงที่ LEGACY_API_URL ใน src/config.ts — ไม่มีใครเรียกใช้อีกแล้ว
→ ปลอดภัยพอที่จะลบทิ้งไหม?
```

## Review Speed (ความเร็วในการรีวิว)

การดองงานรีวิว จะทำให้ทั้งทีมงานติดขัด ต้นทุนของการสลับหน้าจอ (context-switching) เพื่อมารีวิวงานนั้นน้อยกว่าต้นทุนการปล่อยให้คนอื่นต้องมานั่งรอ

- **ตอบกลับภายในหนึ่งวันทำการ** — นี่คือระยะเวลาสูงสุดที่ยอมรับได้, ไม่ใช่เป้าหมายที่ควรไปถึง
- **จังหวะที่ลงตัว:** ตอบกลับหลังจากได้รับคำขอรีวิวในเวลาไม่นานนัก, เว้นเสียแต่ว่าคุณกำลังจมดิ่งกับการเขียนโค้ดอย่างหนักหน่วง การเปลี่ยนแปลงทั่วไปควรจะผ่านการรีวิวโต้ตอบไปมาหลายๆ รอบจนจบได้ภายในวันเดียว
- **ให้ความสำคัญกับคำตอบที่รวดเร็วของแต่ละคน** มากกว่าการรีบอนุมัติให้จบๆ ไป ข้อเสนอแนะที่ส่งมาไวๆ จะช่วยลดความหงุดหงิดลงได้ แม้ว่าจะต้องแก้กลับไปกลับมาหลายรอบก็ตาม
- **งานขนาดใหญ่:** ให้สั่งคนเขียนนำไปหั่นแบ่งมาใหม่ ดีกว่ามานั่งตาแฉะตรวจโค้ดก้อนมหึมา

## Handling Disagreements (การจัดการข้อขัดแย้ง)

เมื่อต้องตัดสินข้อพิพาทในการรีวิว, ให้ยึดลำดับชั้นนี้:

1. **ข้อเท็จจริงทางเทคนิคและข้อมูล (Technical facts and data)** ต้องอยู่เหนือความเห็นส่วนตัวและความชอบ
2. **คู่มือสไตล์โค้ด (Style guides)** คือประกาศิตสูงสุดในเรื่องของรูปแบบ (style)
3. **การออกแบบซอฟต์แวร์ (Software design)** ต้องประเมินตามหลักวิศวกรรม, ไม่ใช่ประเมินตามความชอบส่วนตัว
4. **ความคงเส้นคงวาของโค้ดเบส (Codebase consistency)** เป็นเรื่องที่อนุโลมยอมรับได้ ตราบใดที่มันไม่ได้ทำให้สุขภาพระบบโดยรวมพัง

**อย่ารับข้ออ้าง "เดี๋ยวค่อยกลับมาตามล้างตามเช็ด".** ประสบการณ์สอนเราว่า การผลัดผ่อน มักจะไม่ได้กลับมาทำจริง บังคับให้เก็บกวาดให้เสร็จก่อนจะบันทึกผล (submission) เว้นแต่จะเป็นเหตุฉุกเฉินไฟไหม้จริงๆ ถ้าระบบรอบข้างไม่สามารถถูกอัปเดตตามไปได้ในครั้งนี้, ให้บังคับเปิดบั๊กสั่งงาน (file a bug) แล้วใส่ชื่อคนพูดเป็นคนรับผิดชอบไปซะ

## Honesty in Review (ความซื่อสัตย์ในการรีวิว)

เมื่อกำลังรีวิวโค้ด — ไม่ว่าใครจะเป็นคนเขียน คุณ, Agent ตัวอื่น, หรือมนุษย์:

- **อย่าทำตัวเป็นตรายาง (rubber-stamp).** การแปะข้อความ "LGTM (ดูดีนะ)" โดยไม่มีหลักฐานการตรวจว่าได้ดูจริงๆ ไม่ได้มีประโยชน์กับใครเลย
- **อย่าทำเรื่องใหญ่ให้เป็นเรื่องเล็ก.** การบอกว่า "นี่อาจจะเป็นปัญหาจิ๊บจ๊อยนะ" ทั้งๆ ที่มันคือบั๊กที่จะไประเบิดบน Production ถือเป็นการไม่ซื่อสัตย์
- **ประเมินปัญหาออกมาเป็นตัวเลขให้ได้.** การบอกว่า "N+1 query ตัวนี้จะทำให้หน่วงขึ้น ~50ms ต่อหนึ่งแถวข้อมูล" ดีกว่าการบอกลอยๆ ว่า "นี่มันอาจจะช้านะ"
- **กล้าแย้งวิธีการที่ดูแล้วเห็นปัญหาชัดเจน.** อาการช่างประจบสอพลอคือจุดบกพร่องของการรีวิว ถ้าวิธีการนี้มันมีปัญหา ให้บอกตรงๆ และเสนอทางเลือกอื่น
- **ยอมรับการโดนปัดตกอย่างสง่างาม.** ถ้าคนเขียนเขามีบริบทครบถ้วนและไม่เห็นด้วยกับคุณ ให้เชื่อใจการตัดสินใจของเขา วิจารณ์ตัวโค้ด ไม่ใช่วิจารณ์ตัวคน — ปรับคำพูดจากการโจมตีตัวบุคคลมาโฟกัสที่ตัวโค้ดแทน

## Dependency Discipline (วินัยในการใช้ส่วนเสริมพึ่งพา)

การรีวิวโค้ดรวมถึงการตรวจส่วนเสริมพึ่งพา (dependency review) ด้วย:

**ก่อนที่จะติดตั้ง dependency ใดๆ:**
1. เครื่องมือที่เรามีอยู่แล้วมันแก้ปัญหานี้ได้ไหม? (ซึ่งมักจะทำได้)
2. Dependency ตัวนี้ใหญ่ขนาดไหน? (เช็คขนาด bundle impact)
3. ยังมีคนดูแลอัปเดตอยู่ไหม? (เช็ค commit ล่าสุด, ปัญหาที่ยังเปิดค้างอยู่)
4. มีช่องโหว่ความปลอดภัยติดมาไหม? (`npm audit`)
5. ใช้สัญญาอนุญาตแบบไหน (License)? (ต้องไม่ขัดกับการทำงานของโปรเจกต์เรา)

**กฎระเบียบ:** เน้นการใช้ library มาตรฐานและระบบเครื่องมือที่ระบบมีให้อยู่แล้ว มากกว่าการไปโหลด dependency ใหม่เข้ามา Dependency ทุกตัวคือภาระผูกพัน

## The Review Checklist (เช็คลิสต์การรีวิว)

```markdown
## Review: [PR/Change title]

### Context (บริบท)
- [ ] ฉันเข้าใจว่าการเปลี่ยนแปลงครั้งนี้ทำไปทำไมและมันคืออะไร

### Correctness (ความถูกต้อง)
- [ ] สิ่งที่แก้ตรงตาม spec/task requirements
- [ ] ดักจับเคสขอบหลืบ (Edge cases) แล้ว
- [ ] ดักจับ Error paths แล้ว
- [ ] Tests ครอบคลุมสิ่งที่แก้ไขเพียงพอแล้ว

### Readability (ความอ่านง่าย)
- [ ] ตั้งชื่อชัดเจนและเป็นไปในทิศทางเดียวกัน
- [ ] ตรรกะความคิดตรงไปตรงมา
- [ ] ไม่มีความซับซ้อนที่เกินจำเป็น

### Architecture (สถาปัตยกรรม)
- [ ] ทำตามแพทเทิร์นเดิมที่มีอยู่
- [ ] ไม่มีการเกาะเกี่ยว (coupling) หรือพึ่งพากันจนเกินพอดี
- [ ] สร้างการครอบ (abstraction level) ในระดับที่พอเหมาะ

### Security (ความปลอดภัย)
- [ ] ไม่ทิ้งความลับใดๆ ไว้ในโค้ด
- [ ] ตรวจสอบความถูกต้องของ Input ทันทีที่เขตแดนแรก
- [ ] ไม่มีช่องโหว่ประเภทการหลอกฝังโค้ด (injection)
- [ ] มีระบบเช็ค Auth ตามจุดต่างๆ
- [ ] มองข้อมูลภายนอกว่าเป็นสิ่งแปลกปลอมและไม่น่าไว้ใจ

### Performance (ประสิทธิภาพ)
- [ ] ไม่มีแพทเทิร์นประเภท N+1
- [ ] ไม่มีการทำงานประเภทวนลูปไม่รู้จบ (unbounded)
- [ ] Endpoints ที่ส่งรายการลิสต์ มีการทำระบบแบ่งหน้า (Pagination)

### Verification (การตรวจสอบ)
- [ ] Tests รันผ่าน
- [ ] Build สำเร็จ
- [ ] ทดสอบด้วยการรันเองแล้ว (ถ้าเป็นไปได้)

### Verdict (คำตัดสิน)
- [ ] **Approve (อนุมัติ)** — พร้อมนำไปรวม (merge)
- [ ] **Request changes (ขอให้แก้)** — ปัญหาต้องถูกแก้ไขก่อน
```

## Common Rationalizations (ข้ออ้างทั่วไป)

| ข้ออ้าง | ความเป็นจริง |
|---|---|
| "โค้ดทำงานได้แล้ว แค่นี้ก็พอแล้วมั้ง" | โค้ดที่ทำงานได้แต่ แต่อ่านไม่รู้เรื่อง, ไม่ปลอดภัย, หรือสถาปัตยกรรมมั่วซั่ว จะก่อให้เกิดหนี้เทคนิคที่จะมาตามหลอกหลอน |
| "ฉันเป็นคนเขียนเอง ฉันรู้ว่ามันถูกน่า" | คนเขียนมักจะตาบอดมองไม่เห็นข้อผิดพลาดของตัวเอง การเปลี่ยนแปลงใดๆ ก็ตามล้วนได้ประโยชน์จากมุมมองของคนอื่นเสมอ |
| "เดี๋ยวค่อยกลับมาคลีนโค้ดละกัน" | คำว่า 'เดี๋ยว' ไม่มีอยู่จริง การรีวิวคือด่านคัดกรองคุณภาพ — ใช้งานมันซะ บังคับให้เขาคลีนโค้ดให้เสร็จก่อนเอามา merge, ไม่ใช่ค่อยมาคลีนเอาทีหลัง |
| "โค้ดจาก AI น่าจะถูกต้องแล้วแหละ" | โค้ด AI ต้องการการตรวจสอบอย่างหนัก ไม่ใช่น้อยลง มันมีความมั่นใจสูงและดูน่าเชื่อถือ แม้ว่ามันจะเขียนผิดก็ตาม |
| "รัน Test ผ่านแล้ว ก็ถือว่าโคเคแล้วแหละ" | Test คือสิ่งจำเป็น แต่ยังไม่พอ มันไม่สามารถตรวจสอบความผิดพลาดด้านสถาปัตยกรรม, ปัญหาความปลอดภัย, หรือความยากในการอ่านโค้ดได้หรอก |

## Red Flags (สัญญาณเตือนภัย)

- PRs ถูกนำไปรวม (merge) โดยที่ไม่มีการรีวิว
- การรีวิวที่ดูแค่ผลการรัน test (เพิกเฉยมิติการตรวจอื่นๆ)
- แปะข้อความ "LGTM" โดยไม่มีร่องรอยการตรวจสอบอย่างจริงจัง
- งานที่มีความสำคัญด้านความปลอดภัย แต่ดันไม่มีการรีวิวที่โฟกัสด้านความปลอดภัย
- PRs ที่ใหญ่จน "กว้างเกินกว่าจะรีวิวไหว" (จับหั่นแบ่งซะ)
- งานแก้บั๊กใน PRs แต่ไม่มีเขียน regression tests กำกับไว้เลย
- คอมเมนต์รีวิวที่ไม่ได้แปะป้ายระดับความรุนแรง (severity) — ทำให้สับสนว่าอันไหนบังคับต้องแก้ อันไหนแค่อยากเสนอแนะ
- ยอมรับข้ออ้างที่บอกว่า "เดี๋ยวมาแก้" — ซึ่งเอาจริงๆ มันไม่เคยเกิดขึ้นเลย

- [ ] รัน Tests ผ่าน
- [ ] Build สำเร็จ
- [ ] มีการบันทึกเรื่องราวการตรวจสอบเอาไว้ (มีอะไรเปลี่ยนไป, และตรวจความถูกต้องอย่างไร)
