# Spec-Driven Development

## Overview

เขียนเอกสารข้อกำหนด (Spec) ให้เป็นระบบก่อนจะลงมือเขียนโค้ด Spec คือจุดเชื่อมต่อความจริงระหว่างคุณและวิศวกรผู้ใช้งาน — มันจะอธิบายว่าเรากำลังจะสร้างอะไร, เพื่ออะไร, และจะรู้ได้อย่างไรว่ามันเสร็จสมบูรณ์แล้ว การเขียนโค้ดโดยไม่มี Spec ก็คือการเดา

## When to Use

- เมื่อเริ่มต้นโปรเจกต์ใหม่ หรือฟีเจอร์ใหม่
- ข้อกำหนดยังคลุมเครือ หรือไม่สมบูรณ์
- เป็นการเปลี่ยนแปลงที่ต้องแก้ไขหลายไฟล์ หรือข้ามโมดูล
- เมื่อคุณกำลังจะตัดสินใจเรื่องสถาปัตยกรรม (Architectural decision)
- เมื่อเป็นงานที่น่าจะใช้เวลาทำมากกว่า 30 นาที

**เมื่อใดที่ไม่ควรใช้:** งานแก้โค้ดแค่บรรทัดเดียว, แก้คำผิด, หรือการเปลี่ยนแปลงที่ข้อกำหนดชัดเจนสุดๆ และจบภายในตัวเอง

## The Gated Workflow

กระบวนการ Spec-driven development มี 4 ระยะ (Phase) ห้ามข้ามไปยังระยะถัดไปจนกว่าระยะปัจจุบันจะได้รับการรับรอง

```
SPECIFY ──→ PLAN ──→ TASKS ──→ IMPLEMENT
   │          │        │          │
   ▼          ▼        ▼          ▼
 มนุษย์        มนุษย์     มนุษย์      มนุษย์
 ตรวจสอบ      ตรวจสอบ   ตรวจสอบ    ตรวจสอบ
```

### Phase 1: Specify (ระบุข้อกำหนด)

เริ่มต้นจากภาพวิสัยทัศน์กว้างๆ ตั้งคำถามเจาะจงกับผู้ใช้จนกว่าข้อกำหนดจะเป็นรูปธรรม

**เปิดเผยข้อสมมติฐานทันที** ก่อนที่จะเริ่มเขียนข้อความใดๆ ใน Spec ให้ลิสต์ออกมาก่อนว่าคุณแอบทึกทักอะไรไปบ้าง:

```
ข้อสมมติฐานที่ฉันตั้งไว้ (ASSUMPTIONS I'M MAKING):
1. นี่คือเว็บแอปพลิเคชัน (ไม่ใช่แอปมือถือ)
2. ระบบ Authentication ใช้ session-based cookies (ไม่ใช่ JWT)
3. ฐานข้อมูลคือ PostgreSQL (อ้างอิงจาก Prisma schema เดิมที่มีอยู่)
4. เรารองรับเฉพาะเบราว์เซอร์ยุคใหม่ (ไม่ทำเผื่อ IE11)
→ โปรดแก้ไขทันที หากไม่ถูกต้อง มิเช่นนั้นฉันจะดำเนินการตามนี้
```

อย่าแอบเติมข้อกำหนดที่กำกวมด้วยตัวเอง จุดประสงค์ทั้งหมดของ Spec คือการเปิดเผยความเข้าใจผิด *ก่อน* ที่จะเริ่มลงมือเขียนโค้ด — การคิดทึกทักเอาเองคือรูปแบบของความเข้าใจผิดที่อันตรายที่สุด

**เขียนเอกสาร Spec โดยครอบคลุมพื้นที่ 6 ส่วนนี้:**

1. **Objective (วัตถุประสงค์)** — เรากำลังจะสร้างอะไร และทำไม? ผู้ใช้คือใคร? หน้าตาความสำเร็จเป็นแบบไหน?

2. **Commands (คำสั่งการทำงาน)** — คำสั่งที่รันได้จริงแบบเต็มยศพร้อมการตั้งค่าธง (flags) ต่างๆ ไม่ใช่ใส่มาแค่ชื่อเครื่องมือ
   ```
   Build: npm run build
   Test: npm test -- --coverage
   Lint: npm run lint --fix
   Dev: npm run dev
   ```

3. **Project Structure (โครงสร้างโปรเจกต์)** — โค้ดต้นฉบับจะเก็บไว้ที่ไหน, Test ไว้ที่ไหน, เอกสารไว้ที่ไหน
   ```
   src/           → ซอร์สโค้ดของแอปพลิเคชัน
   src/components → คอมโพเนนต์ React
   src/lib        → ยูทิลิตี้สำหรับเรียกใช้งานร่วมกัน
   tests/         → การทดสอบ Unit และ integration tests
   e2e/           → การทดสอบระบบหน้าบ้านยันหลังบ้าน (End-to-end tests)
   docs/          → เอกสาร (Documentation)
   ```

4. **Code Style (รูปแบบการเขียนโค้ด)** — การหยิบโค้ดตัวอย่างจริงๆ มาโชว์แค่หนึ่งชุด มีประโยชน์กว่าการอธิบายด้วยตัวอักษร 3 ย่อหน้า ระบุหลักการตั้งชื่อ (Naming conventions), กฎการจัดรูปแบบโค้ด, และผลลัพธ์ตัวอย่างที่ดี

5. **Testing Strategy (กลยุทธ์การทดสอบ)** — จะใช้ Framework อะไร, เก็บไฟล์ Test ไว้ตรงไหน, เปอร์เซ็นต์ความครอบคลุมที่คาดหวัง, ใช้ระดับการทดสอบแบบใดในการตรวจสอบจุดใด

6. **Boundaries (ขอบเขตการทำงาน)** — แบ่งเป็น 3 ระดับ:
   - **สิ่งที่ต้องทำเสมอ (Always do):** รัน Test ก่อน Commit, ทำตามกฎการตั้งชื่อ, ตรวจสอบข้อมูลก่อนรับเข้ามา
   - **ให้ถามก่อนเสมอ (Ask first):** เมื่อจะแก้โครงสร้างฐานข้อมูล, เมื่อจะเพิ่ม Dependencies, เมื่อจะแก้ CI config
   - **ห้ามทำเด็ดขาด (Never do):** Commit พวกรหัสผ่านหรือความลับ, แก้ไขไฟล์ใน vendor directories, ลบไฟล์ทดสอบที่ไม่ผ่านทิ้งโดยไม่ได้ขออนุญาต

**Spec template (โครงร่างของ Spec):**

```markdown
# Spec: [ชื่อโปรเจกต์/ฟีเจอร์]

## Objective
[เราสร้างสิ่งนี้เพื่ออะไร และสร้างทำไม. เล่าเป็น User stories หรือ Acceptance criteria.]

## Tech Stack
[Framework, ภาษา, ข้อมูล dependencies สำคัญพร้อมเลขเวอร์ชัน]

## Commands
[คำสั่งสำหรับ Build, test, lint, dev — แบบเต็มยศ]

## Project Structure
[ผังไดเรกทอรีพร้อมคำอธิบาย]

## Code Style
[โค้ดตัวอย่างแบบย่อ + กฎการเขียนที่สำคัญ]

## Testing Strategy
[Framework, ตำแหน่งของไฟล์ทดสอบ, ข้อกำหนดความครอบคลุม, ระดับของ Test]

## Boundaries
- Always: [...]
- Ask first: [...]
- Never: [...]

## Success Criteria
[เราจะรู้ว่างานนี้เสร็จสมบูรณ์เมื่อไหร่ — เงื่อนไขเจาะจง ที่ต้องทดสอบได้จริง]

## Open Questions
[คำถามปลายเปิดที่ยังคงรอคำตอบจากผู้ใช้]
```

**แปลงความต้องการแบบกว้างๆ ให้เป็นเงื่อนไขความสำเร็จ** เมื่อเจอกับคำขอแบบคลุมเครือ ให้แปลงเป็นเงื่อนไขที่วัดผลได้จริง:

```
ความต้องการ: "ทำให้แดชบอร์ดโหลดเร็วขึ้น"

แปลงเป็นเงื่อนไขความสำเร็จ (REFRAMED SUCCESS CRITERIA):
- ค่า LCP ของแดชบอร์ดต้องน้อยกว่า 2.5 วินาทีเมื่อใช้เน็ตมือถือ 4G
- การดึงข้อมูลรอบแรก (Initial data load) ต้องเสร็จในเวลาไม่เกิน 500ms
- ห้ามมี Layout Shift ระหว่างการโหลด (CLS < 0.1)
→ ข้อมูลเหล่านี้คือเป้าหมายที่ถูกต้องใช่ไหม?
```

วิธีการนี้ทำให้คุณสามารถวนซ้ำ, ลองผิดลองถูก, และแก้ปัญหาเพื่อไปสู่เป้าหมายที่ชัดเจนได้ ดีกว่าการมัวเดาว่า "เร็วขึ้น" หมายความว่าอย่างไร

### Phase 2: Plan (วางแผน)

จาก Spec ที่ได้รับการยืนยันแล้ว ให้สร้างแผนงานขั้นตอนเชิงเทคนิค:

1. ระบุถึงคอมโพเนนต์หลักและความเชื่อมโยง
2. กำหนดลำดับการทำงาน (อะไรต้องสร้างเสร็จก่อน)
3. ลิสต์ความเสี่ยงที่เป็นไปได้พร้อมกลยุทธ์รับมือ
4. ระบุส่วนที่สามารถสร้างไปพร้อมๆ กันได้ และส่วนไหนต้องทำตามลำดับก่อนหลัง
5. กำหนดจุด Checkpoint ของแต่ละเฟสในการทำงาน

แผนงานนี้จะต้องผ่านการตรวจสอบ: มนุษย์จะต้องสามารถอ่านมันแล้วบอกว่า "ใช่ นี่คือแนวทางที่ถูก" หรือ "ไม่ เอาส่วน X ไปแก้ใหม่"

### Phase 3: Tasks (สร้างรายการงาน)

แตกแผนออกเป็นงานย่อยๆ ที่สามารถปฏิบัติได้:

- งานแต่ละชิ้นจะต้องสามารถทำจบได้ในรอบการโฟกัส 1 รอบ (single focused session)
- งานแต่ละชิ้นต้องมีเกณฑ์การประเมิน (Acceptance criteria) ที่ชัดเจน
- งานแต่ละชิ้นต้องมีขั้นตอนตรวจสอบด้วยตัวเอง (ทดสอบรันโค้ด, Build ตัวระบบ, หรือตรวจมือด้วยคน)
- รายการงานต้องเรียงลำดับจากสิ่งที่ต้องพึ่งพากันก่อนหลัง ไม่ใช่เรียงตามความรู้สึกสำคัญ
- ไม่มีงานไหนที่ควรต้องเปิดแก้ไฟล์ทีละมากกว่า ~5 ไฟล์

**Task template:**
```markdown
- [ ] Task: [อธิบายชิ้นงาน]
  - Acceptance: [สิ่งที่เป็นเป้าหมายเมื่อทำเสร็จ]
  - Verify: [วิธีการตรวจสอบผล — ใช้คำสั่ง Test, รัน Build, ทดสอบหน้าจอ]
  - Files: [ไฟล์ไหนที่น่าจะถูกแก้ไขบ้าง]
```

### Phase 4: Implement (ลงมือสร้าง)

ลงมือทำทีละงานตามคู่มือที่ `.claude/skills/incremental-implementation/SKILL.md` และ `.claude/skills/test-driven-development/SKILL.md` ใช้ `.claude/skills/context-engineering/SKILL.md` เพื่อโหลดบริบท Spec ที่เกี่ยวข้อง และไฟล์โค้ดต้นฉบับมาในแต่ละขั้น แทนการให้ Agent ยัดตัวอักษรของ Spec ทั้งหมดลงมาทุกๆ รอบ

## Keeping the Spec Alive (ทำให้ Spec คงอยู่ต่อไป)

เอกสาร Spec เป็นเหมือนสิ่งมีชีวิต ไม่ใช่สิ่งประดิษฐ์ที่ทำครั้งเดียวจบ:

- **อัปเดตเมื่อเปลี่ยนการตัดสินใจ** — ถ้าคุณพบว่าหน้าตาข้อมูลต้องเปลี่ยน ให้ไปแก้ใน Spec ก่อน แล้วค่อยเขียนโค้ดใหม่
- **อัปเดตเมื่อขอบเขตเปลี่ยน** — การลดหรือเพิ่มฟีเจอร์ ควรสะท้อนอยู่ภายใน Spec เสมอ
- **Commit ตัว Spec** — ไฟล์ Spec สมควรเข้าไปเก็บอยู่ในระบบควบคุมเวอร์ชันโค้ด (Git) เช่นเดียวกับโค้ดต้นฉบับ
- **อ้างอิง Spec ในตอนทำ PRs** — ทำลิงก์เชื่อมจากคำอธิบาย PR กลับมายังส่วนที่เกี่ยวข้องใน Spec

## Common Rationalizations (ข้ออ้างทั่วไป)

| ข้ออ้าง | ความเป็นจริง |
|---|---|
| "งานนี้มันง่ายนิดเดียว ฉันไม่ต้องเขียน Spec หรอก" | งานง่ายไม่ได้ต้องการ Spec *ยาวๆ* แต่ยังคงต้องการเกณฑ์การยอมรับ (Acceptance criteria) อยู่ดี Spec แบบแค่สองบรรทัดก็ถือว่าโอเค |
| "เดี๋ยวฉันค่อยไปเขียน Spec ตอนเขียนโค้ดเสร็จ" | นั่นเรียกว่าคู่มือ (Documentation) ไม่ใช่ Specification คุณค่าของ Spec คือการทำทุกอย่างให้แจ่มแจ้ง *ก่อน* จะเริ่มเขียนโค้ด |
| "การเขียน Spec มันทำให้งานเราเดินหน้าช้าลง" | การเสียเวลาเขียน Spec เพียง 15 นาที ช่วยป้องกันการแก้งานนับชั่วโมง ทำงานแบบ Waterfall 15 นาที ย่อมดีกว่าการเสียเวลานั่งดีบักหาบั๊กเป็นเวลา 15 ชั่วโมง |
| "เดี๋ยวความต้องการมันก็เปลี่ยนอยู่ดี" | นั่นแหละคือสาเหตุว่าทำไมเราต้องทำให้ Spec เป็นเหมือนสิ่งมีชีวิตไงล่ะ Spec เก่าที่ไม่อัปเดต ก็ยังดีกว่าการไม่มี Spec เลย |
| "ผู้ใช้รู้ว่าพวกเขาต้องการอะไรอยู่แล้ว" | ต่อให้เป็นคำขอที่ชัดเจนที่สุดก็ยังแอบแฝงไปด้วยสมมติฐาน และ Spec นี่แหละคือตัวเปิดเผยความลับเหล่านั้น |

## Red Flags (สัญญาณเตือนภัย)

- เริ่มลงมือเขียนโค้ดโดยไม่ยอมบันทึกข้อกำหนดใดๆ ออกมาเลย
- แอบถามว่า "ให้เริ่มลงมือสร้างเลยไหม?" ก่อนที่จะถามเพื่อยืนยันว่าคำว่า "เสร็จสิ้น" แปลว่าอะไรกันแน่
- เริ่มสร้างฟีเจอร์ที่ดันไม่มีบันทึกไว้ใน Spec หรือ Task list ใดๆ
- ตัดสินใจปรับสถาปัตยกรรม (Architecture) โดยไม่ระบุไว้ในเอกสาร
- มองข้ามการสร้าง Spec เพราะคิดเข้าข้างตัวเองว่า "เรื่องแค่นี้ใครๆ ก็รู้ว่าต้องทำยังไง"

- [ ] ขอบเขตการทำงาน (Always/Ask First/Never) ถูกกำหนดไว้ครบถ้วน
- [ ] ไฟล์ Spec ได้ถูกบันทึกลงไปในพื้นที่ Repository แล้ว
