# Planning and Task Breakdown

## Overview

แตกงานออกเป็น Task ย่อยๆ ที่เล็กและสามารถตรวจสอบผลลัพธ์ได้ พร้อมกับมีเกณฑ์การยอมรับ (Acceptance criteria) ที่ชัดเจน การแตกงานที่ดีคือความแตกต่างระหว่าง Agent ที่ทำงานเสร็จอย่างน่าเชื่อถือ กับ Agent ที่สร้างผลงานที่ยุ่งเหยิง ทุกๆ Task ควรจะเล็กพอที่จะสร้าง, ทดสอบ, และตรวจสอบผลได้จบในรอบการทำงานเพียงรอบเดียว

## When to Use

- คุณมี Spec แล้วและต้องการแตกมันออกเป็นหน่วยงานที่พร้อมลงมือทำ
- งานรู้สึกใหญ่หรือคลุมเครือเกินกว่าจะเริ่มลงมือ
- จำเป็นต้องแบ่งงานให้ทำขนานกันโดย Agent หลายตัว หรือหลายรอบทำงาน (Sessions)
- คุณต้องการสื่อสารขอบเขตงานให้มนุษย์เข้าใจ
- ลำดับการสร้างงานยังไม่ชัดเจน

**เมื่อใดที่ไม่ควรใช้:** การแก้ไขไฟล์เดียวที่มีขอบเขตชัดเจน หรือเมื่อ Spec ได้มีการแตก Task ไว้ให้เรียบร้อยแล้ว

## The Planning Process

### Step 1: Enter Plan Mode (เข้าสู่โหมดวางแผน)

ก่อนที่จะเขียนโค้ด ให้คุณเข้าสู่โหมดอ่านอย่างเดียว (Read-only mode):

- อ่านเอกสาร Spec และส่วนต่างๆ ของ Codebase ที่เกี่ยวข้อง
- สังเกต Pattern และหลักปฏิบัติ (Conventions) ที่มีอยู่เดิม
- ทำแผนผังความเกี่ยวโยง (Dependencies) ระหว่างคอมโพเนนต์ต่างๆ
- จดบันทึกความเสี่ยงและสิ่งที่ยังไม่รู้

**ห้ามเขียนโค้ดระหว่างการวางแผน** ผลลัพธ์ที่ได้ต้องเป็นเอกสารแผนงาน ไม่ใช่ตัวโค้ดที่รันได้

### Step 2: Identify the Dependency Graph (หาโครงข่ายสิ่งที่ต้องพึ่งพากัน)

วาดแผนผังว่าอะไรต้องพึ่งพาอะไร:

```
สคีมาฐานข้อมูล (Database schema)
    │
    ├── โมเดล/ประเภทข้อมูล API (API models/types)
    │       │
    │       ├── เอนด์พอยต์ API (API endpoints)
    │       │       │
    │       │       └── API Client สำหรับหน้าบ้าน (Frontend API client)
    │       │               │
    │       │               └── คอมโพเนนต์ UI (UI components)
    │       │
    │       └── ตรรกะตรวจสอบข้อมูล (Validation logic)
    │
    └── ข้อมูลตั้งต้น / การตั้งค่า Migration (Seed data / migrations)
```

ลำดับการทำงานควรเริ่มจากล่างขึ้นบนตามกราฟ: สร้างรากฐานก่อนเสมอ

### Step 3: Slice Vertically (หั่นงานตามแนวตั้ง)

แทนที่จะสร้างฐานข้อมูลทั้งหมด แล้วไปสร้าง API ทั้งหมด แล้วค่อยไปสร้าง UI ทั้งหมด — ให้เปลี่ยนเป็นสร้างฟีเจอร์ให้จบเป็นเส้นทางเดียวทีละเรื่อง:

**สิ่งที่แย่ (หั่นตามแนวนอน):**
```
Task 1: สร้าง Database schema ทั้งหมด
Task 2: สร้าง API endpoints ทั้งหมด
Task 3: สร้าง UI components ทั้งหมด
Task 4: จับทุกอย่างมาเชื่อมต่อกัน
```

**สิ่งที่ดี (หั่นตามแนวตั้ง):**
```
Task 1: ผู้ใช้สามารถสร้างบัญชีได้ (schema + API + UI สำหรับสมัครสมาชิก)
Task 2: ผู้ใช้สามารถล็อกอินได้ (auth schema + API + UI สำหรับล็อกอิน)
Task 3: ผู้ใช้สามารถสร้าง Task ได้ (task schema + API + UI สำหรับการสร้าง)
Task 4: ผู้ใช้สามารถดูรายการ Task ได้ (query + API + UI สำหรับหน้าจอแสดงผล)
```

การหั่นแต่ละแนวตั้งจะช่วยส่งมอบฟังก์ชันการทำงานที่ใช้งานได้จริงและสามารถทดสอบได้ทันที

### Step 4: Write Tasks (เขียนรายการงาน)

แต่ละงาน (Task) ควรใช้โครงสร้างแบบนี้:

```markdown
## Task [N]: [ชื่ออธิบายสั้นๆ]

**Description (คำอธิบาย):** อธิบาย 1 ย่อหน้าว่างานชิ้นนี้ทำไปเพื่อเป้าหมายใด

**Acceptance criteria (เกณฑ์ความสำเร็จ):**
- [ ] [เงื่อนไขเฉพาะเจาะจง ที่ใช้ทดสอบได้]
- [ ] [เงื่อนไขเฉพาะเจาะจง ที่ใช้ทดสอบได้]

**Verification (วิธีการตรวจสอบ):**
- [ ] ผ่านการทดสอบ (Tests pass): `npm test -- --grep "feature-name"`
- [ ] รัน Build ผ่าน: `npm run build`
- [ ] เช็คด้วยมือ: [อธิบายว่าจะต้องตรวจดูอะไรบ้าง]

**Dependencies (ส่วนที่ต้องรอ):** [หมายเลข Task ที่ต้องทำก่อน หรือใส่ "None" ถ้าไม่มี]

**Files likely touched (ไฟล์ที่น่าจะถูกแก้ไข):**
- `src/path/to/file.ts`
- `tests/path/to/test.ts`

**Estimated scope (การประเมินขนาดงาน):** [เล็ก (Small): 1-2 ไฟล์ | กลาง (Medium): 3-5 ไฟล์ | ใหญ่ (Large): 5+ ไฟล์]
```

### Step 5: Order and Checkpoint (เรียงลำดับ และกำหนดจุดตรวจ)

จัดเรียง Task ต่างๆ เพื่อให้:

1. ลำดับงานที่พึ่งพากันได้ทำก่อน (สร้างรากฐานก่อน)
2. ทุกๆ Task ทำเสร็จแล้วระบบจะต้องยังคงทำงานได้ (ไม่พัง)
3. มีจุดตรวจ (Verification checkpoints) แทรกทุกๆ 2-3 Tasks
4. งานที่มีความเสี่ยงสูงต้องถูกทำก่อน (เผื่อล้ม จะได้ล้มเร็วๆ)

แทรกจุดตรวจที่ชัดเจน:

```markdown
## Checkpoint: หลังจากจบ Tasks 1-3
- [ ] Tests ทุกตัวต้องผ่าน
- [ ] แพลปพลิเคชันต้องรัน Build ได้โดยไม่มี error
- [ ] กระบวนการผู้ใช้หลัก (Core user flow) ต้องทำงานตั้งแต่ต้นจนจบได้
- [ ] ให้มนุษย์รีวิวก่อนดำเนินงานขั้นต่อไป
```

## Task Sizing Guidelines (แนวทางการประเมินขนาด Task)

| ขนาด | จำนวนไฟล์ | ขอบเขต | ตัวอย่าง |
|------|-------|-------|---------|
| **XS** | 1 | แค่ฟังก์ชันเดียวหรือแก้ config | การเพิ่มเงื่อนไข validate ข้อมูล |
| **S** | 1-2 | คอมโพเนนต์ หรือ API endpoint เดียว | การเพิ่ม API endpoint ใหม่ |
| **M** | 3-5 | ฟีเจอร์ 1 หั่นแนวตั้ง | กระบวนการสมัครสมาชิกผู้ใช้ |
| **L** | 5-8 | ฟีเจอร์ที่ต้องแก้หลายส่วน | ระบบค้นหาที่สามารถกรองและแบ่งหน้าได้ |
| **XL** | 8+ | **ใหญ่เกินไป — ย่อยมันออกมาอีก** | — |

ถ้างานมีขนาด L หรือใหญ่กว่านั้น ควรถูกหั่นออกมาเป็นงานย่อยๆ Agent จะทำงานได้ประสิทธิภาพดีที่สุดเมื่อได้งานขนาด S และ M

**เมื่อไหร่ที่ควรหั่นงานให้เล็กลงอีก:**
- เมื่องานนั้นจะใช้เวลามากกว่า 1 รอบการโฟกัส (ประเมินว่า Agent น่าจะใช้เวลาเกิน 2 ชั่วโมง)
- เมื่อคุณไม่สามารถเขียน Acceptance criteria ให้จบได้ภายใน 3 บรรทัด (bullet points)
- เมื่อมันต้องไปยุ่งกับระบบ 2 ระบบที่เป็นอิสระต่อกัน (เช่น ระบบ auth และ ระบบ billing)
- เมื่อคุณเผลอเขียนคำว่า "และ" ในชื่อ Task (เป็นสัญญาณว่ามันคือ 2 งานรวมกัน)

## Parallelization Opportunities (โอกาสในการทำงานแบบขนาน)

เมื่อคุณสามารถใช้ Agent ได้หลายตัว หรือสามารถเปิด Session ได้หลายหน้าต่าง:

- **ปลอดภัยที่จะให้ทำขนานกัน:** การทำฟีเจอร์สไลซ์ที่เป็นอิสระต่อกัน, การเขียน Tests ให้ฟีเจอร์ที่เสร็จแล้ว, การเขียนเอกสารคู่มือ (Documentation)
- **บังคับว่าต้องทำตามลำดับ:** การ Migration ฐานข้อมูล, การเปลี่ยน State ส่วนกลาง, งานที่เป็นห่วงโซ่ต่อกัน
- **ต้องการการประสานงานสูง:** ฟีเจอร์ที่ต้องใช้ API Contract เดียวกัน (ให้กำหนด Contract ให้จบก่อน แล้วค่อยแยกย้ายกันไปทำขนาน)

## Common Rationalizations (ข้ออ้างทั่วไป)

| ข้ออ้าง | ความเป็นจริง |
|---|---|
| "เดี๋ยวฉันค่อยไปหาทางเอาดาบหน้า" | นั่นแหละคือเส้นทางสู่ความเละเทะและต้องมาแก้งานใหม่ การเสียเวลาวางแผนแค่ 10 นาที ช่วยประหยัดเวลาเป็นชั่วโมง |
| "งานมันก็เห็นๆ กันอยู่ว่าต้องทำอะไร" | เขียนมันออกมาอยู่ดี. การเขียนงานแบบละเอียดจะช่วยเปิดเผยว่ามีอะไรที่เราต้องพึ่งพาและดักจับ edge cases ที่ลืมไป |
| "การวางแผนคือภาระงาน" | การวางแผนนั่นแหละคืองาน การลงมือสร้างโดยไม่มีแผนคือการสักแต่พิมพ์โค้ด |
| "ฉันจำทั้งหมดไว้ในหัวได้" | Context windows ของคุณมีจำกัด แผนที่ถูกเขียนไว้จะยังคงอยู่แม้การสนทนาข้าม Session จะถูกสรุปรวบไปแล้วก็ตาม |

## Red Flags (สัญญาณเตือนภัย)

- เริ่มต้นลงมือสร้าง (Implementation) โดยไม่มีการเขียน Task list
- Task มีเขียนแค่ว่า "สร้างฟีเจอร์" โดยไม่มี Acceptance criteria
- ไม่มีขั้นตอนตรวจสอบใดๆ อยู่ในแผน
- Task ทั้งหมดกลายเป็นขนาด XL
- ไม่มีจุดตรวจ (Checkpoints) ระหว่าง Task
- ไม่ได้คำนึงถึงลำดับก่อนหลังของสิ่งที่พึ่งพากัน

## Verification

ก่อนจะข้ามไปขั้นตอนสร้างงาน (Implementation) โปรดยืนยันสิ่งเหล่านี้ให้ครบ:

- [ ] ทุกๆ Task ต้องมี Acceptance criteria
- [ ] ทุกๆ Task ต้องมีวิธีการตรวจสอบ
- [ ] มีการระบุลำดับก่อนหลังของงานและเรียงอย่างถูกต้อง
- [ ] ไม่มีงานไหนที่ไปกระทบไฟล์เกิน ~5 ไฟล์
- [ ] มี Checkpoints แทรกอยู่ระหว่างแต่ละระยะเวลาทำงาน (phases)
- [ ] มนุษย์ตรวจสอบและอนุมัติแผนงานเรียบร้อยแล้ว
