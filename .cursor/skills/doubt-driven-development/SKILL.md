# Doubt-Driven Development

## Overview

คำตอบที่คุณมั่นใจ ไม่ได้แปลว่ามันจะเป็นคำตอบที่ถูกต้องเสมอไป การทำงานในเซสชันที่ยาวนานจะสะสมบริบทจนทำให้คุณแอบทึกทักเอาข้อสันนิษฐานต่างๆ มาเป็น "ความจริง" โดยไม่รู้ตัว Doubt-driven development (การพัฒนาแบบตั้งข้อสงสัย) คือวินัยในการตั้งตนเป็นผู้ตรวจสอบที่ใช้บริบทแบบสดใหม่ — โดยมุ่งเน้นที่จะ **หาข้อผิดพลาด** ไม่ใช่หาข้อถูก — ก่อนที่จะปล่อยผ่านการตัดสินใจสำคัญๆ

นี่ไม่ใช่การใช้คำสั่ง `/review` เพราะ `/review` คือการตัดสินผลลัพธ์ที่เสร็จสมบูรณ์แล้ว แต่ทักษะนี้คือท่าทีระหว่างการทำงาน: การตัดสินใจที่สำคัญๆ จะต้องถูกตั้งคำถามทบทวน ในช่วงเวลาที่การแก้ไขยังทำได้ง่ายและราคาถูก

## When to Use

การตัดสินใจจะถือว่า **สำคัญ (non-trivial)** เมื่อเข้าข่ายอย่างน้อยหนึ่งข้อต่อไปนี้:

- มีการเพิ่มหรือแก้ไขตรรกะแบบมีเงื่อนไข (branching logic)
- มีการทำงานข้ามโมดูลหรือข้ามเซอร์วิส
- มีการยืนยันคุณสมบัติบางอย่างที่ตัวตรวจสอบชนิดข้อมูล (type system) หรือคอมไพเลอร์ ไม่สามารถเช็คให้ได้ (เช่น thread safety, idempotence, ordering, invariants)
- ความถูกต้องของมันขึ้นอยู่กับบริบทที่คนอ่านโค้ดในอนาคตจะมองไม่เห็น
- รัศมีทำลายล้างของมันไม่สามารถกู้คืนได้ (เช่น deploy ขึ้น Production, การย้ายข้อมูลฐานข้อมูล, การเปลี่ยน Public API)

จงใช้ทักษะนี้เมื่อ:

- กำลังจะตัดสินใจเรื่องสถาปัตยกรรมท่ามกลางความไม่แน่นอน
- กำลังจะ Commit โค้ดที่ซับซ้อน
- กำลังจะเคลมความจริงบางอย่างที่ไม่ชัดเจน ("วิธีนี้ปลอดภัย", "วิธีนี้ขยายระบบได้", "วิธีนี้ตรงตาม Spec")
- กำลังทำงานในโค้ดที่คุณไม่ได้เข้าใจมันอย่างถ่องแท้

**เมื่อใดที่ไม่ควรใช้:**

- งานปรับแก้ทั่วไป (เปลี่ยนชื่อ, จัดฟอร์แมต, ย้ายไฟล์)
- ทำตามคำสั่งของผู้ใช้ที่มีความชัดเจนและไม่มีความคลุมเครือ
- การเปิดอ่านหรือการสรุปโค้ดที่มีอยู่
- การแก้ไขแค่บรรทัดเดียวที่ดูออกทันทีว่าถูกต้องแน่ๆ
- การทำงานกับเครื่องมือเพียวๆ (การรัน test, การลิสต์รายชื่อไฟล์)
- เมื่อผู้ใช้สั่งชัดเจนว่าต้องการความรวดเร็วมากกว่าการตรวจสอบ ("รีบๆ ทำเลย")

ถ้าคุณมัวแต่สงสัยทุกลมหายใจ คุณก็จะไม่ได้สร้างผลงานอะไรเลย ทักษะนี้ใช้เฉพาะกับ "การตัดสินใจสำคัญ" ตามที่ระบุไว้เท่านั้น

## Loading Constraints (ข้อจำกัดในการเรียกใช้)

ทักษะนี้ถูกออกแบบมาเพื่อ **ผู้สั่งการในเซสชันหลัก (main-session orchestrator)** ซึ่งใน Step 3 (DOUBT ที่ระบุไว้ด้านล่าง) สามารถเสกผู้ตรวจสอบแบบสดใหม่ (fresh-context reviewer) ขึ้นมาได้

- **ห้ามเพิ่มทักษะนี้ลงในข้อมูล `skills:` frontmatter ของ Persona ย่อยเด็ดขาด** Persona ที่ทำตาม Step 3 จะต้องทำการสร้าง Persona อีกตัวขึ้นมา — ซึ่งการสั่งการซ้อนกันแบบนี้ถือเป็นข้อห้ามที่ระบุไว้ใน `references/orchestration-patterns.md` ("personas do not invoke other personas")
- **หากคุณพบว่ากำลังเรียกใช้ทักษะนี้จากบริบทที่เป็น Subagent** (ซึ่ง Claude Code ไม่อนุญาตให้สร้าง Subagent ซ้อนกันได้): ทางออกที่ดีที่สุดคือแจ้งผู้ใช้ว่า Doubt-driven ไม่สามารถรันซ้อนกันได้ และให้เซสชันหลักเป็นคนจัดการ แต่ถ้าจำเป็นจริงๆ (Last resort) ให้ใช้วิธีลดระดับเป็นการตั้งคำถามกับตัวเอง — โดยเขียน ARTIFACT + CONTRACT ขึ้นมาใหม่ เสมือนเป็นการล้างสมองแยกขาดจากบริบทเดิมที่เพิ่งทำมา แล้วเดินตาม Step 1-5 แต่นี่ **ไม่ใช่การรีวิวด้วยบริบทใหม่ (fresh-context review)** (เพราะคุณยังพกความทรงจำเดิมมาด้วย) ดังนั้นให้แปะป้ายว่าผลลัพธ์นี้ถูกลดคุณภาพ และให้พยายามผลักภาระไปให้ผู้ใช้ตัดสินใจแทนเมื่อสามารถทำได้

## The Process (กระบวนการ)

ให้คัดลอกเช็คลิสต์นี้เมื่อจะใช้ทักษะนี้:

```
วงจรความสงสัย (Doubt cycle):
- [ ] Step 1: CLAIM (เคลม) — เขียนคำเคลม + บอกว่าทำไมมันถึงสำคัญ
- [ ] Step 2: EXTRACT (สกัด) — แยกเฉพาะโค้ด (Artifact) + กฎข้อตกลง (Contract) ตัดพวกการอธิบายเหตุผลทิ้ง
- [ ] Step 3: DOUBT (สงสัย) — เรียกใช้ตัวรีวิวแบบ fresh-context ด้วย Prompt แบบจับผิด
- [ ] Step 4: RECONCILE (ไกล่เกลี่ย) — จัดหมวดหมู่ทุกข้อผิดพลาดที่เจอ โดยเทียบกับตัวโค้ด (Artifact)
- [ ] Step 5: STOP (หยุด) — บรรลุเงื่อนไขการหยุด (เจอแต่เรื่องจิ๊บจ๊อย, ครบ 3 รอบ, หรือผู้ใช้สั่งข้าม)
```

### Step 1: CLAIM — Surface what stands (เคลม — เปิดเผยสิ่งที่จะตัดสินใจ)

ตั้งชื่อการตัดสินใจนั้นให้จบใน 2-3 บรรทัด:

```
สิ่งที่ฉันเคลม (CLAIM): "ระบบ caching layer ตัวใหม่ มีการป้องกัน thread-safe 
        ภายใต้สถานการณ์ที่มีการอ่านข้อมูลหนักๆ ตามที่ระบุใน Spec"
ทำไมมันถึงสำคัญ (WHY THIS MATTERS): การเกิด race condition ตรงนี้จะทำให้ข้อมูลผู้ใช้พัง
                  และเป็นเรื่องที่ QA ตรวจจับได้ยากมาก
```

ถ้าคุณไม่สามารถเขียนคำเคลมให้กระชับแบบนี้ได้ แปลว่าคุณแค่รู้สึกไปเอง (vibe) ยังไม่ได้ตัดสินใจจริงๆ ให้ไปเปิดเผยความรู้สึกนั้นออกมาก่อนที่จะมานั่งวิเคราะห์จับผิดมัน

### Step 2: EXTRACT — Smallest reviewable unit (สกัด — หั่นเป็นชิ้นเล็กที่สุดที่พอจะรีวิวได้)

ผู้ตรวจสอบที่เข้ามาแบบสมองโล่งๆ (fresh-context reviewer) ต้องการแค่ **ชิ้นงาน (artifact)** และ **ข้อตกลง (contract)** เขาไม่ต้องการรู้เรื่องราวการเดินทางของคุณ

- โค้ด: เอาแค่ตัว diff หรือฟังก์ชันนั้นๆ — ไม่ใช่ส่งไปทั้งไฟล์
- ข้อตัดสินใจ (Decision): ข้อเสนอความยาว 3-5 ประโยค พร้อมเงื่อนไขข้อจำกัดที่มันต้องทำได้
- ข้อพิสูจน์ (Assertion): คำเคลมบวกด้วยหลักฐานที่เชื่อว่าสนับสนุนเคลมนั้น (แยกให้ต่างจากข้อ CLAIM ใน Step 1 ซึ่งเป็นเพียงแค่สมมติฐานของผู้สั่งการที่กำลังจะถูกจับผิด)

จงลบพวกเหตุผลและการให้เหตุผลของคุณทิ้งให้หมด ถ้าคุณส่งข้อสรุปของคุณไปให้คนรีวิว คุณก็จะได้แค่คำอวยพ้องตามข้อสรุปของคุณกลับมา ชิ้นงานต้องเล็กพอที่คนรีวิวจะอ่านแล้วจำไว้ในหัวได้ทั้งหมดในการอ่านรอบเดียว — ถ้ามันเป็นโค้ดใหญ่ขนาด 500 บรรทัด ให้ไปย่อยมันมาก่อน

### Step 3: DOUBT — Invoke the fresh-context reviewer (สงสัย — เรียกใช้ผู้ตรวจสอบแบบสมองโล่ง)

Prompt ที่ส่งให้ผู้ตรวจสอบ **ต้องเป็นแบบจับผิด (adversarial)** เพราะการตั้งกรอบคำถาม จะเป็นตัวกำหนดคำตอบ

```
การรีวิวแบบจับผิด (Adversarial review) จงหาว่ามีอะไรผิดปกติในชิ้นงานนี้
สมมติว่าผู้เขียนมันมีความมั่นใจเกินเหตุ ให้มองหา:
- ข้อสันนิษฐานที่ไม่ได้ถูกเขียนบอกไว้ (Unstated assumptions)
- เคสแปลกๆ ที่ไม่ได้ดักจับไว้ (Edge cases not handled)
- การเชื่อมโยงที่แอบแฝง หรือการแชร์สถานะทับกัน (Hidden coupling/shared state)
- ช่องโหว่ที่จะทำให้แหกกฎของข้อตกลง (contract)
- กฎธรรมเนียมเดิมของโปรเจกต์ที่อาจจะถูกแหก
- โอกาสพังเมื่อเจอข้อมูลที่ไม่ได้คาดหวัง

ห้ามอวย ห้ามสรุป. จงหาข้อผิดพลาด, หรือประกาศให้ชัดเจนว่า
คุณหาข้อผิดพลาดไม่เจอเลยหลังจากที่ตรวจสอบอย่างละเอียดแล้ว

ชิ้นงาน (ARTIFACT): <วางชิ้นงาน>
ข้อตกลง (CONTRACT): <วางข้อตกลง>
```

**ส่งไปแค่ ARTIFACT + CONTRACT เท่านั้น ห้ามส่งข้อ CLAIM ไปด้วยเด็ดขาด** การส่งข้อสรุปของคุณไปให้ผู้ตรวจสอบ จะเป็นการชี้นำให้เขาเห็นด้วย ผู้ตรวจสอบต้องตัดสินด้วยตัวเองว่าตัวชิ้นงานมันบรรลุข้อตกลง (contract) จริงหรือไม่

ใน Claude Code, ผู้ตรวจสอบตามบทบาท (role-based reviewers) ในโฟลเดอร์ `agents/` ถูกออกแบบมาให้รับบริบทแบบแยกขาด (isolated context) อยู่แล้ว และสามารถนำมาใช้ที่นี่ได้ — ดูไฟล์ใน `agents/` สำหรับรายชื่อและความถนัดแต่ละด้าน

**Prompt แบบจับผิดด้านบนมีความสำคัญเหนือกว่ารูปแบบการตอบปกติของ Persona** Persona อย่าง `code-reviewer` ถูกเขียนมาให้ตอบแบบเป็นกลางคือมีทั้งข้อดีและข้อเสีย; แต่ doubt-driven ต้องการผลลัพธ์ที่มีแต่การชี้ข้อผิดพลาดเท่านั้น ให้วาง Prompt แบบจับผิดนี้ลงไปตรงๆ เพื่อบังคับทับ (override) ค่าเริ่มต้นของ Persona ถ้ารูปแบบการตอบของ Persona ไม่ยอมให้ทับค่าได้ง่ายๆ ให้ลดระดับไปใช้ Subagent ทั่วไปพร้อมกับ Prompt แบบจับผิดแทน

#### Cross-model escalation (การยกระดับไปใช้ AI โมเดลค่ายอื่น)

ผู้ตรวจสอบที่ใช้โมเดลเดียวกัน มักจะมีจุดบอดแบบเดียวกันกับผู้เขียน — โมเดลจากค่ายอื่นที่มีสถาปัตยกรรมต่างกันจะช่วยดักจับสิ่งเหล่านั้นได้ การทำ Doubt-driven ถือเป็นการตั้งใจทำกับ "การตัดสินใจสำคัญ" อยู่แล้ว ดังนั้นการเสนอให้ใช้โมเดลข้ามค่าย จึงถือเป็นคุณค่าของทักษะนี้ ไม่ใช่เรื่องยุ่งยากที่ข้ามได้

**ในเซสชันโต้ตอบ (Interactive sessions): ต้องเสนอทางเลือกนี้เสมอ ห้ามแอบข้ามเงียบๆ**

**Step 1: ถามผู้ใช้**

หลังจากจบการรีวิวด้วยโมเดลเดี่ยวใน Step 3 ด้านบน, แต่ต้องทำก่อนจะถึงขั้น RECONCILE, ให้หยุดและถามว่า:

> *"การรีวิวด้วยโมเดลเดี่ยวเสร็จแล้ว ต้องการความเห็นที่สองจากโมเดลค่ายอื่นไหม? ทางเลือก: ใช้ Gemini CLI, Codex CLI, ให้ผู้ใช้เอาไปวางให้บอทอื่นตรวจเอง (manual), หรือจะข้ามไปเลย"*

คำถามนี้ถือเป็นภาคบังคับในทุกๆ รอบของ doubt cycle ที่มีการโต้ตอบ — แม้แต่กับชิ้นงานที่ดูเหมือนความเสี่ยงต่ำ ผู้ใช้ต่างหาก — ไม่ใช่ Agent — ที่จะเป็นคนตัดสินใจว่าคุ้มค่าที่จะทำไหม หน้าที่ของ Agent คือการเสนอทางเลือกให้

**Step 2: ถ้าผู้ใช้เลือกใช้ CLI — ตรวจสอบ, แล้วค่อยเรียกใช้งาน**

1. เช็คว่าเครื่องมืออยู่ใน PATH ไหม (`which gemini`, `which codex`)
2. ทดสอบว่ามันทำงานได้จริง (`gemini --version` หรืออะไรทำนองนั้น) ก่อนที่จะส่ง Prompt เต็มๆ ไป — เพราะบางทีไฟล์เก่าหรือไฟล์เสีย อาจจะรัน `which` ผ่านแต่พังตอนทำงานจริง
3. ยืนยันรูปแบบคำสั่งกับผู้ใช้ รวมถึงพวก flags ที่จำเป็น, การยืนยันตัวตน, และ env vars (เช่น API keys) การทำงานแต่ละเครื่องไม่เหมือนกัน ห้ามทึกทักเอาเอง
4. ส่งไปเฉพาะ ARTIFACT + CONTRACT + adversarial prompt **เท่านั้น** ห้ามส่งประวัติเซสชัน ห้ามส่ง CLAIM
5. ระวังเรื่องอักขระพิเศษใน shell (escaping) ถ้าชิ้นงานมีเครื่องหมายคำพูด, `$(...)`, หรือ backticks ควรใช้วิธีส่งผ่าน stdin (`echo … | gemini`) หรือใช้ heredoc แทนการพิมพ์สดต่อท้าย `-p "…"` หากไม่แน่ใจ ให้ถามผู้ใช้เพื่อยืนยันคำสั่งก่อนรัน
6. นำผลลัพธ์ที่ได้เข้าสู่ Step 4 (RECONCILE)

**ห้ามแทรกชิ้นงานเข้าไปเป็นส่วนหนึ่งของ argument (คำสั่งต่อท้าย) ใน shell เด็ดขาด** โค้ด, markdown, และ prompt มักจะมีพวก backticks, `$(...)`, และเครื่องหมายคำพูด ซึ่งจะทำให้คำสั่งขาดตอน หรือกลายเป็นการแอบรัน shell command โดยไม่ตั้งใจ ให้เขียน Prompt เต็มๆ ลงไฟล์แล้วส่งผ่าน stdin เอา

ตัวอย่างรูปแบบคำสั่ง (เช็ค flags ให้ตรงกับเครื่องมือที่คุณลงไว้ — syntax แตกต่างกันไปตามแต่ละค่ายและเวอร์ชัน):

```bash
# เขียน adversarial prompt + ARTIFACT + CONTRACT ลงไฟล์ temp ก่อน
# จากนั้นพ่นผ่าน stdin เพื่อป้องกันไม่ให้อักขระพิเศษใน shell ทำงาน

# Codex (โหมด read-only sandbox ช่วยป้องกันไม่ให้ CLI แอบเขียนไฟล์ในโปรเจกต์ของคุณ):
codex exec --sandbox read-only -C <repo-path> - < /tmp/doubt-prompt.md

# Gemini (โหมด '--approval-mode plan' คือ read-only; ส่วน '-p ""' คือการเรียกใช้แบบไม่โต้ตอบ
# และจะอ่าน prompt จาก stdin แทน):
gemini --approval-mode plan -p "" < /tmp/doubt-prompt.md
```

โหมด Read-only sandbox เป็นเรื่องคอขาดบาดตาย: ชิ้นงานที่ส่งไปรีวิวอาจจะมีคำสั่งแฝงอยู่ (ไม่ว่าจะตั้งใจโจมตี prompt หรือแค่บังเอิญ) ซึ่ง cross-model CLI อาจจะเผลอรันคำสั่งนั้นใส่ระบบของคุณได้

**Step 3: ถ้า CLI ใช้งานไม่ได้ หรือพัง**

รายงานปัญหาให้ชัดเจน เสนอทางเลือกให้ผู้ใช้: ให้ผู้ใช้รันเอง, ลองใช้เครื่องมือตัวอื่น, หรือจะกดข้ามไปเลย ห้ามแอบสลับกลับไปใช้โมเดลเดี่ยวแบบเงียบๆ — ผู้ใช้ต้องรับรู้ว่ากระบวนการ cross-model ทำไม่สำเร็จ

**Step 4: ถ้าผู้ใช้กดข้าม (Skip)**

รับทราบการข้ามในข้อความโต้ตอบ (*"กำลังดำเนินการต่อโดยใช้ข้อค้นพบจากโมเดลเดี่ยวเท่านั้น"*) และไปต่อยังขั้นตอน RECONCILE การข้ามเป็นเรื่องปกติ; แต่การแอบข้ามเงียบๆ ไม่ปกติ

**ในบริบทที่ไม่มีการโต้ตอบ (Non-interactive contexts)** (เช่น CI, `/loop`, autonomous-loop, หรือสคริปต์ตั้งเวลา):

- การทำ Cross-model จะต้อง **ถูกข้าม**, และต้อง **ประกาศ** บอกในข้อความ Output ว่า: *"ข้ามการทำ Cross-model: เนื่องจากอยู่ในบริบทที่ไม่มีการโต้ตอบ (non-interactive context)."*
- **ห้ามเรียกใช้ external CLI เด็ดขาดหากไม่ได้รับอนุญาตอย่างชัดเจนจากผู้ใช้** — นี่คือกฎความปลอดภัยสูงสุด

การทำ Cross-model เพิ่มค่าใช้จ่าย, เพิ่มความช้า, และเครื่องมืออาจจะเปราะบาง Agent มีหน้าที่เสนอทางเลือกนี้ในทุกๆ รอบ; ผู้ใช้จะเป็นคนเลือกว่าชิ้นงานนั้นๆ คุ้มที่จะทำไหม

### Step 4: RECONCILE — Fold findings back (ไกล่เกลี่ย — จัดการข้อค้นพบ)

ผลลัพธ์จากผู้ตรวจสอบคือ 'ข้อมูล (data)' ไม่ใช่ 'คำพิพากษา (verdict)' **คุณยังคงเป็นผู้สั่งการอยู่ (orchestrator)** ให้กลับไปอ่านข้อความในชิ้นงานเทียบกับแต่ละข้อผิดพลาดที่เจอ ก่อนที่จะจัดหมวดหมู่ — การทำตัวเป็นตรายางแสตมป์เห็นดีเห็นงามไปกับผู้ตรวจสอบ ถือเป็นความล้มเหลวพอกันกับการเมินเฉย

สำหรับข้อค้นพบแต่ละข้อ ให้จัดหมวดหมู่ตาม **ลำดับความสำคัญ** นี้ (อันไหนเข้าข่ายก่อน ชนะ):

1. **อ่านข้อตกลงผิด (Contract misread)** — ผู้ตรวจสอบจับผิดบางอย่าง เพียงเพราะข้อตกลง (CONTRACT) ที่คุณส่งให้มันเขียนไม่เคลียร์ หรือไม่ครบถ้วน ให้ไปแก้ข้อตกลงก่อน แล้วค่อยจัดหมวดหมู่อีกทีในรอบถัดไป
2. **เป็นเรื่องจริง + ต้องแก้ (Valid + actionable)** — เป็นปัญหาของจริงที่ต้องแก้ที่ตัวชิ้นงาน (artifact) ลงมือแก้ซะ แล้ววนลูปใหม่
3. **เป็นเรื่องจริง แต่มันคือการแลกเปลี่ยน (Valid trade-off)** — ปัญหามีอยู่จริง แต่ค่าใช้จ่ายในการแก้ดันแพงกว่าการปล่อยผ่าน ให้จดบันทึกการแลกเปลี่ยน (Trade-off) อย่างชัดเจน เพื่อให้ผู้ใช้ได้รับรู้
4. **ข้อความขยะ (Noise)** — ผู้ตรวจสอบโวยวายจับผิดในสิ่งที่จริงๆ แล้วมันถูกต้องแล้ว เพียงแต่เขาไม่มีบริบทที่มากพอ จดโน้ตไว้ ข้ามมันไป แล้วถามตัวเองว่า: ถ้าเพิ่มบริบทเรื่องนี้เข้าไปในข้อตกลง จะช่วยกันไม่ให้เกิดการจับผิดมั่วซั่วแบบนี้ได้ไหม?

ผู้ตรวจสอบแบบสมองโล่ง ก็สามารถเดาผิดได้เพราะขาดบริบท อย่าไปยอมน้อมรับแค่เพียงเพราะเขามาพร้อม "บริบทใหม่"

### Step 5: STOP — Bounded loop, not recursion (หยุด — วนลูปอย่างมีลิมิต ไม่ใช่วนไปเรื่อยๆ)

หยุดเมื่อ:

- ลูปรอบถัดไปเจอแต่เรื่องจิ๊บจ๊อย หรือเป็นเรื่องเดิมๆ ที่พิจารณาไปแล้ว, **หรือ**
- ทำครบ 3 รอบ (โยนเรื่องให้ผู้ใช้ตัดสินใจ อย่าดันทุรังทำรอบที่ 4 คนเดียว), **หรือ**
- ผู้ใช้ออกคำสั่งชัดเจนว่า "พอ ปล่อยผ่าน (ship it)"

ถ้าทำไปแล้ว 3 รอบ ผู้ตรวจสอบยังขุดเจอเรื่องใหญ่ๆ มาได้ ชิ้นงานนั้นอาจจะยังไม่พร้อมจริงๆ ให้รายงานเรื่องนี้แก่ผู้ใช้ — การแก้ไม่จบใน 3 รอบคือข้อมูลบ่งชี้สถานะของชิ้นงาน ไม่ใช่ข้ออ้างที่จะขอวนลูปต่อไป

ถ้า 3 รอบรู้สึกว่า "มันน้อยไปเห็นๆ" เพราะชิ้นงานมันใหญ่: แปลว่าชิ้นงานมันใหญ่เกินไป — ให้กลับไป Step 2 แล้วหั่นชิ้นงานให้เล็กลง ห้ามขยายลิมิตจำนวนรอบเด็ดขาด

## Common Rationalizations (ข้ออ้างทั่วไป)

| ข้ออ้าง | ความเป็นจริง |
|---|---|
| "ฉันมั่นใจ ข้ามการทำ doubt เถอะ" | ความมั่นใจไม่ได้การันตีความถูกต้องในปัญหาที่ไม่คุ้นเคย ห้วงเวลาแห่งความมั่นใจนี่แหละคือจุดซ่อนเร้นของจุดบอด |
| "การเสกคนมารีวิวมันสิ้นเปลือง" | การนั่งงมแก้บั๊กบน Production สิ้นเปลืองกว่า การรีวิวมีขีดจำกัด แต่การตามแก้บั๊กนั้นไม่มีสิ้นสุด |
| "ไอ้คนรีวิวมันก็แค่หาเรื่องจับผิดไปเรื่อย" | ใช่ ถ้าคุณไม่ตีกรอบให้มัน จำกัด Prompt ให้หาเฉพาะ "ปัญหาที่จะทำให้ข้อตกลง (contract) แตก" |
| "เดี๋ยวฉันค่อยไปทำ doubt เอาตอนสุดท้ายด้วยคำสั่ง `/review`" | `/review` คือประตูด่านสุดท้าย Doubt-driven คือการดักจับเส้นทางที่ผิดพลาดตั้งแต่เนิ่นๆ ตอนที่ยังแก้ได้ง่ายๆ รอไปแก้ตอนทำ PR มันสายไปแล้ว |
| "ถ้าฉันมัวแต่สงสัยทุกฝีก้าว ฉันคงไม่ต้องส่งงานพอดี" | ทักษะนี้ใช้กับการตัดสินใจสำคัญๆ (non-trivial) ไม่ใช่ทุกฝีก้าว กลับไปอ่าน "เมื่อใดที่ไม่ควรใช้" ซะ |
| "สองหัวย่อมดีกว่าหัวเดียว" | ไม่เสมอไป ถ้าหัวที่สองดันมีบริบทน้อยกว่าและพ่นแต่ขยะ (noise) ออกมา ให้ไกล่เกลี่ย (Reconcile) แต่อย่าไปยอมตามมันหมด |
| "ผู้ตรวจสอบไม่เห็นด้วย แปลว่าฉันคงทำผิด" | ผู้ตรวจสอบไม่ได้มีบริบทแบบคุณ — ความไม่เห็นด้วยคือข้อมูล ไม่ใช่คำพิพากษา กลับไปอ่านชิ้นงาน จัดหมวดหมู่ แล้วค่อยตัดสินใจ |
| "ใช้ Cross-model ย่อมดีกว่าเสมอ" | Cross-model ช่วยอุดรอยรั่วที่โมเดลตัวเดิมมักจะมองไม่เห็น แต่มันก็เพิ่มค่าใช้จ่ายและความเสี่ยงที่เครื่องมือจะพัง จงเสนอทางเลือกให้ผู้ใช้ในทุกๆ รอบของการโต้ตอบ — ผู้ใช้จะตัดสินใจเองว่าชิ้นงานนั้นคุ้มที่จะทำไหม หน้าที่ของ Agent คือการเสนอทางเลือก ไม่ใช่ไปเป็นผู้ปิดกั้น |
| "ผู้ใช้เคยตอบ 'ตกลง' ไปรอบนึงแล้ว ฉันรัน CLI รัวๆ เลยละกัน" | การอนุญาตคือการอนุญาตแบบครั้งต่อครั้ง ชิ้นงาน, prompt, และ flags มันเปลี่ยนไปตลอดในแต่ละรอบ — ยืนยันคำสั่งเป๊ะๆ กับผู้ใช้ก่อนรันทุกครั้งเสมอ |

## Red Flags (สัญญาณเตือนภัย)

- เสกผู้ตรวจสอบแบบสมองโล่ง เพียงเพื่อมาเช็คเรื่องการเปลี่ยนชื่อไฟล์ หรือจัดฟอร์แมตบรรทัดเดียว
- มองว่าคำพูดจากผู้ตรวจสอบคือประกาศิต โดยไม่ยอมกลับไปอ่านทวนในชิ้นงานอีกรอบ
- วนลูปเกิน >3 รอบ โดยไม่ยอมรายงานผู้ใช้
- ป้อน Prompt ให้ผู้ตรวจสอบว่า "แบบนี้ดีไหม?" แทนที่จะสั่งว่า "จงหาข้อผิดพลาด"
- แอบข้ามการทำ Doubt ในการตัดสินใจที่มีความเสี่ยงสูง เพียงเพราะกดดันเรื่องเวลา
- เสกผู้ตรวจสอบแบบสมองโล่งซ้ำซ้อนกับชิ้นงานเดิมที่ไม่ได้ถูกแก้ (คุณก็จะได้ผลลัพธ์เดิมๆ; นี่คือการถ่วงเวลา)
- **แกล้งทำเป็นรีวิว (Doubt theater - สัญญาณที่จับผิดได้ง่าย)**: รีวิวไปแล้ว 2 รอบหรือมากกว่า เจอข้อผิดพลาดใหญ่ๆ เพียบ แต่ดันจัดว่ามันเป็น "actionable" เท่ากับ 0 (ศูนย์) นี่คุณกำลังหลอกตัวเอง (validating) ไม่ใช่การสงสัย (doubting) หยุดซะ แล้วรายงานผู้ใช้
- เพิ่งมาตั้งข้อสงสัยเอาตอนกด commit ไปแล้ว — นั่นมัน `/review`, ไม่ใช่ doubt-driven development
- ฝังคำสั่งเรียกใช้ external CLI แบบแข็งทื่อ โดยไม่ยอมตรวจสอบกับผู้ใช้ว่า เครื่องมือมีอยู่จริงไหม, ตั้งค่าไว้หรือยัง, และรองรับ syntax นั้นเป๊ะๆ หรือเปล่า
- **แอบข้ามการทำ cross-model แบบเงียบๆ ในเซสชันแบบโต้ตอบ (interactive)** ถึงแม้จะไม่ได้แนะนำให้ทำ แต่ก็ต้องโชว์ข้อเสนอนี้ให้ผู้ใช้เห็น การที่ผู้ใช้กดข้ามคือเรื่องปกติ; แต่การที่บอทแอบข้ามให้เองคือสิ่งที่ไม่ปกติ
- แอบสลับกลับมาใช้โมเดลเดี่ยวอย่างเงียบๆ เมื่อ external CLI error หรือหายไป — จงเปิดเผยข้อผิดพลาด แล้วให้ผู้ใช้เป็นคนบอกทาง
- ดึงเอา Contract ออกจาก input ของผู้ตรวจสอบ
- ส่งข้อ CLAIM ไปให้ผู้ตรวจสอบอ่าน (จะเป็นการชี้นำให้เขาเห็นด้วย)

## Interaction with Other Skills (การใช้งานร่วมกับทักษะอื่น)

- **`code-review-and-quality` / `/review`**: ใช้คู่กันได้ `/review` คือการพิพากษา PR ขั้นสุดท้าย; ส่วน doubt-driven คือการตัดสินใจแบบรายชิ้นระหว่างบิน ใช้ทั้งสองอย่างเลย
- **`source-driven-development`**: SDD ใช้พิสูจน์ *ความจริงเกี่ยวกับ framework* เทียบกับคู่มือทางการ Doubt-driven ใช้พิสูจน์ *ตรรกะเหตุผลที่คุณใส่ลงในชิ้นงาน*. SDD เช็คว่า API มีอยู่จริงไหม; doubt-driven เช็คว่าคุณเรียกใช้มันได้ถูกต้องตาม contract หรือเปล่า
- **`test-driven-development`**: ขั้นตอน RED ของ TDD คือความสงสัยที่ถูกทำให้เป็นรูปธรรม — Test ที่ล้มเหลวคือความพยายามในการพิสูจน์ว่าผิด (disproof) เมื่อไหร่ที่ TDD ออกโรง Test ที่ล้มเหลวนั้นแหละ *คือ* ขั้นตอน doubt สำหรับใช้ยืนยันเรื่องพฤติกรรม
- **`debugging-and-error-recovery`**: เมื่อผู้ตรวจสอบชี้เป้าเห็นการทำงานที่ผิดพลาด (failure mode), ให้สลับไปใช้ทักษะ debugging เพื่อจำกัดวงและซ่อมมัน
- **กฎการสั่งการใน Repo** (`references/orchestration-patterns.md`): ทักษะนี้ต้องถูกสั่งการจากเซสชันหลัก (main session) การให้ Persona เรียกใช้ Persona อีกตัว ถือเป็นการทำผิดกฎข้อ B — ดูที่ Loading Constraints ด้านบน

## Verification

หลังจากประยุกต์ใช้ doubt-driven development:

- [ ] ทุกๆ การตัดสินใจที่สำคัญๆ (non-trivial) (ตามนิยามข้างบน) ถูกประกาศตั้งชื่ออย่างชัดเจนในฐานะ CLAIM ก่อนที่จะรับรอง
- [ ] มีการทำ fresh-context review อย่างน้อย 1 รอบต่อ 1 ชิ้นงานสำคัญ (สำหรับข้อเรียกร้องด้านพฤติกรรม หากผ่านการทำ Failing test ในขั้นตอน RED ของ TDD ถือว่าอนุโลมให้ผ่านข้อนี้ได้ ตามหัวข้อ 'Interaction with Other Skills')
- [ ] ผู้ตรวจสอบได้รับ ARTIFACT + CONTRACT — ไม่ใช่รับข้อ CLAIM, และ ไม่ใช่เหตุผลในใจของคุณ
- [ ] Prompt ที่ส่งให้ผู้ตรวจสอบมีลักษณะแบบจับผิด ("จงหาข้อผิดพลาด"), ไม่ใช่อวย ("มันดีไหม")
- [ ] ข้อค้นพบ ถูกจัดหมวดหมู่โดยการเปรียบเทียบกับตัวชิ้นงาน (ไม่ได้เป็นตรายางแสตมป์รับรอง) โดยใช้ลำดับความสำคัญดังนี้: contract misread / actionable / trade-off / noise
- [ ] บรรลุเงื่อนไขการหยุดแล้ว (เจอแต่เรื่องจิ๊บจ๊อย, ครบ 3 รอบ, หรือผู้ใช้ออกคำสั่งข้าม)
- [ ] ในโหมดโต้ตอบ, เสนอทางเลือก cross-model **อย่างชัดเจน** ให้แก่ผู้ใช้ (ไม่ว่าชิ้นงานจะเสี่ยงมากหรือน้อย) และมีการรับทราบคำตอบใน Output
- [ ] ในโหมดไม่มีการโต้ตอบ (non-interactive), cross-model ถูกข้าม และได้มีการประกาศบอกการข้ามไว้
- [ ] ทุกๆ การเรียกใช้ external CLI มีการตรวจ PATH ก่อนหน้า, มีการทดสอบว่าตัวรันทำงานได้จริง, มีการยืนยันคำสั่ง syntax กับผู้ใช้, และได้รับอนุญาตอย่างชัดเจนให้ทำการรันได้
