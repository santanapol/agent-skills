# Deprecation and Migration

## Overview

โค้ดคือภาระ (liability), ไม่ใช่ทรัพย์สินสมบัติ โค้ดทุกบรรทัดมีค่าบำรุงรักษาอย่างต่อเนื่อง — มีบั๊กต้องคอยตามเช็ดตามล้าง, มี dependencies ต้องกดอัปเดต, มีแพทช์ความปลอดภัยที่ต้องแปะ, และพนักงานใหม่ที่เข้ามาก็ต้องเสียเวลาเรียนรู้มัน Deprecation (การเลิกใช้งาน) คือระเบียบวินัยของการเชือดโค้ดที่ไม่ทำประโยชน์ให้กับตนทิ้งซะ, และ Migration (การอพยพ) คือกระบวนการย้ายผู้คนจากของเก่าไปสู่ของใหม่อย่างปลอดภัย

องค์กรวิศวกรรมส่วนใหญ่ต่างก็เก่งเรื่องการ "สร้าง" ของใหม่ แต่กลับมีน้อยแห่งมากที่เก่งในการ "รื้อ" ของทิ้ง ทักษะนี้ถูกสร้างมาเพื่อมาอุดช่องโหว่นั้น

## When to Use

- โละของเก่า ระบบ, API, หรือ library เพื่อสลับไปใช้ตัวใหม่
- จบยุคฟีเจอร์หมดความจำเป็น (Sunsetting)
- รวบตึงงานหน้าตาเหมือนๆ กันที่ดันไปเขียนแยก (Consolidating duplicate implementations)
- ลบพวกซากโค้ดตายที่หาเจ้าภาพไม่ได้ แต่ดันมีคนอาศัยพึ่งพาเต็มไปหมด
- วางแผนวงจรชีวิตของระบบที่จะเกิดใหม่ (แผนโละทิ้ง(deprecation) เริ่มคิดตั้งแต่วันแรกที่ทำดีไซน์)
- ตัดสินใจชั่งใจระหว่างทนแบกรับระบบตกยุคต่อไป หรือยอมตัดใจจ่ายค่าเหนื่อยอพยพ(migration) ซะ

## Core Principles (หลักการสำคัญ)

### Code Is a Liability (โค้ดคือภาระ)

โค้ดทุกหย่อมหญ้ามีต้นทุนแฝง: มันอยากได้ระบบ test, อยากได้คำอธิบาย(documentation), ต้องตามล้างแพทช์กันแฮก(security), ลำบากอัปเดต dependencies, รวมถึงเปลืองสมองเพื่อนร่วมทีมที่ต้องมานั่งทำความเข้าใจโค้ดรอบๆ คุณค่าของตัวโค้ดอยู่ที่พฤติกรรมการทำงาน(functionality)ของมัน, ไม่ใช่ตัวเนื้อโค้ดเอง เมื่อใดที่พฤติกรรมหน้าตาเดิมสามารถเสกได้ด้วยโค้ดที่สั้นกว่า, โคตรคลีน, หรือมีนามธรรมที่หล่อกว่า — โค้ดเก่าก็ควรปลิวไปซะ

### Hyrum's Law Makes Removal Hard (กฎของไฮรัมทำลายยากกว่าที่คิด)

เมื่อมีผู้ใช้งานมากพอ, ทุกพฤติกรรมที่คุณแสดงออกจะกลายเป็นสิ่งที่ถูกพึ่งพาเสมอ — ไม่เว้นแม้กระทั่ง บั๊ก, จังหวะหน่วงๆ, หรือผลลัพธ์ข้างเคียงที่ไม่ได้ถูกบันทึกไว้ในคู่มือด้วยซ้ำ นี่คือเหตุผลว่าทำไมการทำ deprecation ต้องมาคู่กับการลงพื้นที่ทำ migration จริงจัง, ไม่ใช่แค่ร่อนประกาศบอก ผู้ใช้ไม่สามารถ "จู่ๆ ก็แค่สลับตัว" ได้ ถ้าพวกเขาแอบไปติดใจพึ่งพาพฤติกรรมบางอย่างที่ตัวใหม่ไม่มีเหมือนของเก่า

### Deprecation Planning Starts at Design Time (แผนรื้อถอนเริ่มคิดตอนออกแบบ)

เวลาจะสร้างของใหม่, จงตั้งคำถามไว้เสมอว่า: "อีก 3 ปีเราจะรื้อไอ้ระบบนี้ทิ้งยังไงหว่า?" ระบบที่ถูกออกแบบด้วยสะพานเชื่อมอินเทอร์เฟซ (interfaces) ที่สะอาดตา, มีตัวสับสวิตช์ปิดเปิดฟีเจอร์ (feature flags), และเปิดเผยชิ้นส่วนหน้าด่านให้น้อยที่สุด ย่อมรื้อถอนง่ายกว่าระบบประหลาดๆ ที่แอบคายความลับไส้ในของมันออกมาเกลื่อนกลาดไปทั่วทุ่ง

## The Deprecation Decision (ชั่งใจก่อนยุบ)

ก่อนจะถอดใจเลิกใช้ (deprecating) อะไรสักอย่าง, ให้ตอบคำถามเหล่านี้ให้ชื่นใจก่อน:

```
1. ระบบหน้าโง่นี้ยังมีอะไรดีหลงเหลืออยู่ไหม?
   → ถ้ามี, ก็ทนเลี้ยงมันต่อไป. ถ้าไม่มี, เดินเครื่องรื้อเลย.

2. มีกี่ชีวิตที่พึ่งพามันอยู่?
   → ตีมูลค่าขอบเขตการอพยพว่าเรื่องใหญ่ขนาดไหน.

3. แล้วมีของใหม่ตัวตายตัวแทนหรือยังล่ะ?
   → ถ้ายัง, ก็ไปทำของใหม่ให้เสร็จก่อน. อย่ามาทะลึ่งลบของเก่า ถ้ายังไม่ทีทางเลือกใหม่ให้.

4. ค่าใช้จ่ายชดเชยในการอพยพผู้ใช้แต่ละคน ตกอยู่เท่าไหร่?
   → ถ้าใช้ท่าอัตโนมัติหรือง่ายๆ ก็ลุยเลย. ถ้าต้องทำมือเหงื่อตก, ก็ต้องเอาไปชั่งน้ำหนักกับค่าบำรุงรักษาเดิม.

5. แล้วถ้าไม่ยุบทิ้ง ค่าบำรุงรักษาที่ต้องจ่ายต่อไปมีอะไรบ้างล่ะ?
   → พวกความเสี่ยงจากการโดนแฮก(Security), ค่าแรงเสียเวลาของวิศวกร, เสียโอกาสจากการทนอยู่กับความซับซ้อนยุ่งยาก.
```

## Compulsory vs Advisory Deprecation (บีบบังคับ ปะทะ เชิญชวน)

| หมวด | ใช้ตอนไหน | วิธีการ |
|------|-------------|-----------|
| **เชิงเชิญชวน (Advisory)** | การอพยพไม่บังคับใคร, ระบบเก่าของเขาก็เสถียรดีอยู่แล้ว | แจ้งเตือน (Warnings), เขียนลงเอกสาร, ช่วยสะกิดเรียก. ปล่อยผู้ใช้ย้ายบ้านตามสะดวกตามรอบเวลาของเขาเลย. |
| **เชิงบีบบังคับ (Compulsory)** | ระบบเก่าเริ่มมีช่องโหว่ความปลอดภัย, เตะถ่วงรั้งความเจริญชาวบ้าน, หรือต้นทุนบำรุงรักษามันเกินจะรับไหวแล้ว | ขีดเส้นตาย (Hard deadline) วันประหารชัดเจน. ระบบหน้าโง่นี้จะต้องปลิวหายไปในวันที่ X. แต่กรุณาแจกเครื่องมือช่วยอพยพด้วยล่ะ. |

**เริ่มต้นด้วยความสมัครใจ (Advisory).** งัดมาตรการบังคับ (compulsory) ออกมาใช้เฉพาะในกรณีที่ค่าใช้จ่ายบำรุงรักษาหรือระดับความเสี่ยงมันมากพอที่จะบังคับให้คนอื่นย้ายบ้าน การบังคับให้เลิกใช้งานจำเป็นต้องมอบเครื่องมือช่วยอพยพ, เอกสารคู่มือ, และทีมสนับสนุน — ไม่ใช่แค่ประกาศเส้นตายแล้วเดินหนี

## The Migration Process (กระบวนการอพยพ)

### Step 1: Build the Replacement (สร้างตัวตายตัวแทน)

อย่าปลดระวางใครถ้ายังไม่มีของดีกว่ามาแทน ตัวตายตัวแทนต้อง:

- ทำได้ครบครอบคลุมการใช้งาน(use cases) หลักๆ ทั้งหมดของระบบเก่า
- มีเอกสารคู่มือสอนและการนำทางวิธีการอพยพย้ายถิ่น (migration guides)
- ผ่านการทดสอบจากสนามจริง (ไม่ใช่แค่ "ดีกว่าในทฤษฎี")

### Step 2: Announce and Document (เป่าประกาศและลงบันทึก)

```markdown
## ประกาศโละระบบ (Deprecation Notice): ระบบเก่า (OldService)

**สถานะ:** หมดวาระ ตั้งแต่ 2025-03-01
**ตัวตายตัวแทน:** ระบบใหม่เอี่ยม (NewService) (ดูคู่มือสอนอพยพข้างล่างได้เลย)
**วันตัดหางปล่อยวัด:** ยังแค่แนะนำ — ยังไม่มีการขีดเส้นตายใดๆ
**เหตุผล:** OldService มันต้องใช้มนุษย์มานั่งปรับสเกลเอง แถมส่องดูการทำงานก็ลำบาก
             แต่สำหรับ NewService มันจัดการให้แบบออโต้หมดเลย.

### คู่มือสอนอพยพย้ายบ้าน (Migration Guide)
1. สลับหน้ามือเปลี่ยน `import { client } from 'old-service'` เป็น `import { client } from 'new-service'`
2. ขยับปรับปุ่มตั้งค่า (ดูตามตัวอย่างข้างล่างได้เลย)
3. รันสคริปต์ตรวจความเรียบร้อยการอพยพ: `npx migrate-check`
```

### Step 3: Migrate Incrementally (ค่อยๆ ทยอยอพยพทีละนิด)

ย้ายผู้ใช้ไปทีละคน, ไม่ใช่โยนทุกคนไปพร้อมกันรวดเดียว สำหรับแต่ละผู้ใช้ (consumer):

```
1. ล่าตัวจุดที่ระบบเก่าไปเชื่อมต่อไว้ให้หมด
2. เปลี่ยนมาใช้ตัวตายตัวแทนตัวใหม่แทน
3. เช็คความถูกต้องว่าพฤติกรรมมันโต้ตอบได้เหมือนเป๊ะ (ใช้ tests, หรือ integration checks)
4. ตัดขาดไม่เหลือเยื่อใย รื้อซากการอ้างอิงถึงระบบเก่าทิ้งซะ
5. สังเกตการณ์ยืนยันว่าไม่มีบั๊กใหม่ (regressions) งอกออกมา
```

**กฎแห่งการทิ้งตัว (The Churn Rule):** ถ้าคุณคือเจ้าของโครงสร้างที่กำลังจะถูกปลดระวาง, คุณต้องเป็นคนรับผิดชอบอพยพผู้ใช้ของคุณเอง — หรือไม่ก็ส่งอัปเดตแบบ backward-compatible ไปให้โดยที่พวกเขาไม่ต้องลงมืออพยพเอง อย่าแค่ประกาศว่าจะเลิกใช้แล้วทิ้งให้ผู้ใช้งานงมหาทางเอาเอง

### Step 4: Remove the Old System (ปลิดชีพตัวระบบเก่า)

ต้องทำก็ต่อเมื่อผู้ใช้งานทุกคนอพยพไปหมดแล้วเท่านั้น:

```
1. เช็คเป้าชี้ชัดว่ามียอดการใช้งาน (usage) เป็นศูนย์ 0 จริงๆ (ดูจาก metrics, logs, กราฟ dependency)
2. ถอนรากถอนโคน ลบโค้ดทิ้งเลย
3. ตามลบพวก tests, คู่มือ(documentation), และปุ่มตั้งค่า (configuration) ของเดิมทิ้งด้วย
4. แกะป้ายประกาศโละระบบ (deprecation notices) ออก
5. เลี้ยงฉลอง — การกล้าลบโค้ดทิ้งถือเป็นความสำเร็จอันยิ่งใหญ่
```

## Migration Patterns (รูปแบบการอพยพ)

### Strangler Pattern (ท่ารัดคอ)

ให้ระบบเก่าและระบบใหม่รันคู่ขนานกันไป ค่อยๆ สับเปลี่ยนเส้นทางการจราจรจากเก่าไปใหม่ ทีละนิด ทีละหน่อย เมื่อไหร่ที่ระบบเก่าไม่มีปริมาณวิ่งเข้า (handles 0% of traffic) เมื่อนั้นก็ลบมันทิ้งซะ

```
เฟส 1: ระบบใหม่ทำงาน 0%, ของเก่ารับภาระ 100%
เฟส 2: ระบบใหม่รับไป 10% (canary)
เฟส 3: ระบบใหม่รับไป 50%
เฟส 4: ระบบใหม่รับไป 100%, ของเก่านั่งตบยุง
เฟส 5: ถีบของเก่าส่งท้าย
```

### Adapter Pattern (ท่าปลั๊กแปลงไฟ)

เสกตัวสวมทับ (adapter) ที่คอยทำหน้าที่แปลการใช้งานจากหน้าตาอินเทอร์เฟซแบบเก่าให้วิ่งเข้าไปหาระบบของแบบใหม่ ผู้ใช้ก็ยังคงหน้ามืดตามัวเรียกใช้อินเทอร์เฟซตัวเก่าต่อไป ในขณะที่คุณแอบสลับหลังบ้านเสร็จไปเรียบร้อยแล้ว

```typescript
// Adapter: หน้ากากอันเก่า, ไส้ในอันใหม่เอี่ยม
class LegacyTaskService implements OldTaskAPI {
  constructor(private newService: NewTaskService) {}

  // รูปแบบการเรียกของเดิม, แต่แอบโยนให้ไส้ในตัวใหม่จัดการต่อ
  getTask(id: number): OldTask {
    const task = this.newService.findById(String(id));
    return this.toOldFormat(task);
  }
}
```

### Feature Flag Migration (อพยพสลับสวิตช์ปิดเปิด)

ใช้ feature flags สลับทางย้ายคนจากระบบเก่าไปสู่ระบบใหม่ ทีละคนๆ:

```typescript
function getTaskService(userId: string): TaskService {
  if (featureFlags.isEnabled('new-task-service', { userId })) {
    return new NewTaskService();
  }
  return new LegacyTaskService();
}
```

## Zombie Code (ผีดิบซอมบี้)

โค้ดซอมบี้คือโค้ดที่ไม่รู้หัวนอนปลายเท้าไม่มีใครยอมเป็นเจ้าของ แต่ดันมีชาวบ้านไปพึ่งพาเรียกใช้งานกันเต็มไปหมด มันไม่เคยได้รับการเหลียวแล (maintain), ไม่มีชื่อเจ้าของ(owner), และชอบแอบเพาะพันธุ์เชื้อร้ายทั้งช่องโหว่ความปลอดภัยและจุดขัดแย้งของระบบ อาการสังเกต:

- ไม่มีคอมมิตในรอบ 6+ เดือน แต่ดันมีพวกมาเรียกใช้เรื่อยๆ
- ไม่มีคนถูกระบุให้เป็นผู้ดูแล หรือไม่ได้สังกัดทีมไหนเลย
- ตัวรัน tests พังยับเยินแต่ก็ไม่มีหมาตัวไหนยอมมาซ่อม
- ไส้ใน Dependencies มีช่องโหว่รูเบ้อเริ่มที่รู้กันทั่วโลกแต่ก็ไม่มีใครมากดอัปเดต
- เขียนอธิบายในคู่มือถึงระบบเก่าที่ตายปลิวไปนานแล้ว

**การตอบสนอง (Response):** เลือกว่าจะหาคนมาเป็นพ่อบุญธรรมดูแลมันต่อไปซะ, หรือไม่ก็ส่งมันลงหลุมปลดระวาง(deprecate) พร้อมกับแผนอพยพชัดเจน โค้ดซอมบี้ห้ามปล่อยทิ้งเป็นผีเร่ร่อน — ถ้าไม่ลงทุนปั้นต่อ ก็ปลิดชีพทิ้งซะ

## Common Rationalizations (ข้ออ้างทั่วไป)

## Red Flags (สัญญาณเตือนภัย)

- มีระบบรอสั่งโละทิ้งเต็มไปหมดแต่ดันไม่มีตัวตายตัวแทนมารองรับ
- ป่าวประกาศโละทิ้งอย่างดีแต่ไม่มีห่าเครื่องมือช่วยอพยพหรือหน้าคู่มือใดๆ เลย
- การโละแบบ "หย่อนๆ" เป็นการเตือนให้ย้ายเป็นปีๆ แล้วก็ไม่มีความคืบหน้าอะไรเลย
- โค้ดผีดิบซอมบี้ที่ไร้เจ้าของแต่มีพวกปรสิตพึ่งพาเกาะกิน
- ดันมีหน้าไปเพิ่มฟีเจอร์ใหม่ใส่เข้าไปในระบบที่กำลังจะยุบทิ้ง (เอาเวลาไปทุ่มกับตัวตายตัวแทนดีกว่ามั้ย)
- สั่งโละทิ้งโดยที่ไม่เคยวัดปริมาณผู้ใช้งาน(usage) เลย
- กดปุ่มลบโค้ดทิ้ง ทั้งที่ยังไม่ตรวจสอบเลยว่ามียอดการใช้งาน (consumers) เป็น 0 จริงๆ หรือเปล่า
