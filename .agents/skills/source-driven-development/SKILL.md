# Source-Driven Development

## Overview

ทุกการตัดสินใจเมื่อต้องเขียนโค้ดเฉพาะทางของเฟรมเวิร์ค (framework-specific) จะต้องถูกค้ำยันด้วยเอกสารที่เชื่อถือได้ (official documentation) อย่าพยายามเขียนโค้ดจากความทรงจำเด็ดขาด — ตรวจสอบ, อ้างอิง, และปล่อยให้ผู้ใช้เห็นว่าคุณเอาแหล่งอ้างอิงมาจากไหน ข้อมูลชุดเทรนนิ่งมักจะเก่าไปตามกาลเวลา, ชุด API มักจะตกรุ่น, หลักปฏิบัติที่ดีที่สุดก็ถูกเปลี่ยนไปมาอยู่เรื่อยๆ ทักษะนี้จะรับประกันว่าผู้ใช้จะได้รับโค้ดที่พวกเขาไว้ใจได้ เพราะทุกๆ รูปแบบที่ถูกนำมาใช้สามารถสืบสาวกลับไปยังแหล่งอ้างอิงอันดับหนึ่งที่สามารถตรวจสอบได้เสมอ

## When to Use

- ผู้ใช้ต้องการได้โค้ดที่เป็น "บทปฏิบัติที่ดีที่สุด (best practices)" สำหรับตัว framework นั้นๆ
- ตอนกำลังจะสร้างพวกไฟล์แม่แบบ (boilerplate), ตัวตั้งต้น (starter code), หรือ โค้ดต้นแบบที่จะถูกคัดลอกเอาไปใช้อีกร้อยที่ในโปรเจกต์
- เมื่อผู้ใช้ร้องขอโค้ดที่ 'เชื่อถือได้', 'พิสูจน์มาแล้ว', หรือ 'แม่นยำถูกต้อง'
- กำลังสร้างฟีเจอร์ที่วิธีปฏิบัติของเฟรมเวิร์คนั้นส่งผลชี้เป็นชี้ตาย (การทำฟอร์ม, routing, data fetching, state management, auth)
- ตอนนั่งรีวิว หรือช่วยพัฒนาโค้ดที่ใช้เทคนิคเฉพาะของเฟรมเวิร์ค
- เมื่อใดก็ตามที่คุณตั้งท่าจะลุยเขียนโค้ดตามความจำเก่าๆ ในหัว

**เมื่อใดที่ไม่ควรใช้:**

- ความถูกต้องไม่ได้ไปผูกมัดกับเวอร์ชันใดๆ (เช่น การเปลี่ยนชื่อตัวแปร, แก้คำผิด, ย้ายไฟล์)
- การใช้ตรรกะแบบตรงๆ ทื่อๆ ที่ไม่ว่าจะเวอร์ชันไหนก็เขียนเหมือนกัน (loops, conditionals, data structures)
- ผู้ใช้ตั้งใจสั่งให้ทำแบบเร่งด่วน โดยไม่ต้องเช็คอะไรทั้งนั้น ("รีบๆ ทำให้หน่อย")

## The Process

```
ตรวจสอบ (DETECT) ──→ ค้นหา (FETCH) ──→ ลงมือทำ (IMPLEMENT) ──→ แนบอ้างอิง (CITE)
       │                 │                      │                   │
       ▼                 ▼                      ▼                   ▼
    สแตคอะไร?       ดึงเอกสารมาซะ          เขียนโค้ดตามเอกสาร       โชว์เครดิตที่มา
```

### Step 1: Detect Stack and Versions (ตรวจสอบเทคโนโลยีและเวอร์ชัน)

อ่านไฟล์จัดการส่วนเสริม (dependency file) ของโปรเจกต์ เพื่อดึงเวอร์ชันเป๊ะๆ ของเทคโนโลยีออกมา:

```
package.json    → Node/React/Vue/Angular/Svelte
composer.json   → PHP/Symfony/Laravel
requirements.txt / pyproject.toml → Python/Django/Flask
go.mod          → Go
Cargo.toml      → Rust
Gemfile         → Ruby/Rails
```

แล้วรายงานสรุปออกมาให้ชัดเจนว่าเจออะไรบ้าง:

```
สแตคที่เจอ (STACK DETECTED):
- React 19.1.0 (พบใน package.json)
- Vite 6.2.0
- Tailwind CSS 4.0.3
→ กำลังดึงเอกสารคู่มือทางการ (official docs) เพื่อหาวิธีเขียนที่ถูกต้อง
```

หากเวอร์ชันหายไปหรือไม่ชัดเจน **ให้ถามผู้ใช้** อย่าสุ่มเดา — การรู้เลขเวอร์ชันคือตัวชี้วัดว่าควรจะเขียนโค้ดออกมาในรูปแบบไหน

### Step 2: Fetch Official Documentation (ค้นหาคู่มือทางการ)

ไปดึงเอาหน้าเอกสารคู่มือเฉพาะของเรื่องที่คุณจะทำ ไม่ใช่ไปเอาหน้าโฮมเพจมา ไม่ใช่ไปเอาคู่มือตัวเต็มมา — เอาเฉพาะหน้าที่มันเกี่ยว

**ลำดับความน่าเชื่อถือของแหล่งข้อมูล (จากมากไปน้อย):**

| ความสำคัญ | แหล่งข้อมูล | ตัวอย่าง |
|----------|--------|---------|
| 1 | คู่มือหลักทางการ (Official documentation) | react.dev, docs.djangoproject.com, symfony.com/doc |
| 2 | บล็อกของระบบ / กระดานข่าวอัปเดต (Official blog / changelog) | react.dev/blog, nextjs.org/blog |
| 3 | คู่มือมาตรฐานทางอินเตอร์เน็ต (Web standards references) | MDN, web.dev, html.spec.whatwg.org |
| 4 | ตารางความเข้ากันได้ของบราวเซอร์ (Browser/runtime compatibility) | caniuse.com, node.green |

**สิ่งที่ไม่น่าเชื่อถือ — ห้ามนำมาใช้อ้างอิงเป็นเอกสารหลักเด็ดขาด:**

- คำตอบใน Stack Overflow 
- บทความตาม Blog posts หรือ บทสอน (tutorials) (ถึงแม้มันจะดังก็เถอะ)
- เอกสารสรุปเนื้อหาที่เกิดจาก AI ปั้นขึ้นมา
- ข้อมูลเก่าที่ติดอยู่ในหัวของคุณเองจากชุดเทรนนิ่ง (นั่นคือประเด็นหลักเลยนะ — คุณต้องพิสูจน์ความจริงสิ)

**ระบุความต้องการในการดึงข้อมูลให้แม่นยำ:**

```
แย่ (BAD):  ดึงเอกสารหน้าโฮมเพจ React (Fetch the React homepage)
ดี (GOOD):  ดึงเอกสารหน้า react.dev/reference/react/useActionState

แย่ (BAD):  เสิร์ช "django authentication best practices"
ดี (GOOD):  ดึงหน้า docs.djangoproject.com/en/6.0/topics/auth/
```

หลังจากที่ได้เอกสารมาแล้ว ให้ถอดรูปแบบที่ถูกต้องออกมา พร้อมจดพวกคำเตือนโค้ดตกยุค (deprecation warnings) หรือคำแนะนำตอนย้ายระบบ (migration guidance) ติดมาด้วย

หากคู่มือทางการสองหน้าดันขัดแย้งกันเอง (เช่น ข้อมูลหน้า migration ดันเขียนไม่เหมือนข้อมูลในหน้า API reference) ให้รายงานปัญหานี้ให้ผู้ใช้ทราบ แล้วตรวจสอบดูว่ารูปแบบไหนที่ใช้งานได้จริงในเวอร์ชันที่คุณใช้อยู่

### Step 3: Implement Following Documented Patterns (สร้างงานโดยล้อตามรูปแบบที่คู่มือบอก)

เขียนโค้ดของคุณให้ออกมาหน้าตาตรงตามที่คู่มือระบุไว้:

- ลอกวิธีการส่งตัวแปร (API signatures) มาจากคู่มือ อย่าเขียนตามความทรงจำในหัว
- ถ้าคู่มือเสนอวิธีใหม่ที่เจ๋งกว่า ให้ใช้วิธีใหม่
- ถ้าคู่มือบอกว่าอย่าใช้วิธีนี้เพราะล้าหลัง (deprecate) ก็ห้ามใช้
- ถ้าคู่มือไม่ได้ครอบคลุมเรื่องที่คุณกำลังทำ ให้แปะป้ายแจ้งเตือนไว้ว่าสิ่งนี้ไม่มีอ้างอิงยืนยัน (unverified)

**เมื่อเอกสารคู่มือดันขัดแย้งกับโค้ดที่มีในโปรเจกต์:**

```
ตรวจพบความขัดแย้ง (CONFLICT DETECTED):
โค้ดในโปรเจกต์ของคุณใช้ `useState` ในการควบคุมสถานะตอนโหลดแบบฟอร์ม,
แต่เอกสาร React 19 แนะนำให้ใช้ `useActionState` แทน
(อ้างอิง: react.dev/reference/react/useActionState)

ทางเลือก:
A) ใช้วิธีสมัยใหม่เลย (useActionState) — ซึ่งจะตรงกับคำแนะนำในคู่มือ
B) เอาให้เหมือนโค้ดเก่าในโปรเจกต์ (useState) — เพื่อให้สไตล์งานยังคงเดิม
→ คุณอยากให้ทำแบบไหนดี?
```

เปิดเผยความขัดแย้งออกมา อย่าสุ่มเลือกเองเงียบๆ

### Step 4: Cite Your Sources (อ้างอิงแหล่งข้อมูลของคุณ)

ทุกๆ การเขียนโค้ดที่อ้างอิงเฟรมเวิร์คเฉพาะ จะต้องมีการแนบแหล่งอ้างอิง(citation)เสมอ ผู้ใช้จะต้องสามารถกดลิงก์ตรวจสอบได้ด้วยตัวเองในทุกๆ การตัดสินใจ

**อยู่ในบรรทัดคอมเมนต์ของโค้ด (In code comments):**

```typescript
// ทำระบบฟอร์มของ React 19 ด้วย hook `useActionState`
// อ้างอิง: https://react.dev/reference/react/useActionState#usage
const [state, formAction, isPending] = useActionState(submitOrder, initialState);
```

**อยู่ในบทสนทนา (In conversation):**

```
ฉันเลือกที่จะใช้ `useActionState` แทนการทำ `useState` ด้วยตัวเอง
เพื่อจัดการสถานะของฟอร์มนะ เพราะ React 19 ได้โละการทำแบบเดิม (isPending/setIsPending) ทิ้ง
แล้วให้มาใช้ hook ตัวนี้แทน

อ้างอิง: https://react.dev/blog/2024/12/05/react-19#actions
"useTransition now supports async functions [...] to handle
pending states automatically"
```

**กฎของการสร้างตัวอ้างอิง (Citation rules):**

- ต้องเป็น URLs แบบเต็ม ห้ามใช้ลิงก์ย่อ
- ให้ความสำคัญกับการลิงก์ตรงดิ่งเจาะลึก (deep links) ไปยังจุดที่ต้องการให้มากที่สุด (ตัวอย่าง: ใช้ `/useActionState#usage` แทนที่จะโยนไปแค่ `/useActionState`) — เพราะจุดอ้างอิง (anchors) มักจะทนต่อการโดนจัดหน้าเพจใหม่ได้ดีกว่าลิงก์หลักๆ
- แปะข้อความคำคมจากบทความนั้น (Quote) มาด้วย ถ้าการตัดสินใจนั้นแอบดูไม่ค่อยสมเหตุสมผล 
- แปะเรื่องการรองรับในบราวเซอร์มาด้วย ถ้าคุณกำลังเชียร์ให้ใช้ฟีเจอร์ระดับบราวเซอร์
- ถ้าคุณหาคู่มือที่ใช้อ้างอิงไม่เจอ ให้บอกผู้ใช้ไปตามตรง:

```
ไม่ได้รับการยืนยัน (UNVERIFIED): ฉันไม่สามารถหาเอกสารทางการที่พูดถึง
วิธีการเขียนรูปแบบนี้ได้ โค้ดนี้ถูกดึงมาจากความจำในหัว (Training data) ซึ่งอาจจะล้าสมัยไปแล้ว
โปรดตรวจสอบให้แน่ใจก่อนนำไปใช้บน Production
```

ความซื่อสัตย์ในการบอกว่าหาหลักฐานอ้างอิงไม่เจอ มีค่ามากกว่าการอวดอ้างแบบส่งเดช

## Common Rationalizations (ข้ออ้างทั่วไป)

| ข้ออ้าง | ความเป็นจริง |
|---|---|
| "ฉันมั่นใจสุดๆ เรื่อง API ตัวนี้" | ความมั่นใจไม่ใช่หลักฐาน ข้อมูลเทรนนิ่งมักจะมี Pattern โค้ดตกยุคหลอกหลอนอยู่เต็มไปหมด ซึ่งใช้กับเวอร์ชันปัจจุบันไม่ได้หรอกนะ จงตรวจสอบก่อน |
| "การไปดึงเอกสารมันสิ้นเปลือง Tokens" | การให้บอทนั่งมโน API เปลืองกว่าเยอะ ผู้ใช้ต้องมานั่งเสียเวลาดีบักเป็นชั่วโมงเพื่อจะค้นพบว่าอ้อ ฟังก์ชันมันแอบเปลี่ยนชื่อนี่เอง การเช็คเอกสารแค่ครั้งเดียวจะช่วยประหยัดเวลาการแก้งานได้มหาศาล |
| "คู่มือคงไม่ได้มีทุกอย่างที่ฉันต้องการหรอก" | ถ้าคู่มือไม่มีบอกไว้ นั่นแปลว่าคุณได้ข้อมูลสำคัญแล้วล่ะ — แปลว่ารูปแบบที่คุณพยายามจะเขียน มันอาจจะไม่ได้รับการยอมรับหรือใช้งานอย่างเป็นทางการ |
| "เดี๋ยวฉันค่อยพิมพ์เตือนว่าโค้ดมันอาจจะเก่าก็ได้" | ข้อความกันหมาไม่ได้ช่วยอะไร ไม่ลงมือตรวจสอบพร้อมแนบลิงก์ ก็จงติดป้ายเตือนเลยว่านี่คือสิ่งที่ยังไม่ยืนยัน การเขียนแบบกั๊กๆ คือทางเลือกที่เลวร้ายที่สุด |
| "แค่งานง่ายๆ, ไม่จำเป็นต้องเช็คหรอก" | งานง่ายๆ ที่เขียนผิดรูปแบบ มักจะถูกลอกเอาไปทำเทมเพลตซ้ำๆ ผู้ใช้อาจจะก๊อปฟอร์มยุคหินของคุณไปวางซ้ำๆ ถึงสิบที่ ก่อนที่จะรู้ตัวว่าเอ๊ะ มันมีวิธีสมัยใหม่ที่ง่ายกว่านี้นี่หว่า |

## Red Flags (สัญญาณเตือนภัย)

- นั่งพิมพ์โค้ดของเฟรมเวิร์คเฉพาะ โดยไม่ได้เปิดเช็คคู่มือว่าต้องเขียนแบบไหนสำหรับเวอร์ชันนั้นๆ
- ขึ้นประโยคว่า "ฉันเชื่อว่า" หรือ "ฉันคิดว่า" เมื่อคุยเรื่อง API แทนที่จะอ้างอิงแหล่งที่มา
- ลงมือเขียนโค้ดโดยไม่รู้ว่ามันถูกใช้สำหรับเวอร์ชันไหน
- อ้างอิงลิงก์จาก Stack Overflow หรือ blog posts แทนที่จะเป็นเอกสารทางการ
- ใช้งาน APIs รุ่นคุณปู่ (deprecated) เพราะถูกหลอกด้วยความจำเก่าๆ
- ไม่ยอมเปิดอ่าน `package.json` / หรือ dependency files อื่นๆ ก่อนลงมือ
- ส่งมอบงานโค้ด โดยไม่ยอมแปะลิงก์ว่าเอาวิธีคิดพวกนี้มาจากที่ไหน
- ไปสูบดึงคู่มือมาหมดทั้งหน้าโฮมเพจ ทั้งที่ต้องการใช้แค่หน้าเพจเดียว

## Verification

หลังจากลงมือเขียนโค้ดด้วยแนวคิด source-driven development:

- [ ] อ่านระบุเวอร์ชันของ เฟรมเวิร์ค/library ออกมาจาก dependency file เรียบร้อยแล้ว
