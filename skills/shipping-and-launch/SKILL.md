# Shipping and Launch

## Overview

ปล่อยยานด้วยความมั่นใจ (Ship with confidence) เป้าหมายไม่ใช่แค่การกด deploy — แต่มันคือการปล่อยระบบอย่างปลอดภัย, โดยมีระบบจอมอนิเตอร์คอยเฝ้ามอง, มีแผนถอยทัพรอสแตนด์บาย, และความเข้าใจอย่างถ่องแท้ว่าชัยชนะนั้นหน้าตาเป็นอย่างไร ทุกๆ การปล่อยตัวควรจะสามารถย้อนกลับได้ (reversible), สามารถส่องมองผลลัพธ์ได้ (observable), และปล่อยทยอยฉายแบบมีลำดับขั้น (incremental)

## When to Use

- การ Deploy ฟีเจอร์ขึ้นหน้าจอ production เป็นครั้งแรก
- การปล่อยตัวอัปเดตโค้ดครั้งมโหฬาร ให้ชาวโลกรับรู้
- จังหวะอพยพข้อมูลดิบ (data) หรืองานเสาเข็มโครงสร้างพื้นฐาน (infrastructure)
- การสับสวิตช์เปิดโครงการตัวทดลอง (beta / early access program)
- อะไรก็ตามแต่ที่มีความเสี่ยงแฝงอยู่ (ซึ่งแม่งก็คือทุกอันนั่นแหละ)

## The Pre-Launch Checklist (เช็คลิสต์ก่อนพุ่งทะยาน)

### Code Quality (ความคลีนของโค้ด)

- [ ] ทุก Tests รันผ่านหมด (unit, integration, e2e)
- [ ] Build สำเร็จโดยไม่มีคำขู่คำเตือนโผล่มาเลย (no warnings)
- [ ] Lint และ type checking แจ่มใส
- [ ] โค้ดได้รับการตรวจตรา(reviewed) และประทับตราผ่าน(approved) จากผู้มีบุญ
- [ ] ไม่มีเงาคอมเมนต์ TODO ที่สมควรจะทำเสร็จไปนานแล้วก่อนปล่อย
- [ ] ไม่ทิ้งซากปลอกกระสุน `console.log` แอบอยู่ใน production
- [ ] สร้างบ่อดักตกสวรรค์ (Error handling) ครอบคลุมท่าลงจอดที่คิดว่าน่าจะพังไว้หมดแล้ว

### Security (ความปลอดภัย)

- [ ] ไม่มีตู้เซฟความลับใดๆ โผล่หลุดมาในโค้ดหรือหลุดบนระบบอัปโหลด (version control)
- [ ] กดเช็ค `npm audit` แล้วไม่มีระดับโคตรวิกฤต(critical) หรือระดับสูง(high) ติดมาด้วย
- [ ] วางระบบกรองคำหยาบ (Input validation) ตรงจุดประตูด่านหน้าที่ผู้ใช้ต้องแหย่เข้ามา
- [ ] ป้อมยามตรวจบัตร(Authentication) และแบ่งระดับชั้น(authorization) ทำงานอย่างแข็งขัน
- [ ] ติดตั้ง Security headers เรียบร้อย (CSP, HSTS, ฯลฯ)
- [ ] ระบบกันสแปมยิงรัว (Rate limiting) ตรงหน้าประตูเข้าสู่ระบบทำงาน
- [ ] ปรับตั้งค่า CORS ให้ล็อกเป้าเฉพาะคนที่รู้จัก (ห้ามเปิดเสรี wildcard เด็ดขาด)

### Performance (ความเร็วระดับจรวด)

- [ ] คะแนน Core Web Vitals ยังคงดูดีมีชาติตระกูล (Good)
- [ ] โบกมือลา N+1 queries ขวางทางจราจรเส้นหลัก
- [ ] รูปภาพผ่านการฉีดโบท็อกซ์ลดขนาดแล้ว (บีบไฟล์ compression, กะขนาดหน้าจอ responsive sizes, และรันตอนเห็น lazy loading)
- [ ] หุ่น Bundle size อยู่ในเกณฑ์งบประมาณ
- [ ] ตัว queries ดาต้าเบสมีดัชนี (indexes) คอยหนุนหลัง
- [ ] ติดสปริงช่วยเด้ง Caching ให้กับไฟล์ตายตัว(static assets) และคิวรีเดิมๆ

### Accessibility (ทางลาดสำหรับทุกคน)

- [ ] แป้นคีย์บอร์ด (Keyboard navigation) สามารถตะลุยไปได้ทุกปุ่มที่ต้องคลิก
- [ ] เครื่องอ่านจอ (Screen reader) แปลงร่างเป็นเสียงบอกสาระหน้าเพจได้ถูกต้อง
- [ ] สีสันตัดกันได้มาตรฐาน (Color contrast) ตามสูตร WCAG 2.1 AA (4.5:1 สำหรับข้อความ)
- [ ] กล่องโฟกัส (Focus management) ทำงานถูกต้องกับพวกหน้าต่างเด้ง(modals) หรืออะไรที่โผล่มาเซอร์ไพรส์
- [ ] คำเตือน Error (Error messages) อธิบายชัดเจนและไปผูกพันใกล้ชิดกับกล่องที่ถูกกรอกผิด
- [ ] ไม่มีเสียงกรีดร้องเตือนเรื่องการเข้าถึง(accessibility) จากพี่ axe-core หรือ Lighthouse

### Infrastructure (เสาเข็มโครงสร้าง)

- [ ] เตรียมตัวแปร (Environment variables) สำหรับตัว production ไว้ให้เรียบร้อยแล้ว
- [ ] นำดาต้าเบสเวอร์ชันใหม่ขึ้นฝังร่างแล้ว (หรืออยู่ในสภาพพร้อมลุยทันที)
- [ ] DNS และใบประกัน SSL ถูกปักหมุดแล้ว
- [ ] กระจายตัวไฟล์ (CDN) สำหรับพวกไฟล์ตายตัวเสร็จสรรพ
- [ ] ติดตั้งจอมอนิเตอร์ตรวจจับ log และ error
- [ ] ด่านเช็คสุขภาพหน้าบ้าน (Health check endpoint) สามารถตะโกนตอบโต้ได้

### Documentation (เอกสารคู่มือ)

- [ ] ตัว README ถูกปรับปรุงเพิ่มคู่มือการเซ็ตอัปเรียบร้อย
- [ ] หน้าตา API documentation สดใหม่
- [ ] ADRs ใบอนุญาตถูกจรดปากกาเขียนเพื่อตัดสินใจลงเสาเข็มหลัก
- [ ] หน้า Changelog ถูกประดับเพิ่มแล้ว
- [ ] เอกสารคู่มือสั่งสอนผู้ใช้ (User-facing documentation) ถูกอัปเดตใหม่ (ถ้ามี)

## Feature Flag Strategy (กลยุทธ์การปักธงฟีเจอร์)

จับผลผลิตแอบซ่อนอยู่ข้างหลังธง(feature flags) เพื่อแยกตัวมันออกจากการปล่อยให้เห็น(deployment vs release):

```typescript
// สุ่มตรวจธง (Feature flag check)
const flags = await getFeatureFlags(userId);

if (flags.taskSharing) {
  // ฟีเจอร์ใหม่เอี่ยม: หน้าต่างแชร์
  return <TaskSharingPanel task={task} />;
}

// ค่าตั้งต้น: หน้าตาดั้งเดิมของมัน
return null;
```

**วงจรชีวิตของธง (Feature flag lifecycle):**

```
1. ปล่อยขึ้น DEPLOY แบบหุบธง (flag OFF)     → โค้ดหลุดไปอยู่บน production แล้วนะ แต่นอนหลับพักผ่อนอยู่
2. กางธงให้ ENABLE สำหรับทีม/เบต้า     → ให้ลองกดเทสพังกันเองวงใน บนเซิร์ฟเวอร์จริง
3. ทยอยกาง (GRADUAL ROLLOUT)          → 5% → 25% → 50% → 100% ต่อสายตาผู้ใช้ทั่วไป
4. เฝ้ามอง (MONITOR) ทุกย่างก้าว    → จ้องมองตัวเลข error, ความเร็ว, เสียงก่นด่าของผู้ใช้งาน
5. ล้างกระดาน (CLEAN UP)                 → โละระบบธงทิ้งให้หมด รวมทั้งถางทางโค้ดเก่า เมื่อคุณยึดครองพื้นที่ครบ 100% 
```

**กฎ (Rules):**
- ธงทุกผืนมีเจ้าของเป็นตัวเป็นตน และมีวันหมดอายุของตัวมันเอง
- เก็บกวาดเช็ดถูทำลายธงทิ้ง ภายใน 2 สัปดาห์ หลังจากเปิดไฟเขียว rollout เต็มตัว 100% แล้ว
- ห้ามจับธงมาเสียบซ้อนกันเด็ดขาด (ทำให้เกิดผลกระทบสลับซับซ้อนทวีคูณ)
- จงทดสอบธงทั้ง 2 สถานะ (เปิดและปิด) ในการทำ CI เสมอ

## Staged Rollout (ปล่อยแผ่รัศมีทีละชั้น)

### The Rollout Sequence (ลำดับขั้นการปล่อยทีละชั้น)

```
1. ปล่อยร่างไป DEPLOY ในคอก staging
   └── สาด test suite ใส่แบบเต็มพิกัดในเขต staging 
   └── ส่งมนุษย์มาดมกลิ่นควันปืนลอง (smoke test) กับฟีเจอร์หลักๆ 

2. ปล่อยร่างไป DEPLOY หน้าสมรภูมิ production (กางธง flag OFF เอาไว้ก่อน)
   └── จับชีพจรการลงพื้นว่าราบรื่นดีไหม (health check)
   └── จ้องตาเขม็งที่จอมอนิเตอร์ error (ต้องไม่มี error แปลกๆ โผล่)

3. เบิกเนตร ENABLE ให้คนกันเองในทีม (กางธง flag ON เฉพาะคนใน)
   └── ทีมงานทดสอบฟีเจอร์กันกลางสมรภูมิ production จริง
   └── จับตาดูตลอด 24 ชั่วโมงเต็ม

4. ส่งนกกระเรียน CANARY ไปส่อง (กางธง flag ON ให้ผู้ใช้สัก 5%)
   └── จับตาดูยอด error, หน้าเว็บหน่วงไหม, และส่องพฤติกรรมผู้ใช้
   └── เปรียบเทียบตัวเลข: ระหว่างนกกระเรียน vs. นกเดิม
   └── ปล่อยบินทดสอบ 24-48 ชั่วโมง
   └── จะก้าวต่อไปได้ก็ต่อเมื่อคะแนนผ่านเกณฑ์ที่ตั้งไว้เท่านั้น (ดูตารางคะแนนด้านล่างประกอบ)

5. ทยอยกางปีก (GRADUAL) ขยายวงกว้าง (25% -> 50% -> 100%)
   └── จับตามองทุกย่างก้าวเหมือนเดิมเป๊ะ
   └── กำกุญแจพร้อมจะกระชากถอยกลับ (roll back) เปอร์เซ็นต์ลงได้ทุกเสี้ยววินาที

6. ยึดสมรภูมิ 100% FULL rollout (กางธง flag ON กระแทกหน้าผู้ใช้ทุกคน)
   └── จับตามองต่อเนื่องอีก 1 สัปดาห์เต็ม
   └── ถอนรากถอนโคน feature flag ทิ้งซะ
```

### Rollout Decision Thresholds (เกณฑ์การชี้ชะตาว่าจะปล่อยต่อหรือดึงกลับ)

เอาเกณฑ์พวกนี้มากางดูเลยว่าจะตัดสินใจ ก้าวต่อ, ยั้งทัพ, หรือสับเกียร์ถอยกลับ (roll back) ในแต่ละจังหวะ:

| เกณฑ์ชี้วัด (Metric) | ก้าวต่อ (ไฟเขียว) | ยั้งทัพรอดูลาดเลา (ไฟเหลือง) | ดึงกลับด่วน (ไฟแดง) |
|--------|-----------------|-------------------------------|-----------------|
| ค่า Error (Error rate) | สวิงอยู่ในช่วง 10% จากเดิม | โดดขึ้น 10-100% จากเดิม | กระฉูดทะลุ >2 เท่าตัว |
| ปิงความหน่วง (P95 latency) | สวิงอยู่ในช่วง 20% จากเดิม | โดดขึ้น 20-50% จากเดิม | อืดทะลุ >50% จากเดิม |
| เออเร่อหน้าบ้าน (Client JS errors) | ไร้เงา error หน้าใหม่โผล่ | มี error โฉมใหม่โผล่มาแบบจิ๊บๆ <0.1% ของ sessions | มี error โฉมใหม่โผล่มากระทบ >0.1% ของ sessions |
| ตัวชี้วัดทางธุรกิจ (Business metrics) | เงียบสงบ หรือกราฟหัวพุ่งขึ้น | ร่วงหล่นไม่เกิน <5% (อาจจะแค่ความผันผวนปกติ) | ร่วงหนักกว่า >5% ดึงกลับเลย |

### When to Roll Back (จังหวะกระชากถอยกลับ)

สับเกียร์ถอย (Roll back) แบบไม่คิดชีวิต ทันทีที่เจอ:
- ยอด Error ทะยานขึ้นปรี๊ด 2 เท่าตัวของยอดเดิม
- ปิงความหน่วง (P95 latency) อืดขึ้นมากกว่า 50%
- ได้รับสายด่วนแจ้งปัญหาจากลูกค้า ถาโถมเข้ามา
- ข้อมูลสำคัญของระบบตีกันเละเทะ
- เจอช่องโหว่ความปลอดภัยระดับชาติหลุดรอดไป

## Monitoring and Observability (ตาข่ายสอดส่องและจอมอนิเตอร์)

### What to Monitor (ส่องอะไรดี)

```
ตัวเลขแอปพลิเคชัน (Application metrics):
├── ยอด Error (รวมทั้งหมด และแยกตามจุด endpoint)
├── ระยะเวลาตอบโต้ Response time (p50, p95, p99)
├── ยอดการร้องขอเข้ามา (Request volume)
├── คนออนระบบ (Active users)
└── ตัวเลขทำกินทางธุรกิจ (ยอดสั่งซื้อ, ยอดกดไลก์)

ตัวเลขโครงสร้าง (Infrastructure metrics):
├── พลังการซด CPU และ memory
├── อาการสำลักของท่อต่อ Database (Database connection pool)
├── ที่เหลือในฮาร์ดดิสก์ (Disk space)
├── อาการแล็กของ Network
└── คิวที่ยังไม่ได้เคลียร์ (ถ้ามีระบบคิว)

ตัวเลขฝั่งหน้าบ้านลูกค้า (Client metrics):
├── คะแนนเว็บ (Core Web Vitals - LCP, INP, CLS)
├── บั๊กฝั่ง JavaScript
├── API errors จากฝั่งลูกค้าที่ตะโกนบอกเรา
└── เวลาที่ใช้โหลดหน้าเพจ
```

### Error Reporting (ระบบตะโกนแจ้งเหตุ)

```typescript
// เซ็ตอัปการวางขอบเขตจับแพะ (error boundary) หน้าที่คอยตะโกนบอก
class ErrorBoundary extends React.Component {
  componentDidCatch(error: Error, info: React.ErrorInfo) {
    // โยนรายงานไปที่หน่วยงานตรวจสอบ (error tracking service)
    reportError(error, {
      componentStack: info.componentStack,
      userId: getCurrentUser()?.id,
      page: window.location.pathname,
    });
  }

  render() {
    if (this.state.hasError) {
      return <ErrorFallback onRetry={() => this.setState({ hasError: false })} />;
    }
    return this.props.children;
  }
}

// หน่วยรับเรื่องหน้าจอฝั่งหลังบ้าน (Server-side error reporting)
app.use((err: Error, req: Request, res: Response, next: NextFunction) => {
  reportError(err, {
    method: req.method,
    url: req.url,
    userId: req.user?.id,
  });

  // ห้ามหลุดความลับหลังบ้านให้ลูกค้าเห็นนะเว้ย
  res.status(500).json({
    error: { code: 'INTERNAL_ERROR', message: 'Something went wrong' },
  });
});
```

### Post-Launch Verification (ภารกิจสอดส่องหลังกดยิง)

ใน 1 ชั่วโมงแรกหลังจากกดปล่อย (launch):

```
1. กดเช็คชีพจร ด่าน health endpoint เพื่อดูว่ามันยิ้มตอบ 200 หรือยัง
2. ส่องแดชบอร์ด error (ต้องไม่มีของเล่นใหม่ error หน้าแปลกๆ โผล่)
3. ส่องแดชบอร์ดความเร็ว (latency) (อย่าให้มันอืดลงนะ)
4. ลองเอามือกดจำลองบทบาทผู้ใช้งานจุดสำคัญๆ ดู
5. นั่งส่องหน้าต่าง logs ดูบรรทัดมันวิ่งและอ่านให้รู้เรื่อง
6. เช็คให้ชัวร์ว่าปุ่มถอยทัพ (rollback) ยังทำงานได้ (ถ้าซ้อมกดเล่น dry run ก่อนได้จะดีมาก)
```

## Rollback Strategy (แผนสับเกียร์ถอยหลัง)

ทุกๆ การปล่อยทัพต้องมีแผนถอยหลังไว้เสมอก่อนจะได้ฤกษ์กดปุ่มจริง:

```markdown
## แผนสับเกียร์ถอยหลัง Rollback สำหรับ ฟีเจอร์ [Feature/Release]

### Trigger Conditions (จุดชี้ชะตาให้ถอย)
- ยอด Error > กระฉูด 2 เท่าตัว
- ปิงความหน่วง (P95 latency) > ล่อไป [X]ms
- โดนลูกค้าโวยวายแจ้งรัวๆ ว่าเจอปัญหา [ระบุมาตรงนี้]

### Rollback Steps (สเตปการถอย)
1. สับสวิตช์ฟีเจอร์ลง (Disable feature flag) (ถ้ามีให้ใช้)
   หรือ (OR)
1. กดปล่อยเวอร์ชันก่อนหน้ามาทับซะ: `git revert <commit> && git push`
2. ตรวจผลงานการถอย: health check, ส่องแดชบอร์ด error
3. ป่าวประกาศ: แจ้งลูกเรือว่าเราสั่งถอยทัพแล้วนะ

### Database Considerations (ข้อควรระวังสำหรับดาต้าเบส)
- ถอยการย้ายข้อมูล Migration [X] ด้วย: `npx prisma migrate rollback`
- ข้อมูลขยะที่ดันถูกฟีเจอร์นี้ฝังไว้แล้ว: [จะเก็บไว้รอ / หรือล้างบางทิ้ง]

### Time to Rollback (เวลาที่ใช้ตั้งสติก่อนถอยเสร็จ)
- ถอยด้วย Feature flag: < ไวกว่า 1 นาที
- เสกของเก่า (Redeploy) มาทับ: < ไวกว่า 5 นาที
- รื้อถอนฐานข้อมูล Database rollback: < ไวกว่า 15 นาที
```

## Common Rationalizations (ข้ออ้างทั่วไป)

| ข้ออ้าง | ความเป็นจริง |
|---|---|
| "รันบน staging ผ่านฉลุยแล้วนี่, มันก็ต้องรอดบน production แหละ" | สนามรบ Production มันเป็นคนละโลก ข้อมูลต่างกัน, พฤติกรรมการโดนถล่มไม่เหมือนกัน, มีเคสประหลาดล้านแปด จงจ้องมันหลังปล่อยไปแล้วด้วย. |
| "ฟีเจอร์เล็กๆ แค่นี้ ไม่ต้องง้อ feature flags หรอกน่า" | ทุกฟีเจอร์คู่ควรกับสวิตช์ปิดฉุกเฉิน (kill switch) การเปลี่ยนแปลงที่ดูว่า "เล็กๆ" นี่แหละ ตัวทำบรรลัยเลย. |
| "เอาเวลาไปนั่งติดจอมอนิเตอร์ มันเปลืองแรงจะตาย" | การที่คุณไม่ติดจอมอนิเตอร์ แปลว่าคุณจะต้องมารู้ตัวว่าพังก็ตอนที่ลูกค้ารุมสวดด่าคุณแล้วไงล่ะ. |
| "เดี๋ยวเราค่อยไปติดจอมอนิเตอร์เอาวันหลังละกัน" | ติดซะตั้งแต่ก่อนจะกดยิงปล่อยระบบสิ. คุณไม่มีทางแกะรอยแก้บั๊กในสิ่งที่คุณมองไม่เห็นหรอกนะ. |
| "การยอมกดย้อนกลับถอยทัพ มันคือการยอมรับความพ่ายแพ้นะเว้ย" | การกดย้อนกลับมันคือความรับผิดชอบของวิศวกร การที่เอ็งฝืนปล่อยฟีเจอร์พังๆ ออกไปนั่นแหละคือความพ่ายแพ้ของจริง. |

## Red Flags (สัญญาณเตือนภัย)

- นำขึ้น Deploy โดยไม่มีการเตรียมแผนถอยทัพ
- ปล่อยสมรภูมิ Production ว่างเปล่าไร้จอมอนิเตอร์และไร้เสียงตะโกนแจ้ง error
- ปล่อยบอมบ์แบบตูมเดียวจบ (Big-bang) (ลงทีเดียวหมดหน้าตัก, ไม่ยอมผ่านด่าน staging)
- ธง Feature flags เก่าฝุ่นเกาะ ไม่มีวันหมดอายุ หรือไม่มีหัวหน้ามารับผิดชอบ
- ปล่อยเสร็จแล้วไม่มีใครหน้าไหนมาเฝ้าจอมอนิเตอร์ใน 1 ชั่วโมงแรก
- การปรับแต่งสวิตช์เซิร์ฟเวอร์บน Production ทำด้วยความจำส่วนตัว, ไม่ยอมใช้โค้ดรันจัดการ
- "นี่มันบ่ายวันศุกร์แล้วนะพี่, ปล่อยแม่งขึ้นเซิร์ฟเวอร์ไปเลยละกัน" (คำต้องห้ามก่อนสุดสัปดาห์บรรลัย)
