# Code Simplification

> ได้รับแรงบันดาลใจมาจาก [Claude Code Simplifier plugin](https://github.com/anthropics/claude-plugins-official/blob/main/plugins/code-simplifier/agents/code-simplifier.md). ถูกนำมาประยุกต์ใหม่ที่นี่ในฐานะของทักษะกระบวนการทำงานที่ไม่ยึดติดกับโมเดล เหมาะสำหรับ AI coding agent ทุกรูปแบบ.

## Overview

ทำโค้ดให้เรียบง่ายโดยการลดความซับซ้อน แต่ยังคงรักษาพฤติกรรมการทำงานเดิมไว้เป๊ะๆ เป้าหมายไม่ใช่การทำให้บรรทัดน้อยลง — แต่เป็นโค้ดที่ต้องอ่านง่าย, ทำความเข้าใจง่าย, แก้ไขง่าย, และหาบั๊กง่าย (debug) ทุกการปรับปรุงความเรียบง่ายต้องผ่านการทดสอบคำถามนี้ให้ได้: "ถ้ามีสมาชิกทีมคนใหม่เข้ามา เขาจะทำความเข้าใจโค้ดเวอร์ชันนี้ได้เร็วกว่าโค้ดต้นฉบับไหม?"

## When to Use

- หลังจากเขียนฟีเจอร์เสร็จแล้วและรัน tests ผ่านหมด, แต่ตัวโค้ดที่เขียนดูยุ่งยากวุ่นวายเกินเหตุ
- ระหว่างการทำ code review เมื่อมีคนร้องทักเรื่องความยากในการอ่าน หรือความซับซ้อนเกินไป
- เมื่อคุณเผชิญกับตรรกะที่ซ้อนกันลึกๆ (deeply nested logic), ฟังก์ชันที่ยาวเหยียด, หรือตัวแปรที่ตั้งชื่อไม่สื่อความหมาย
- เมื่อต้องกลับมาทำ refactor โค้ดที่เคยปั่นไฟลุกท่ามกลางความกดดันเรื่องเวลา
- เมื่อต้องรวบรวมตรรกะหน้าตาเหมือนๆ กันที่กระจัดกระจายอยู่ตามไฟล์ต่างๆ
- หลังจากเพิ่ง merge การเปลี่ยนแปลงที่ส่งผลให้เกิดการทำงานซ้ำซ้อนหรือขาดความคงเส้นคงวา

**เมื่อใดที่ไม่ควรใช้:**

- โค้ดมันคลีนและอ่านง่ายอยู่แล้ว — อย่าหาทำเพียงเพื่อตอบสนองความคัน
- คุณยังไม่รู้เลยด้วยซ้ำว่าโค้ดนี้มันทำอะไร — ไปทำความเข้าใจให้รู้เรื่องก่อนค่อยมาคิดเรื่อง simplify
- โค้ดนี้มีความสำคัญยิ่งยวดเรื่องความเร็ว (performance-critical) และเวอร์ชันที่ "ดูง่ายกว่า" จะทำให้ระบบอืดลงอย่างเห็นได้ชัด
- คุณกำลังเตรียมรื้อโมดูลนี้เขียนใหม่ทั้งยวง — การไปนั่งปรับปรุงโค้ดที่กำลังจะโดนเททิ้ง เป็นเรื่องเสียเวลาเปล่า

## The Five Principles (5 กฎเหล็ก)

### 1. Preserve Behavior Exactly (รักษาพฤติกรรมดั้งเดิมให้เป๊ะ)

ห้ามเปลี่ยนว่าโค้ด 'ทำอะไร' — ให้เปลี่ยนแค่โค้ด 'เขียนอย่างไร' ทั้งหมดไม่ว่าจะเป็น inputs, outputs, ผลข้างเคียง (side effects), พฤติกรรมเมื่อเกิด error, และพวก edge cases จะต้องแสดงผลลัพธ์แบบเดียวกันเป๊ะๆ ถ้าคุณไม่มั่นใจว่าการปรับปรุงของคุณจะไปเปลี่ยนพฤติกรรมการทำงานเดิมไหม ก็ไม่ต้องทำ

```
ก่อนจะแก้ไขอะไร ให้ถามตัวเองก่อนว่า (ASK BEFORE EVERY CHANGE):
→ โค้ดใหม่ยังคงพ่น output แบบเดิม สำหรับ input แบบเดิมทุกอันไหม?
→ โค้ดใหม่ยังคงรักษารูปแบบ error แบบเดิมไว้ไหม?
→ โค้ดใหม่ยังคงผลข้างเคียงและลำดับการทำงานแบบเดิมไว้ไหม?
→ tests เดิมทั้งหมดที่มีอยู่ยังรันผ่าน โดยที่ไม่ต้องแอบไปแก้ตัว tests ใช่ไหม?
```

### 2. Follow Project Conventions (ทำตามธรรมเนียมของโปรเจกต์)

การทำให้อ่านง่าย (Simplification) หมายถึงการทำให้โค้ดสอดคล้องกับสภาพโค้ดโดยรวมของโปรเจกต์ ไม่ใช่การไปยัดเยียดรสนิยมส่วนตัวของคุณใส่เข้าไป ก่อนทำการ simplify:

```
1. อ่านข้อตกลงใน CLAUDE.md / ธรรมเนียมปฏิบัติของโปรเจกต์
2. ศึกษาว่าโค้ดบริเวณรอบๆ เขาจัดการกับแพทเทิร์นลักษณะเดียวกันอย่างไร
3. ลอกเลียนแบบสไตล์ของโปรเจกต์ในเรื่อง:
   - การเรียงลำดับ Import และระบบการเรียกใช้งานโมดูล
   - สไตล์การประกาศฟังก์ชัน
   - กฎการตั้งชื่อ
   - แพทเทิร์นดักจับ error 
   - ระดับความลึกของการประกาศ Types
```

การ simplify ที่แหกกฎความสม่ำเสมอของโปรเจกต์ ไม่ถือว่าเป็นการทำ simplification — มันคือการปั่นป่วน (churn) ให้รกหูรกตา

### 3. Prefer Clarity Over Cleverness (เลือกความกระจ่างชัด ดีกว่าโชว์ความล้ำ)

โค้ดที่เขียนแบบตรงๆ ทื่อๆ ย่อมดีกว่าโค้ดสั้นกุดแต่บีบอัดจนต้องหยุดคิดเพื่อทำความเข้าใจ

```typescript
// ดูไม่รู้เรื่อง (UNCLEAR): เขียนเงื่อนไข ternary อัดเป็นลูกโซ่
const label = isNew ? 'New' : isUpdated ? 'Updated' : isArchived ? 'Archived' : 'Active';

// ดูง่าย (CLEAR): การแปลงค่า(mapping) ที่อ่านเข้าใจทันที
function getStatusLabel(item: Item): string {
  if (item.isNew) return 'New';
  if (item.isUpdated) return 'Updated';
  if (item.isArchived) return 'Archived';
  return 'Active';
}
```

```typescript
// ดูไม่รู้เรื่อง (UNCLEAR): ต่อคิว reduce เป็นลูกโซ่ แอบยัดตรรกะไว้ข้างใน
const result = items.reduce((acc, item) => ({
  ...acc,
  [item.id]: { ...acc[item.id], count: (acc[item.id]?.count ?? 0) + 1 }
}), {});

// ดูง่าย (CLEAR): สร้างตัวแปรแวะพักระหว่างทางแล้วตั้งชื่อบอกให้รู้
const countById = new Map<string, number>();
for (const item of items) {
  countById.set(item.id, (countById.get(item.id) ?? 0) + 1);
}
```

### 4. Maintain Balance (รักษาสมดุล)

การ Simplification มีข้อบกพร่องที่พบบ่อย คือ: มักง่ายจนเกินไป ระวังหลุมพรางเหล่านี้ให้ดี:

- **การอัดรวมโค้ดที่รุนแรงเกินไป (Inlining too aggressively)** — การลบฟังก์ชันผู้ช่วย(helper) ที่เคยช่วยตั้งชื่อบรรยายให้แนวคิดบางอย่างออกไป ทำให้จุดที่เรียกใช้ตัวโค้ดหลักอ่านรู้เรื่องยากขึ้น
- **รวมเอาตรรกะที่ไม่เกี่ยวข้องกันมาอยู่ด้วยกัน (Combining unrelated logic)** — การยุบ 2 ฟังก์ชันง่ายๆ มารวมร่างกันกลายเป็น 1 ฟังก์ชันที่โคตรซับซ้อน ไม่ใช่ความเรียบง่าย 
- **ลบรูปแบบที่ดูเหมือนจะ "ไม่จำเป็น" ออกไป (Removing "unnecessary" abstraction)** — โครงสร้างบางแบบถูกสร้างขึ้นมาเพื่อให้ง่ายต่อการต่อเติม (extensibility) หรือเพื่อให้ง่ายต่อการเขียน test (testability), มันไม่ใช่เพื่อความซับซ้อน
- **พุ่งเป้าไปที่การลดจำนวนบรรทัด (Optimizing for line count)** — โค้ดสั้นบรรทัดน้อย ไม่ใช่เป้าหมายสูงสุด; การทำให้อ่านง่ายรู้เรื่องไวต่างหากคือเป้าหมาย

### 5. Scope to What Changed (จำกัดขอบเขตอยู่แค่ตรงส่วนที่แก้ไข)

โดยปกติให้ตีกรอบการทำ simplify อยู่แค่เฉพาะจุดที่มีการแก้ไขโค้ดล่าสุด หลีกเลี่ยงการทำตัวเป็นฮีโร่แวะไปแก้โค้ดชาวบ้านที่ไม่เกี่ยวข้อง เว้นเสียแต่ว่าโดนสั่งมาโดยตรงให้ขยายวงกว้าง การทำ simplify มั่วซั่วผิดจุด ทำให้ข้อมูลการดัดแปลง (diff) บวมเป่ง และเสี่ยงให้เกิดบั๊กเดิมกลับมา (regressions) แบบไม่ได้ตั้งใจ

## The Simplification Process (ขั้นตอนการทำ)

### Step 1: Understand Before Touching (Chesterton's Fence) (ทำความเข้าใจก่อนจะแตะ)

ก่อนที่จะเริ่มเปลี่ยนหรือลบอะไร, จงทำความเข้าใจก่อนว่าทำไมมันถึงมาอยู่ตรงนั้น นี่คือทฤษฎีรั้วของเชสเตอร์ตัน (Chesterton's Fence): ถ้าคุณเจอรั้วกั้นอยู่กลางถนน แล้วคุณไม่เข้าใจว่ารั้วมาตั้งตรงนี้ทำไม, อย่าทะลึ่งไปพังมันทิ้ง ให้ไปสืบเหตุผลก่อนว่าสร้างมาทำไม, จากนั้นคุณถึงจะมีสิทธิ์ตัดสินว่าเหตุผลนั้นยังสมควรมีอยู่ไหม

```
ก่อนจะทำการ SIMPLIFYING, ให้ตอบคำถามนี้:
- โค้ดก้อนนี้ทำหน้าที่อะไร?
- ใครเรียกใช้มัน? มันไปเรียกใช้อะไรต่อไหม?
- มีเคสขอบหลืบ (edge cases) หรือ เส้นทาง error อันไหนบ้าง?
- มี tests ตัวไหนที่อธิบายความคาดหวังพฤติกรรมนี้ไว้ไหม?
- ทำไมเขาถึงเขียนมาท่านี้แต่แรก? (เน้นความเร็ว? ข้อจำกัดของตัวระบบ? หรือแค่ร่องรอยประวัติศาสตร์?)
- ลองเช็ค git blame ดู: ก่อนหน้านี้ใครหน้าไหนเป็นคนเขียนมาแต่แรก?
```

ถ้าคุณตอบคำถามพวกนี้ไม่ได้, แปลว่าคุณยังไม่พร้อมจะ simplify กลับไปนั่งอ่านบริบทรอบๆ ให้เยอะขึ้นก่อน

### Step 2: Identify Simplification Opportunities (มองหาโอกาส)

สอดส่องสายตามองหารูปแบบพวกนี้ — พวกมันคือสัญญาณไฟ ไม่ใช่กลิ่นตุๆ แบบคิดไปเอง:

**ความซับซ้อนเชิงโครงสร้าง (Structural complexity):**

| แพทเทิร์น | สัญญาณเตือน | วิธีการทำ Simplify |
|---------|--------|----------------|
| ซ้อนลึกจัดๆ (Deep nesting 3 ชั้นขึ้นไป) | ตามอ่านตรรกะเงื่อนไข(control flow) ไม่ไหว | สกัดเอาเงื่อนไขออกมาเป็น guard clauses หรือไปสร้างเป็นฟังก์ชันตัวช่วย (helper functions) |
| ฟังก์ชันยาวเป็นหางว่าว (Long functions 50 บรรทัดขึ้น) | เหมาทำงานหลายหน้าที่เกิน | แตกฟังก์ชันใหม่ให้โฟกัสทำแค่หน้าที่เดียว พร้อมตั้งชื่อดีๆ |
| ซ้อน ternary มั่วตั้ว (Nested ternaries) | สมองมนุษย์แปลผลไม่ทัน | แทนที่ด้วย if/else ซ้อนกัน, ใช้ switch, หรือสร้างอ็อบเจกต์ไว้ดึงค่า (lookup objects) |
| พารามิเตอร์แบบตั้งสถานะเปิดปิด (Boolean parameter flags) | เช่น `doThing(true, false, true)` | แทนที่ด้วยการส่ง options objects แทน หรือจับแยกฟังก์ชันไปเลย |
| เช็คเงื่อนไขซ้ำซาก (Repeated conditionals) | เช็ค `if` เดิมซ้ำๆ ในหลายๆ ที่ | สกัดออกไปเป็นฟังก์ชันตรวจสอบเงื่อนไขที่ตั้งชื่อบอกชัดเจน (predicate function) |

**การตั้งชื่อและความอ่านง่าย (Naming and readability):**

| แพทเทิร์น | สัญญาณเตือน | วิธีการทำ Simplify |
|---------|--------|----------------|
| ตั้งชื่อหยาบๆ (Generic names) | `data`, `result`, `temp`, `val`, `item` | เปลี่ยนชื่อให้อธิบายตัวตนของมัน: `userProfile`, `validationErrors` |
| ตั้งชื่อย่อๆ (Abbreviated names) | `usr`, `cfg`, `btn`, `evt` | ใช้คำเต็มๆ ไปเลย ยกเว้นว่าเป็นคำย่อระดับสากล (`id`, `url`, `api`) |
| ตั้งชื่อหลอกลวง (Misleading names) | ฟังก์ชันชื่อขึ้นต้นด้วย `get` แต่ดันทะลึ่งแอบไปเปลี่ยนค่า state | เปลี่ยนชื่อให้ตรงกับการกระทำของมัน |
| ใส่คอมเมนต์บอก "ว่ากำลังทำอะไร" (what) | `// เพิ่มทีละหนึ่ง` อยู่เหนือคำสั่ง `count++` | ลบคอมเมนต์ขยะทิ้งซะ — ตัวโค้ดมันบอกชัดอยู่แล้ว |
| ใส่คอมเมนต์บอก "เหตุผลว่าทำไม" (why) | `// พยายามใหม่ เพราะ API นี้มันชอบล่มตอนคนใช้เยอะ` | อันนี้เก็บไว้ — มันบอกความตั้งใจในแบบที่ตัวโค้ดแสดงออกมาไม่ได้ |

**ความซ้ำซ้อน (Redundancy):**

| แพทเทิร์น | สัญญาณเตือน | วิธีการทำ Simplify |
|---------|--------|----------------|
| โค้ดเหมือนกันเป๊ะ (Duplicated logic) | โค้ด 5 บรรทัดหน้าตาเหมือนกันไปโผล่ในหลายๆ ที่ | ดึงออกไปเป็น shared function ซะ |
| ซากโค้ดตาย (Dead code) | โค้ดที่ไม่มีทางรันไปถึง, ตัวแปรที่ตั้งมาแต่ไม่ได้ใช้, โค้ดที่โดนปิดการใช้งาน (commented-out) | ลบทิ้ง (หลังจากที่มั่นใจแน่ๆ แล้วว่ามันตายชัวร์) |
| ครอบสิ่งที่ไม่จำเป็น (Unnecessary abstractions) | เอาอะไรไปห่อหุ้มไว้ แต่ไม่ได้เพิ่มประโยชน์อะไรขึ้นมาเลย | รื้อตัวหุ้มทิ้งไป, เรียกใช้ฟังก์ชันไส้ในตรงๆ เลย |
| แพทเทิร์นเว่อร์วังอลังการ (Over-engineered patterns) | เช่น Factory-for-a-factory, strategy-with-one-strategy | เปลี่ยนกลับไปใช้ท่าทั่วไปแบบตรงไปตรงมา |
| การยัดเยียดประทับตรา Type (Redundant type assertions) | แอบไปบังคับ casting ชนิด Type ของตัวแปร ทั้งที่ระบบก็เดาได้เองอยู่แล้ว | ถอดตัว assertion ออกไป |

### Step 3: Apply Changes Incrementally (ทยอยแก้ทีละนิด)

ทำการ simplify ทีละรายการ รัน tests หลังจากแก้เสร็จแต่ละจุด **ถ้าเป็นการ refactor แบบหนักๆ ให้แยกการบันทึกรวม(Submit) ออกจากการเพิ่มฟีเจอร์หรือการแก้บั๊ก** การยัดรวมงาน refactor กับงานฟีเจอร์ลงใน PR เดียวถือเป็นความผิด — ให้แยกมันเป็นสอง PR 

```
ทุกๆ ครั้งที่จะทำ SIMPLIFICATION:
1. แก้ไขโค้ด
2. รันระบบ test
3. ถ้า tests ผ่าน → commit (แล้วค่อยทำจุดอื่นต่อไป)
4. ถ้า tests พัง → ดึงโค้ดกลับไปสภาพเดิม (revert) แล้วคิดทบทวนใหม่
```

หลีกเลี่ยงการทำทีละเยอะๆ แล้วรวบยอดส่งตู้มเดียวโดยไม่เทส ถ้าเกิดมีอะไรพังขึ้นมา คุณจะได้รู้ได้ทันทีว่าการ simplify จุดไหนเป็นต้นเหตุ

**กฎ 500 บรรทัด (The Rule of 500):** ถ้าการ refactoring ครั้งนี้ต้องไปแก้โค้ดเกิน 500 บรรทัด ให้ไปใช้เครื่องมือทุ่นแรงประเภทอัตโนมัติ (เช่น codemods, sed scripts, AST transforms) ดีกว่าไปนั่งหลังขดหลังแข็งไล่แก้เองด้วยมือ การแก้ด้วยมือทีละเยอะๆ ขนาดนั้นโอกาสพลาดสูงและทำให้คนรีวิวตาลาย

### Step 4: Verify the Result (ตรวจสอบผลลัพธ์)

หลังจากที่ simplify เสร็จทุกจุด, ให้ถอยมาหนึ่งก้าวแล้วประเมินภาพรวม:

```
เปรียบเทียบ ก่อน และ หลัง:
- เวอร์ชันใหม่ที่ทำเสร็จแล้วมันดูอ่านง่ายขึ้นอย่างแท้จริงใช่ไหม?
- คุณเผลอเอาแพทเทิร์นประหลาดๆ ที่ผิดเพี้ยนไปจาก codebase เดิมเข้ามาไหม?
- ข้อมูลความต่าง (diff) คลีนและง่ายต่อการรีวิวไหม?
- ถ้าเพื่อนร่วมทีมมาเห็น เขาจะอนุมัติการแก้ไขที่เป็นประโยชน์นี้ไหม?
```

ถ้าโค้ดเวอร์ชันที่ถูกอ้างว่า "simplified แล้ว" กลับดูทำความเข้าใจยากขึ้น หรือรีวิวยากกว่าเดิม, ให้ย้อนกลับ (revert) ทิ้งไปเลย เพราะไม่ใช่ทุกความพยายามในการ simplify จะประสบความสำเร็จ

```python
def process(data):
    if data is not None:
        if data.is_valid():

                return do_work(data)
            else:
                raise PermissionError("No permission")
        else:
            raise ValueError("Invalid data")
    else:
        raise TypeError("Data is None")
# หลังแก้
def process(data):
    if data is None:
        raise TypeError("Data is None")
    if not data.is_valid():
        raise ValueError("Invalid data")
    if not data.has_permission():
        raise PermissionError("No permission")
    return do_work(data)
```

## Common Rationalizations (ข้ออ้างทั่วไป)

| ข้ออ้าง | ความเป็นจริง |
|---|---|
| "ก็มันรันผ่านแล้วไง, ไม่เห็นต้องไปยุ่งกับมันเลย" | โค้ดที่รันผ่านแต่อ่านไม่รู้เรื่อง เวลาเจอบั๊กจะหาทางแก้ได้ยาก การทำให้ง่ายตั้งแต่ตอนนี้จะช่วยประหยัดเวลาการแก้ไขในอนาคตไปตลอดชีวิต |
| "บรรทัดยิ่งน้อย ยิ่งดูเรียบง่ายอยู่แล้ว" | การจับ ternary ไปซ้อนยัดกันไว้ในบรรทัดเดียว ไม่ได้ช่วยให้อ่านง่ายกว่าการเขียน if/else ใน 5 บรรทัด ความเรียบง่ายวัดจากระยะเวลาที่สมองใช้ทำความเข้าใจ ไม่ใช่วัดที่จำนวนบรรทัด |
| "เดี๋ยวฉันแอบแว้บไปแก้โค้ดที่ไม่เกี่ยวข้องกันตัวนั้นด้วยเลยละกัน" | การแก้มั่วซั่วผิดจุด ทำให้ข้อมูลการดัดแปลง (diff) บวมเป่ง และเสี่ยงให้เกิดบั๊กเดิมกลับมา (regressions) ในส่วนที่คุณไม่ได้ตั้งใจ โฟกัสหน่อย |
| "ตัว Types ที่แปะไว้มันก็อธิบายตัวเองอยู่แล้วนี่นา" | Types บอกได้แค่โครงสร้าง, ไม่ได้บอกเจตนา การตั้งชื่อฟังก์ชันดีๆ อธิบายได้ดีกว่าว่า *ทำไม* ถึงต้องมีมัน, ซึ่งเหนือกว่าการที่ type signatures มัวแต่อธิบายว่าระบบคือ *อะไร* |
| "แพทเทิร์นโครงสร้างที่เราทำเผื่อๆไว้นี้ อาจจะได้ใช้ประโยชน์ในอนาคตนะ" | อย่าพยายามรักษาสิ่งที่ถูกคาดเดาว่าจะได้ใช้ในอนาคต ถ้าตอนนี้มันยังไม่ถูกใช้งาน, มันก็คือความซับซ้อนที่ไร้ประโยชน์ ลบทิ้งไปซะ วันไหนได้ใช้จริงๆ ค่อยเอากลับมาใหม่ |
| "คนที่เขียนทิ้งไว้น่าจะมีเหตุผลบางอย่างแหละ" | อาจจะใช่ ลองเช็ค git blame ดู — อิงหลัก Chesterton's Fence. แต่ส่วนมากความซับซ้อนที่สะสมไว้มันไม่มีเหตุผลอะไรหรอก; มันเป็นแค่เศษซากการปั่นงานตอนไฟลนก้นนั่นแหละ |
| "เดี๋ยวฉันทำ refactor ไปพร้อมกับปั่นฟีเจอร์ใหม่ไปด้วยเลยละกัน" | แยกงาน refactor ออกจากงานเพิ่มฟีเจอร์ซะ การเปลี่ยนโค้ดแบบลูกผสมจะทำให้รีวิวโค้ดยาก, จะถอยกลับ (revert) ก็ยาก, แล้วก็งงประวัติ (history) อีก |

## Red Flags (สัญญาณเตือนภัย)

- การ Simplify ที่ทำให้ต้องกลับไปแก้โค้ดทดสอบ (tests) ให้รันผ่าน (แสดงว่าคุณแอบไปเปลี่ยนพฤติกรรมโค้ดเข้าแล้วล่ะ)
- โค้ดที่อ้างว่า "Simplified" มาแล้ว แต่ดันยาวเหยียดและตามแกะยากกว่าต้นฉบับ
- ไปแอบเปลี่ยนชื่อชาวบ้านตามความชอบส่วนตัว แทนที่จะอิงตามธรรมเนียมโปรเจกต์
- แอบลบระบบจัดการ error ทิ้ง เพียงอ้างเหตุผลว่า "มันทำให้โค้ดดูสะอาดขึ้น"
- ไป Simplify โค้ดที่ตัวเองก็ยังไม่เข้าใจ
- ยัดการทำ Simplification เป็นร้อยอันเข้าไปในก้อนคอมมิตยักษ์ (commit) แค่ก้อนเดียว ทำให้คนมารีวิวโค้ดหัวหมุน
- ทำ Refactor นอกเหนือคำสั่งงานโดยพลการ โดยไม่มีใครร้องขอ

## Verification

หลังจากกระบวนการ simplification สิ้นสุดลง:

- [ ] ทุกๆ existing tests รันผ่านโดยไม่ต้องแก้ไขโค้ด test
- [ ] Build สำเร็จโดยไม่มีคำแจ้งเตือน (warnings) โผล่มาใหม่
- [ ] เครื่องมือ Linter/formatter รันผ่านหมดฉลุย (แสดงว่าไม่ได้แอบเปลี่ยนสไตล์)
- [ ] ทุกๆ simplification ที่ทำ สามารถรีวิวได้แยกชิ้น, ทยอยนำส่งทีละเล็กละน้อยได้ (incremental change)
- [ ] ข้อมูลความต่างโค้ด (diff) คลีนสะอาดตา — ไม่มีการไปแอบแก้เรื่องที่ไม่เกี่ยวข้องรวมเข้ามาด้วย
- [ ] โค้ดที่ simplified แล้วยังเป็นไปตามธรรมเนียมโปรเจกต์ (เช็คอิงจาก CLAUDE.md หรือคู่มืออื่นๆ ที่เทียบเท่า)
- [ ] ไม่มีการแอบลบหรือทำให้ระบบจัดการ error อ่อนแอลง
- [ ] ไม่ทิ้งซากโค้ดตายไว้ข้างหลัง (เช่น การ import ที่ไม่ได้ใช้, หรือเส้นทางโค้ดที่ไม่มีทางเข้าถึง unreachable branches)
- [ ] เพื่อนร่วมทีม หรือ review agent พร้อมจะกดอนุมัติการแก้ไขครั้งนี้ในฐานะการพัฒนาโปรเจกต์ที่ดีขึ้น
